Week 1 — Foundations of UX Design
W1

Foundations of UX Design

UX vs UI · Double Diamond · HCD · Why UX matters in India · Nielsen's 10 Heuristics

▾

Learning Objectives

  • Explain the difference between UX and UI
  • Describe the four phases of the Double Diamond
  • Name and explain all 10 Nielsen heuristics
  • List 3 reasons UX matters specifically in India
  • Define task completion rate, SUS, and NPS

UX vs UI — The Core Distinction

  • UX (User Experience): How it works — the logic, the flow, the research, the decisions behind what a product does
  • UI (User Interface): How it looks — the visual layer of buttons, colors, typography, icons, and spacing
  • You can have great UI and terrible UX (beautiful app that is impossible to use)
  • You can have great UX and rough UI (functional app that looks dated)
  • The goal: both. UX strategy drives UI decisions — never the other way around

The Double Diamond Process

  • Discover: Diverge — research, talk to users, gather all problems
  • Define: Converge — synthesise research into a clear problem statement
  • Develop: Diverge — ideate solutions, sketch concepts, prototype options
  • Deliver: Converge — test, refine, ship the best solution
  • Two diamonds = two diverge/converge cycles. Problem space first, solution space second

Human-Centered Design (HCD)

  • Developed by IDEO. Five stages: Empathize → Define → Ideate → Prototype → Test
  • Empathize: Observe and interview users in their real context
  • Define: Write a clear problem statement from your research
  • Ideate: Generate many possible solutions (quantity before quality)
  • Prototype: Build a rough version quickly and cheaply
  • Test: Put the prototype in front of users — observe, don't explain

Key UX Metrics

  • Task Completion Rate (TCR): % of users who successfully finish a task
  • Time on Task: How long it takes — shorter is not always better
  • Error Rate: How many mistakes users make trying to complete a task
  • NPS (Net Promoter Score): "Would you recommend this to a friend?" 0–10
  • SUS (System Usability Scale): 10-question questionnaire. Score above 68 = above average
🇮🇳 Why UX Matters in India: India has 700M+ internet users, the majority on budget Android smartphones on 2G/3G in rural areas. UX failures cost real users — low bandwidth means bloated apps time out, low literacy means icon-only navigation fails, 22 official languages means English-only interfaces exclude millions. Good UX design in India must be mobile-first, fast-loading, vernacular-ready, and work across a 5-year range of device capability.
Nielsen's 10 Usability Heuristics
Heuristic 01
Visibility of System Status

Always keep users informed about what is going on, through appropriate feedback within a reasonable time. Loading spinners, progress bars, success/error toasts all serve this heuristic.

Heuristic 02
Match Between System and Real World

Use language and concepts familiar to users, not internal system terminology. A shopping cart icon works because users already know what a cart is. Avoid jargon.

Heuristic 03
User Control and Freedom

Users often make mistakes. Give them a clearly marked "emergency exit": undo, redo, back navigation, cancel buttons. Never trap a user in a flow with no escape.

Heuristic 04
Consistency and Standards

Users should not have to wonder whether different words, situations, or actions mean the same thing. Follow platform conventions (iOS/Android/Web). Use the same button labels throughout.

Heuristic 05
Error Prevention

Better than good error messages: design so errors cannot happen. Disable the submit button until the form is valid. Confirm before irreversible actions like "Delete account".

Heuristic 06
Recognition Over Recall

Minimise the user's memory load. Show recent items, visible options, autocomplete suggestions. Users should recognise paths rather than recall them from memory.

Heuristic 07
Flexibility and Efficiency of Use

Accelerators (keyboard shortcuts, saved addresses, quick reorder) let experienced users move faster while not interfering with novices. Design for both skill levels simultaneously.

Heuristic 08
Aesthetic and Minimalist Design

Every additional element competes with every other element for attention. Remove anything that does not help the user complete their current task. Whitespace is not waste.

Heuristic 09
Help Users Recognize, Diagnose, Fix Errors

Error messages must use plain language (no error codes), describe the problem precisely, and suggest a solution. "Invalid input" is useless. "Email must include an @ symbol" is helpful.

Heuristic 10
Help and Documentation

Even though it is better if the system can be used without documentation, help must be easy to search, focused on the user's task, and presented at the moment it is needed.

Key Takeaway — Week 1

  • UX is strategy and research; UI is the visual execution. Both matter, but UX leads.
  • The Double Diamond keeps you from designing solutions to the wrong problem.
  • Nielsen's 10 Heuristics are a free, fast way to find usability issues in any existing product.
  • In India, "good enough for a laptop" is not good enough — design for the constrained case first.
Week 2 — User Research Methods
W2

User Research Methods

Qualitative · Quantitative · User Interviews · Usability Testing · Affinity Diagrams

▾

Learning Objectives

  • Distinguish between qualitative and quantitative research
  • Write 5 open-ended user interview questions
  • Describe the think-aloud protocol in usability testing
  • Build an affinity diagram from sticky-note observations
  • Write a research plan with goal, method, and participant criteria

Why Research Matters

  • You are not the user. Your preferences, habits, and mental models are not universal
  • Without research, you design for yourself — a sample size of one
  • Research replaces assumptions with evidence — it changes arguments into decisions
  • Even 3 interviews reveal patterns you would never have guessed
  • Research prevents costly redesigns: fix it in discovery, not after launch

Quantitative Research

  • Answers "how many" and "how much" — gives you numbers you can measure
  • Surveys: Google Forms, Typeform — ask 100+ users rated questions
  • Analytics: Google Analytics — see which pages users leave (bounce rate)
  • Heatmaps: Hotjar, Clarity — visualise where users click, scroll, and hover
  • A/B testing: Show two versions to different users; measure which converts better

Qualitative Research

  • Answers "why" and "how" — gives you understanding, not just numbers
  • User interviews: 1-on-1 conversations to understand motivations and frustrations
  • Contextual inquiry: Watch users in their natural environment (home, office, field)
  • Diary studies: Users log their own experiences over days or weeks
  • Focus groups: Group discussion — good for exploring attitudes, poor for task observation

Running a User Interview

  • Write 5–7 open-ended questions (start with "Tell me about..." or "Walk me through...")
  • Ask about past behaviour, not hypothetical future behaviour
  • Record with permission — you cannot take notes and listen at the same time
  • No leading questions: "Don't you find the checkout confusing?" is leading
  • Follow up on anything surprising: "You paused there — what were you thinking?"
  • Duration: 20–45 minutes. Fewer than 5 participants = too few. More than 10 = diminishing returns at early stage
Jakob's Law of Usability Testing: Research by Jakob Nielsen showed that 5 users will uncover approximately 85% of a product's usability issues. Adding a 6th, 7th, and 8th user starts revealing the same issues again. Test with 5 users, fix those issues, then test 5 more. You get more learning from two rounds of 5 than one round of 10.

Usability Testing — Think-Aloud Protocol

  • Give the participant a real task ("Book a train ticket from Delhi to Chandigarh")
  • Ask them to say everything they are thinking as they do it
  • You observe, you do not help — even if they struggle
  • Record screen + audio (screen recording software or a phone propped up)
  • After the task: debrief ("What was most confusing? What worked well?")

Affinity Diagrams

  • After research, you have hundreds of observations and quotes
  • Write each observation on a separate sticky note (one idea per note)
  • Silently group the notes by theme — let categories emerge from the data
  • Name each cluster with a short label ("Users distrust payment screens")
  • Result: 4–8 insight themes that become your design brief
  • Use FigJam (free) or physical sticky notes on a wall

Key Takeaway — Week 2

  • Quantitative tells you what is happening; qualitative tells you why. Use both.
  • 5 users find 85% of issues. Start testing early, with rough prototypes.
  • Affinity diagrams turn raw research into structured insights — always cluster before designing.
  • Never lead your participants. The moment you help them, you've contaminated your data.
Week 3 — Information Architecture & User Flows
W3

Information Architecture & User Flows

IA · Card Sorting · Sitemaps · User Flows · Navigation Patterns · Labeling

▾

Learning Objectives

  • Define Information Architecture and explain why it matters
  • Conduct an open and a closed card sort
  • Draw a sitemap for a 5–10 screen mobile app
  • Draw a user flow for a checkout task (happy path + 2 error states)
  • Choose the right navigation pattern for a given use case

Information Architecture (IA)

  • IA is how you organise, structure, and label content so users can find what they need
  • Good IA = invisible. Bad IA = users are constantly lost and searching
  • Principles: organisation (grouping), labeling (naming), navigation (wayfinding), search
  • IA is decided before any visual design begins — it is the skeleton of the product
  • The IRCTC app is a famous example of poor IA: important functions are buried 4+ levels deep

Card Sorting

  • Open card sort: Users create their own category names and group cards into them — reveals users' mental models
  • Closed card sort: Users sort cards into categories you've predefined — tests whether your structure matches users' expectations
  • Use for: deciding how to group navigation items, naming sections, testing menu labels
  • Tools: OptimalSort (online), physical index cards
  • Run with 15–20 participants; look for patterns in how cards cluster

Sitemaps

  • A visual diagram showing every screen/page in a product and how they connect hierarchically
  • Use boxes for screens, lines for navigation relationships, dashed lines for conditional links
  • Always shows all screens — not just the happy path
  • Create before wireframing — forces you to think about completeness
  • Example: Home → Browse → Category → Product Detail → Cart → Checkout → Confirmation

User Flows

  • A flowchart showing the step-by-step path a user takes to complete one specific task
  • Shapes: rectangles = screens/pages, diamonds = decision points, parallelograms = user actions
  • Always show the happy path AND the error paths (payment fails, item out of stock)
  • Task analysis first: break the task into micro-steps before drawing
  • One user flow = one task. Don't combine multiple tasks into one diagram

Navigation Patterns — When to Use Each

  • Bottom tab bar: 3–5 main sections, frequently switched between. Best for mobile (thumb reach). Used by Instagram, WhatsApp, Google Maps
  • Hamburger menu: Many secondary options that don't need constant visibility. Hides navigation — use for secondary, not primary actions
  • Breadcrumbs: Deep hierarchical structures (e-commerce categories, documentation). Shows users where they are
  • Mega menus: Content-rich desktop sites with many categories (Amazon, news portals). Too complex for mobile

Labeling — Use the User's Language

  • Labels are what you call things in the navigation, buttons, and headings
  • Use the words your users use — not the words your company uses internally
  • Example: a bank calls it "Remittances" — users call it "Send money"
  • Test labels: give users a task and a label, ask if the label matches what they expected to find
  • Keep labels short (1–3 words), active, and consistent across the product

Key Takeaway — Week 3

  • IA is the skeleton — it must be right before any visual design begins.
  • Card sorting reveals users' mental models; never assume you already know how they think.
  • Draw both the happy path and error paths in your user flow — the edge cases are where UX breaks.
  • Use the user's vocabulary, not your internal company terms.
Week 4 — Visual Design Fundamentals
W4

Visual Design Fundamentals

Gestalt Principles · Visual Hierarchy · Typography · Color Theory · Grid Systems · Accessibility

▾

Learning Objectives

  • Name and apply the 5 key Gestalt principles
  • Build a visual hierarchy using size, weight, and contrast
  • Pair two typefaces (heading + body) correctly
  • Build a 3-color palette using color theory
  • Check WCAG 2.1 AA contrast ratios in Figma using Stark

Gestalt Principles

  • Proximity: Elements placed near each other appear related. Price, quantity, and CTA grouped together signal they belong to the same decision
  • Similarity: Elements that look alike appear related. Consistent card style across a product grid signals all cards are the same type of content
  • Figure-Ground: The eye separates subjects from background. Modals use a darkened overlay to push the background into "ground"
  • Continuity: The eye follows lines and curves. Carousels exploit this — the partly visible card signals "swipe to see more"
  • Closure: The mind completes incomplete shapes. A partially visible item invites scrolling. Pie charts work even without segment labels

Visual Hierarchy

  • Visual hierarchy tells the eye what to look at first, second, and third
  • Size: Larger = more important. A 32px heading dominates a 14px body
  • Weight: Bold text draws attention in a sea of regular text
  • Color: A bright CTA on a neutral background pops — but only if used sparingly
  • Contrast: High contrast = high importance. Low contrast = supporting information
  • Whitespace: Isolating an element with space makes it feel more important

Typography Fundamentals

  • Serif fonts: Have small strokes (serifs) at the end of characters. Better for long-form reading in print. Examples: Georgia, Playfair Display, Lora
  • Sans-serif fonts: No serifs. Cleaner on screens. Examples: Inter, DM Sans, Poppins
  • Font pairing rule: One display/heading font (personality) + one body font (legibility). Avoid using more than 2 typefaces
  • Line height: Set body text to 1.5–1.6× for comfortable reading
  • Type scale: Use consistent size steps: 12, 14, 16, 20, 24, 32, 48px

Color Theory

  • Color wheel: Primary (red, blue, yellow) → Secondary (orange, green, purple) → Tertiary
  • Complementary: Colors opposite on the wheel — high contrast, high energy
  • Analogous: Colors next to each other — harmonious, calm
  • Color associations in India: Saffron = energy/courage, green = growth/prosperity, blue = trust, white = peace/purity
  • 60-30-10 rule: 60% dominant (neutral), 30% secondary, 10% accent

Grid Systems

  • 8pt grid: All spacing values are multiples of 8: 8, 16, 24, 32, 48, 64px. Creates consistent rhythm
  • 12-column grid: Standard for web. Gives you flexibility: 12 columns = 1 full width, 6+6 = two halves, 4+4+4 = thirds
  • Margin and gutter: Margin = space from screen edge. Gutter = space between columns. Mobile: 16px margin, 8px gutter
  • Grids create alignment. Alignment creates professionalism

Contrast & Accessibility (WCAG 2.1)

  • Normal text (below 18px): Minimum 4.5:1 contrast ratio between text and background
  • Large text (18px+ or 14px+ bold): Minimum 3:1 contrast ratio
  • Why it matters: 8% of men have some form of color blindness. 285 million people worldwide have low vision
  • Tool: Stark plugin in Figma — checks contrast in one click
  • Never rely on color alone to convey information — always add an icon or label

Key Takeaway — Week 4

  • Gestalt principles explain why some layouts feel organized and others feel chaotic.
  • Visual hierarchy is designed, not accidental — every element's size and weight is a decision.
  • Stick to 2 typefaces: one for personality (headings), one for readability (body).
  • 4.5:1 contrast ratio is not a suggestion — it is the minimum for AA compliance. Check it every time.
