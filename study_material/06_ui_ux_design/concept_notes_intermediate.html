Week 5 — Figma Mastery
W5

Figma Mastery

Frames · Auto Layout · Components · Variants · Styles · Plugins · Shortcuts

▾

Learning Objectives

  • Create a master component and manage instances
  • Build a button component with variants (type/size/state)
  • Set up color styles and text styles in Figma
  • Use Auto Layout for responsive components
  • Install and use Stark, Unsplash, and Iconify plugins

Why Figma?

  • Browser-based: works on any OS, no download required for viewers
  • Real-time collaboration: multiple designers on the same file simultaneously
  • Free for individuals (Starter plan): unlimited personal files
  • Industry standard: used by Google, Microsoft, Uber, Zomato, Razorpay
  • Inspect panel: developers see CSS values, spacing, and exportable assets without a separate handoff tool

Key Figma Concepts

  • Frame: The artboard. Use frames for every screen (not groups)
  • Auto Layout: Makes frames resize with their content — essential for buttons and lists
  • Constraints: Define how an element moves when its parent frame resizes (left/right/center/scale)
  • Groups: Loose collection of layers. Use sparingly — prefer frames for layout
  • Layers panel: All your elements live here. Name every layer — messy layers = messy collaboration

Components

  • Master component: The original. Edit the master — all instances update automatically
  • Instance: A copy of the master that inherits all changes but can have local overrides
  • Component props: Text (change label text), boolean (toggle visibility), instance swap (change icon)
  • Create component: Ctrl+Alt+K (or right-click → Create Component)
  • Detach instance to make a unique one-off version (right-click → Detach Instance)

Variants

  • Variants group multiple states of a component into one component set
  • Example: Button with variants — Type (Primary/Secondary/Destructive), Size (S/M/L), State (Default/Hover/Disabled)
  • Create variants: select multiple components → right-click → Combine as Variants
  • Design goal: when you swap a variant, the layout should not break
  • Variants + Auto Layout = components that work in any design context

Styles — Define Once, Apply Everywhere

  • Color styles: Name your colors (Primary/Secondary/Error/Success/Text/Background) and apply from the style panel
  • Text styles: H1, H2, Body, Caption, Label — define font, size, weight, line-height once
  • Effect styles: Shadow definitions you can apply to multiple cards consistently
  • When you update a style, every element using that style updates instantly
  • Styles are the Figma equivalent of CSS variables

Useful Figma Plugins

  • Unsplash: Insert royalty-free photos directly into Figma frames
  • Iconify: 100,000+ icons (Material, Feather, Phosphor) searchable and insertable
  • Content Reel: Paste realistic data — names, avatars, addresses — into your designs
  • Stark: Accessibility checker — contrast ratio, color blindness simulation, focus order
  • Install plugins: Main menu → Resources → Plugins → Browse

Key Takeaway — Week 5

  • Auto Layout is the single most important Figma skill. Master it and your components will be resilient.
  • Components + Variants = a design system in miniature. Never design the same button twice.
  • Name every layer clearly. Messy layer names are technical debt in design files.
  • Styles are CSS variables in Figma — define them before you design, not after.
Week 6 — Design Systems
W6

Design Systems

Atomic Design · Design Tokens · Real-World Systems · Building Your Own · Handoff

▾

Learning Objectives

  • Explain the Atomic Design hierarchy (atoms → molecules → organisms)
  • Define design tokens and give 4 examples
  • Name 3 real-world design systems and their use cases
  • Build a mini design system with 5 colors, 3 text styles, 8 components
  • Describe the designer-to-developer handoff workflow
What Is a Design System? A design system is a shared library of reusable components, design tokens, and documented guidelines that allows teams to build consistent products faster. Think of it as a LEGO set: each brick is a component; the colour system is the palette; the instruction booklet is the documentation. With a design system, a team of 20 designers can work on different parts of an app and it still looks like one product.
Atomic Design — Brad Frost
⚛️
Atoms

Smallest unit: button, input, icon, label, color swatch, type style

🧬
Molecules

Atoms combined: a search field = input + icon + button

🦠
Organisms

Molecules combined: a navbar = logo + search + nav links + CTA button

📄

Page layout without real content: the wireframe skeleton

🌐
Pages

Templates with real content: what users actually see

Design Tokens

  • Design tokens are named variables for visual decisions: colors, spacing, typography, shadows
  • Color tokens: --color-primary: #1B3A6B; --color-error: #C62828
  • Spacing tokens: --space-xs: 4px; --space-sm: 8px; --space-md: 16px
  • Typography tokens: --font-size-body: 15px; --line-height-body: 1.6
  • Tokens make it possible to update the entire design system by changing one value — and they map directly to CSS custom properties in code

Real-World Design Systems

  • Google Material Design 3: Open-source. Used by all Google products. Excellent documentation and Figma kit
  • Apple Human Interface Guidelines (HIG): iOS/macOS design rules. Required reading for any iOS app designer
  • Ant Design: Used by Alibaba. Comprehensive React component library with Figma counterpart
  • Chakra UI: Popular open-source React component library with strong accessibility defaults

Mini Design System — 8 Components

  • Button (primary/secondary/destructive variants)
  • Input (empty/filled/error/focused states)
  • Card (with image, title, description, CTA)
  • Badge (success/warning/error/info)
  • Avatar (image with fallback initials)
  • Navbar (logo + navigation + CTA)
  • Modal (header + body + actions + overlay)
  • Toast (success/error/info — appears bottom center)

Designer-to-Developer Handoff

  • Figma Inspect panel: Developers click any element and see CSS values, font details, spacing, and export options
  • Dev Mode: Figma's dedicated developer view — shows measurements, redlines, and code snippets
  • Assets: Export icons and images directly from Figma as SVG, PNG 1x/2x/3x
  • Annotations: Add written notes to Figma frames explaining behaviour, interaction, edge cases
  • Handoff is not a one-time event — plan weekly design reviews with your developer partner

Key Takeaway — Week 6

  • A design system is not a style guide — it is a living product shared between design and engineering.
  • Atomic Design gives you a vocabulary: atoms, molecules, organisms. Use it to structure your Figma files.
  • Design tokens make the design system portable — the same token works in Figma, CSS, iOS, and Android.
  • Handoff is a process, not a file hand-over. Build the developer relationship early.
Week 7 — Interaction Design & Prototyping
W7

Interaction Design & Prototyping

Affordance · Microinteractions · Figma Prototype Mode · Fidelity Levels · Mobile Gestures

▾

Learning Objectives

  • Explain the four interaction design principles (feedback, affordance, signifier, constraint)
  • Describe microinteractions using Saffer's trigger-rules-feedback-loops model
  • Connect Figma frames with prototype arrows and set Smart Animate transitions
  • Choose the right prototype fidelity for the testing stage
  • Design empty states, loading states, and error states for all your screens

Interaction Design Principles

  • Feedback: Every action produces a visible response. A button press changes color. A form submit shows a spinner. Silence after an action creates anxiety
  • Affordance: Objects communicate how they should be used. A button looks pressable. A slider looks draggable. Flat design risks losing affordance signals
  • Signifiers: Visual cues that communicate affordance. A "handle" icon on a draggable card. An underline on a link. A chevron indicating navigation
  • Constraints: Restrict what is possible at each step. Greying out unavailable options. Blocking checkout until address is filled

Microinteractions — Dan Saffer

  • Microinteractions are the small, contained interactions within a product: liking a post, toggling a setting, sending a message
  • Trigger: What initiates the microinteraction? (User taps a heart icon)
  • Rules: What happens? (The heart fills red; count increments by 1)
  • Feedback: What does the user see/feel? (Heart bounces; haptic pulse on mobile)
  • Loops/Modes: Does it repeat? Does the state change next time? (The heart stays filled; tap again to unlike)

Figma Prototyping

  • Connect frames: in Prototype tab, drag the blue arrow from an element to a destination frame
  • Interaction types: On click, On tap, After delay (auto-advance), On drag, While hovering
  • Animations: Instant, Dissolve, Smart Animate (morphs matching layers), Slide, Push
  • Smart Animate: Requires matching layer names in both frames — Figma animates the position/size change automatically
  • Share prototype: top-right Share button → set to "Anyone with link can view" → Present mode

Prototype Fidelity Levels

  • Paper sketch (lo-fi): Fastest. Use for early concepts, rough flows. Test with 2–3 people before committing to digital
  • Lo-fi digital wireframe: Grey boxes, no color, placeholder text. Tests navigation and flow
  • Mid-fi: Structure + typography but no final colors or images. Good for layout decisions
  • Hi-fi mockup: Final colors, components, real content. Tests visual design and detailed interactions
  • Rule: test with the lowest fidelity that answers your question. High-fi too early wastes time

Mobile Gestures — Design Affordances

  • Tap: Primary action. Every CTA, link, and button. Minimum 44×44pt touch target
  • Double tap: Secondary action (like a photo). Must be discoverable — show it in onboarding
  • Long press: Context menu. Show a visible hold-animation to confirm the gesture registered
  • Swipe: Navigate between screens or dismiss items. Provide visible handles or cues (partly visible next card)
  • Pinch: Zoom. Only for content that benefits from zoom (maps, photos, documents)
  • Pull to refresh: Show a spinner animation; confirm with a subtle success state

Design All Three States — Every Screen

  • Loading state: Never show a blank white screen. Use skeleton loaders (grey placeholder shapes) — not spinners alone
  • Empty state: When there is no content yet — new user, empty cart, no results. Include: illustration or icon, explanation of why it is empty, clear CTA to take action
  • Error state: Something went wrong. Specify what (network error, not found, permission denied), offer a fix (retry, go back, contact support)
  • Designing only the happy path is the most common beginner mistake in prototyping

Key Takeaway — Week 7

  • Affordance is the most important interaction concept: things should look like what they do.
  • Microinteractions are how users form trust with a product — get the feedback loop right.
  • Always design loading, empty, and error states. They are not edge cases; they are the first things new users see.
  • Smart Animate requires matching layer names — name your layers before you build your prototype.
Week 8 — Accessibility & Inclusive Design
W8

Accessibility & Inclusive Design

WCAG 2.1 · POUR Principles · Color Blindness · Screen Readers · Inclusive Design Methods

▾

Learning Objectives

  • Explain the three WCAG 2.1 levels (A, AA, AAA)
  • Apply the POUR principles to evaluate a design
  • Simulate three types of color blindness using Stark
  • Use NVDA (Windows screen reader) to test a web page
  • Explain the inclusive design principle of "design for extremes"

WCAG 2.1 Levels

  • Level A (minimum): Basic accessibility. Without this, some users cannot access content at all. Includes alt text on images, keyboard navigation, no auto-play audio
  • Level AA (standard): Target for all commercial products. Required by law in many countries. Includes contrast ratios (4.5:1 normal text), resize to 200% without loss, visible focus indicators
  • Level AAA (enhanced): Highest standard. Not required for all content. Includes 7:1 contrast ratio, sign language interpretation, no time limits
  • Design target for this program: WCAG 2.1 AA

POUR Principles

  • Perceivable: All users can perceive the content. Alt text for images, captions for video, color is never the only indicator of meaning
  • Operable: All interactive elements can be operated. Keyboard navigation (Tab key), skip links, no seizure-inducing flashes, no time limits on tasks
  • Understandable: Content and UI are understandable. Clear language, descriptive error messages, consistent navigation across pages
  • Robust: Works across assistive technologies. Valid HTML, ARIA attributes correctly applied, tested with real screen readers

Color Blindness — Types & Design Solutions

  • Deuteranopia (most common): Cannot distinguish red from green. Affects 5% of men. Fix: use icons + labels alongside color to convey status
  • Protanopia: Reds appear very dark or black. Fix: never use red text alone on white for errors — add a ⚠ icon
  • Tritanopia (rare): Cannot distinguish blue from green or yellow. Fix: use high contrast + patterns in addition to color
  • Use Stark plugin: Simulate → select blindness type → compare your design under each condition

Inclusive Design Principles

  • Design for extremes: Designing for one-handed use (broken arm, holding a baby) benefits all users. Designing for low literacy benefits all users under stress
  • Solve for one, extend to many: Closed captions were created for the deaf — they are now used in noisy offices and by language learners
  • Recognize exclusion: Who are we excluding with this design? Ask this about every design decision
  • Learn from diversity: Designers with disabilities make better accessibility decisions than those without. Build diverse teams
Accessibility Tools Checklist: Stark plugin (Figma) — color contrast + blindness simulation. axe DevTools (Chrome extension) — automated web accessibility audit. NVDA (Windows, free) — screen reader for testing web pages. Chrome Lighthouse (F12 → Lighthouse tab) — accessibility score for any live URL. VoiceOver (Mac/iOS, built-in) — screen reader for Apple platforms.

ARIA Attributes — When to Use

  • ARIA (Accessible Rich Internet Applications) adds meaning to HTML elements that screen readers can use
  • aria-label: Name an element that has no visible text (icon-only button: "Close dialog")
  • aria-expanded: Tell screen readers whether an accordion is open or collapsed
  • aria-live: Announce dynamic content changes (toast notifications, search results updating)
  • Rule: first use native HTML semantics (button, nav, main, header). Only add ARIA when native HTML cannot do the job

Focus Indicators — Why They Matter

  • Keyboard users navigate with Tab — the focus ring shows them where they are
  • Many designs remove focus rings because they look "ugly" — this breaks keyboard navigation
  • WCAG 2.1 AA requires visible focus indicators on all interactive elements
  • Design beautiful focus rings: 2px solid outline, 2px offset, using your brand's primary color
  • Never use outline: none in CSS without providing a custom focus style as a replacement

Key Takeaway — Week 8

  • WCAG 2.1 AA is the legal and ethical minimum. Treat it as a design constraint, not an afterthought.
  • POUR: Perceivable, Operable, Understandable, Robust — a checklist for every screen you design.
  • Never rely on color alone. Always pair color with an icon, label, or pattern.
  • Design for the most constrained user — everyone else benefits.
