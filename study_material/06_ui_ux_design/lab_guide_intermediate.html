Week 5

Figma Components & Variants

▼

Lab Objective

Build reusable Figma components with multiple states using the Variants system so you can design consistently at scale.

Step-by-Step Instructions

  1. Create Figma account & new file
    Sign up at figma.com (free plan is sufficient). Click New design file and name it My Design System. Rename the default page to Components.
  2. Create a Button master component
    Go to View > Rulers to enable snapping. Draw a rectangle: 160 × 44 px. Add a text layer inside it with the text Label. Select both layers, apply Auto Layout (Shift+A). Set horizontal padding to 0 24 px, align centre. Set fill colour to #1B3A6B, corner radius to 8 px. Select the frame, press Ctrl+Alt+K (Windows) to turn it into a Component. Name it Button.
  3. Add 4 Variants for the Button
    With the component selected, click Add variant four times in the right panel. This creates a Variant Set. Open the Variant Properties panel and rename the property to State. Name each variant:
    • Default — fill #1B3A6B, text white, opacity 100%
    • Hover — fill #14305A (darker), subtle drop shadow
    • Disabled — fill #9EB0C8, text #6B8099, cursor not-allowed
    • Loading — fill #1B3A6B, replace label text with a spinner icon (use ⟳ or a circular shape with a 90° rotation)
  4. Create an Input component
    Draw a rectangle 280 × 44 px, stroke #C4C9D4 (1 px), corner radius 6, no fill. Add placeholder text in light grey (#9EA8B3). Apply Auto Layout. Create a Component, then add 4 variants with State property:
    • Empty — placeholder text visible, default border
    • Filled — real text in #1A1A2E, border #1B3A6B
    • Error — border #E76F51 (red), error icon on right, helper text below in red
    • Focused — border #1B3A6B (2 px), subtle glow effect
  5. Create a Card component
    Draw a frame 320 × 380 px, fill white, corner radius 12, drop shadow (0 4px 16px rgba(0,0,0,0.1)). Add three child layers:
    • Image slot: rectangle at top, 320 × 160 px, fill #E8ECF0 (placeholder)
    • Title text: H3 style, bold, 16px
    • Body text: 14px, #4A5568, 2 lines
    • CTA Button: drag an instance of your Button component into the card
    Apply Auto Layout to the content area (below image) with 16 px gap. Convert to Component.
  6. Organise on the Components page
    Lay out all component sets neatly with labels above each group. Add a section frame for Buttons, Inputs, Cards. Group related variants visually.
  7. Create a Playground page
    Add a new page named Playground. Drag instances of all your components onto this page. In the right panel, practice swapping the State variant on your Button — change from Default to Hover to Disabled without breaking the layout. This demonstrates the power of components: one change updates all instances.

Deliverable Check

  • Button component with 4 states (variants)
  • Input component with 4 states
  • Card component using a Button instance
  • Playground page showing instances with swapped variants
Week 6

Building a Mini Design System

▼

Lab Objective

Create a shareable, documented design system in Figma that another designer or developer can pick up without asking you questions.

Step-by-Step Instructions

  1. Create a Brand page with Color Styles
    Add a new page named Brand. Create 6 colour swatches (rectangles 80×80 px) with labels below each:
    • Navy — #1B3A6B (Primary)
    • Gold — #C8941A (Accent)
    • Cream — #F8F6F0 (Background)
    • Success — #2A9D8F
    • Warning — #E9C46A
    • Error — #E76F51
    For each swatch: select it, click the Style icon (4 dots) next to Fill in the right panel, click +, name the style (e.g. Primary/Navy). These are now shared Colour Styles across your file.
  2. Create Text Styles
    On the same Brand page, create text samples and save as Text Styles:
    • H1 — 32px / Bold (700) / Line height 40px
    • H2 — 24px / SemiBold (600) / Line height 32px
    • H3 — 18px / SemiBold (600) / Line height 28px
    • Body — 14px / Regular (400) / Line height 22px
    • Caption — 12px / Regular (400) / Line height 18px
    • Button — 14px / Bold (700) / Letter spacing 0.5px
    Save each as a Text Style using the same Style panel method.
  3. Build 8 core components
    Add a Components page. Create the following, each using your saved Colour and Text Styles:
    1. Button — 3 types (Primary / Secondary / Ghost) × 3 sizes (SM/MD/LG) × 4 states (Default/Hover/Disabled/Loading) = 36 variants
    2. Input — with label, placeholder, helper text, error state
    3. Card — image, title, body, CTA
    4. Badge — 3 statuses: Success (green), Warning (yellow), Error (red)
    5. Avatar — 3 sizes: SM (24px), MD (40px), LG (64px); with image + initials fallback
    6. Navbar — logo left, links centre/right, hamburger for mobile
    7. Modal Dialog — overlay, title, body text, confirm/cancel buttons
    8. Toast Notification — 3 types: success, error, info; icon + message + dismiss X
  4. Create a Documentation page
    Add a page named Docs. For each component, create a documentation block with:
    • Usage: When should you use this component?
    • Do (green tick frame): show the correct usage
    • Don't (red cross frame): show a common misuse
    Example for Button: Do — use a primary button for the main action per screen. Don't — use 3 primary buttons side-by-side (creates visual confusion).
  5. Handoff test
    Share the Figma file link with a classmate. Ask them to build a simple login form using only your components. If they ask more than 2 questions, your documentation needs more detail. Update accordingly.

Deliverable Check

  • Brand page with 6 Colour Styles + 6 Text Styles saved
  • 8 components on Components page, using Styles consistently
  • Documentation page with Do/Don't for each component
  • File shared with a classmate for handoff test
Week 7

Interactive Prototype — Mobile Checkout Flow

▼

Lab Objective

Build a fully interactive Figma prototype covering a 6-screen mobile checkout flow, then test it with real users to find friction points.

Step-by-Step Instructions

  1. Design a 6-screen checkout flow
    Create 6 frames sized 390 × 844 px (iPhone 14 Pro). Name them:
    1. Cart
    2. Delivery Address
    3. Payment Method
    4. Order Review
    5. Processing
    6. Confirmation
  2. Screen 1 — Cart
    Design a vertical list of cart items (2–3 product rows). Each row: product image (48×48), name + variant text, quantity controls (– / 2 / +), price. Below the list: subtotal, delivery fee, total. At the bottom: a sticky "Checkout" CTA button spanning full width. Use your Button component (Primary / LG state).
  3. Screen 2 — Delivery Address
    Form with 5 Input fields: Full Name, Phone Number, Street Address, City, PIN Code. Above the form: a row showing saved addresses as chips (tap to pre-fill form). Use your Input component. Bottom CTA: "Continue to Payment".
  4. Screen 3 — Payment Method
    Three radio-style payment options: Credit / Debit Card, UPI, Cash on Delivery. When "Credit / Debit Card" is selected (use Variants/State to show selected radio), show a card input sub-form: card number, name on card, expiry, CVV. UPI shows a UPI ID input. COD shows a confirmation message. Bottom CTA: "Review Order".
  5. Screen 4 — Order Review
    Full order summary: items list (condensed), delivery address block with "Edit" link, payment method with "Edit" link, total amount. Bottom CTA: "Place Order" (use a distinct, high-contrast button here — this is the final commitment action).
  6. Screen 5 — Processing
    Centre-aligned spinner animation (circular arc with rotation). Below it: "Placing your order..." in Body text style. Minimal screen — no navigation. This screen should auto-advance.
  7. Screen 6 — Confirmation
    Large success checkmark (circle + tick, use your Success colour #2A9D8F). Heading: "Order Placed!" Sub-text: "Your order #ND2024-0042 has been confirmed." Two buttons: "Track Order" (Primary) and "Continue Shopping" (Ghost/Secondary).
  8. Wire up the prototype
    Switch to Prototype mode in Figma (top-right panel). Connect each screen:
    • Cart → Delivery Address: on tap "Checkout" button, Navigate To, Slide Left, Spring easing
    • Delivery Address → Payment: on tap "Continue to Payment", Slide Left
    • Payment → Review: on tap "Review Order", Slide Left
    • Review → Processing: on tap "Place Order", Dissolve
    • Processing → Confirmation: After Delay 1500ms, Navigate To Confirmation, Dissolve
    • Confirmation "Continue Shopping" → Cart: Slide Right (reset flow)
  9. Add Smart Animate transitions
    For the Payment screen: create two variants of the payment section — one with card form hidden, one with it visible. Connect the "Credit Card" radio tap to swap variant using Smart Animate, Spring easing (this smoothly animates the form appearing).
  10. Test with classmates
    Share the prototype link (Figma → Present mode → Copy link). Give it to 2 classmates. Watch where they hesitate:
    • Do they know the CVV field means?
    • Is "Review Order" button label clear?
    • Does the Processing screen cause anxiety (no progress feedback)?
    Note down every hesitation moment — these become your Week 8 usability test tasks.

Deliverable Check

  • 6 screens designed at 390×844 px, using your design system components
  • Prototype wired with Slide Left + After Delay transition on Processing screen
  • Smart Animate on payment method selection
  • Shared with 2 classmates, hesitation moments noted
Week 8

Moderated Usability Test

▼

Lab Objective

Run a structured, moderated usability test on your checkout prototype. Collect real evidence, synthesise findings, and produce a written report with prioritised recommendations.

Step-by-Step Instructions

  1. Write the test plan
    Document the following before testing:
    • Goal: Identify usability issues in the checkout flow prototype
    • Participants: 3 classmates who have not seen your design
    • Session length: 20 minutes each
    • 3 Tasks:
      1. "Add 2 items to your cart and proceed to checkout"
      2. "Change the delivery address after you have entered payment method"
      3. "Find your order confirmation number after placing the order"
    • Success criteria: Task completed without help = pass, completed with hint = partial, could not complete = fail
  2. Create a consent statement
    For classroom context, a verbal statement is sufficient: "I am testing the design, not you. There are no wrong answers. Please think aloud — say whatever comes to mind. I may take notes but will not interrupt you." Ask each participant to verbally agree before starting.
  3. Set up recording
    Use Zoom or Google Meet (free). Share your screen showing the Figma prototype in Present mode. Ask the participant to unmute and speak. Enable recording (inform participant). Alternatively, have a second person observe and take notes while you facilitate.
  4. Run 3 sessions (20 minutes each)
    Script for each session:
    • Introduction (2 min): Explain purpose, get verbal consent, remind them to think aloud
    • Task 1 (5 min): Read task aloud, start timer, observe + note
    • Task 2 (5 min): Same procedure
    • Task 3 (5 min): Same procedure
    • Debrief (3 min): Ask: "What was most confusing?" "What did you expect to happen on the payment screen?" "Would you use this app?"
  5. Record observations
    For each participant and each task, capture:
    • Task completion: ✅ Yes / ⚠️ Partial / ❌ No
    • Time on task (use phone stopwatch)
    • Errors observed (wrong taps, confusion moments)
    • Think-aloud quotes (write verbatim, e.g. "I expected the back button to be here")
  6. Create an Affinity Map
    After all 3 sessions: write each observation / quote on a separate sticky note (use Figma FigJam, Google Jamboard, or physical post-its). Group them into themes. Common themes will emerge: navigation confusion, label clarity, form anxiety, success state reassurance.
  7. Write a Usability Test Report (500–700 words)
    Structure:
    • Methodology: Who, when, how — 3 participants, moderated remote, 3 tasks
    • Participants: Brief description (no names — P1, P2, P3), tech comfort level
    • Findings: Top 3 issues with severity rating (Critical / Moderate / Minor) and supporting quotes + task completion data
    • Recommendations: One specific design change per issue
  8. Prioritise fixes with a 2×2 Matrix
    Draw a 2×2 grid: X axis = Effort (Low → High), Y axis = Impact (Low → High). Plot each recommended fix. Items in the High Impact / Low Effort quadrant (top-left) must be fixed first. Items in High Impact / High Effort go on the backlog.

Deliverable Check

  • Written test plan with 3 tasks and success criteria
  • Data from 3 participant sessions (task completion + time + quotes)
  • Affinity map with grouped themes
  • Usability test report (methodology → findings → recommendations)
  • 2×2 prioritisation matrix
