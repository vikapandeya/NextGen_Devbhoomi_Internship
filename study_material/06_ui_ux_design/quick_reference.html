Quick Reference — UI/UX Design

All the essential checklists, heuristics, shortcuts, and templates in one printable page. Keep this open during design reviews and portfolio prep.

1. Nielsen's 10 Usability Heuristics

#HeuristicIn Practice
1 Visibility of System Status Loading spinners, progress bars, save confirmations, "Step 3 of 5" indicators, upload progress percentages
2 Match Between System & Real World Shopping cart icon, inbox metaphor, familiar terminology — use words users say, not internal product names
3 User Control & Freedom Undo button, Cancel option, back navigation always accessible, "Exit" clearly visible in modals
4 Consistency & Standards Same button label for same action throughout, consistent icon set, "Delete" always means delete (not Remove sometimes)
5 Error Prevention Disable submit until form is valid, confirm before delete, input masks for phone/date, inline validation before submission
6 Recognition Over Recall Show recent items, autocomplete suggestions, visible options in menus — don't make users memorise commands or paths
7 Flexibility & Efficiency Keyboard shortcuts for power users, saved preferences, quick-action buttons, bulk operations, personalised defaults
8 Aesthetic & Minimal Design Every element must earn its place — remove anything that doesn't directly serve a user task. Content competes with content.
9 Help Users Recognise & Recover From Errors "Email is invalid — please include @" not "Error 400". Plain language, specific problem, clear fix instructions
10 Help & Documentation Contextual tooltips on hover, searchable help centre, onboarding walkthrough for first-time users, in-context FAQs

2. WCAG 2.1 Accessibility Quick Check

CheckPass CriteriaTool
Normal text contrast ≥ 4.5:1 ratio Stark (Figma plugin), WebAIM Contrast Checker
Large text (18px+ or 14px+ bold) ≥ 3:1 ratio Stark, axe DevTools browser extension
Focus indicators Visible on all interactive elements Manual keyboard test (Tab through the page)
Images Descriptive alt text present on all meaningful images Chrome Accessibility panel (F12 → Accessibility)
Form labels Each input has a visible, associated label element axe DevTools, Chrome Accessibility panel
Colour as the only indicator Must have a secondary indicator (icon, text, or pattern) Colour blindness simulator (Figma: View → Colour Blindness)
Touch targets Minimum 44 × 44pt for interactive elements Figma measurement tool, Stark
Video / audio content Captions provided for all spoken content Manual check

3. Design Principles Quick Reference

Contrast
Make the important thing stand out. CTAs must be visually dominant — the most important action on the screen should have the highest contrast and visual weight.
Repetition
Repeat visual elements to create consistency and unity. Same button style, same icon family, same border radius throughout — variation signals meaning.
Alignment
Every element should align to an invisible grid. Left-align body text in Latin scripts. Misaligned elements create visual noise and feel unpolished.
Proximity
Related things belong close together. Price + CTA + quantity selector should form one group. Unrelated things need clear space between them.
Hierarchy
Guide the eye: biggest / boldest / highest-contrast = most important. The user should never have to decide where to look first — the design should direct them.
Whitespace
Empty space is a design material. Filling every pixel is a design failure, not a virtue. Let content breathe — whitespace focuses attention and reduces cognitive load.
Affordance
Things that do something should look like they do it. Buttons look tappable, links look clickable, sliders look draggable. Avoid invisible interactions.
Feedback
Every user action deserves a system response. Button press = visual change. Form submit = confirmation. Upload = progress indicator. Silence causes anxiety and re-taps.

6. Common UX Deliverables Checklist

DeliverableMust Include
User Persona Name, photo placeholder, age, occupation, 2–3 goals, 2–3 frustrations, a verbatim quote, tech comfort level
User Journey Map Stages, user actions per stage, thoughts, emotional curve, pain points, design opportunities per stage
Sitemap Full hierarchy of all screens/pages, navigation relationships, section groupings clearly labelled
User Flow Task-specific path, start and end points, decision diamonds with yes/no branches, error paths included
Lo-Fi Wireframes All screens in the core flow, greyscale only, annotations explaining key interactions and content areas
Hi-Fi Mockups Full colour, real content (no Lorem Ipsum), consistent design system, all screens and states represented
Figma Prototype Clickable frames covering the core flow end-to-end, transitions set, must work without guidance
Usability Test Report Participants overview, tasks + task completion rates, top 3–5 findings with severity, recommendations
Design System Colour + text styles, spacing tokens, 5+ components with variants, anatomy labels, usage documentation
Developer Handoff Spec Spacing grid annotated, assets exported and named, component inventory, CSS tokens documented

7. UX Case Study Template

Title: [Project Name] — [1-line description of what you achieved]

01. Context
    App/website, target users, company/client (or personal project),
    your role, team size, timeline

02. The Problem
    "Users of [product] struggle with [specific issue] because [root cause].
     This results in [measurable impact — time, money, frustration, drop-off]."

03. Research
    Methods: [list 2–3 methods: interviews, usability testing, competitive analysis]
    Key Findings:
    • Finding 1 — [what you learned, with numbers or quotes if possible]
    • Finding 2 — [unexpected insight that shaped your direction]
    • Finding 3 — [pain point confirmed across multiple participants]

04. Key Insight
    "We found that [specific insight], which means [design implication].
     This changed our approach from [original direction] to [new direction]."

05. Ideation
    [Show 3 concepts explored — sketches, low-fi screens, or wireframe variations]
    Explain why you chose the direction you went with.

06. Design Solution
    [Show final hi-fi screens]
    [1–2 sentences of rationale per key design decision — why, not just what]

07. Prototype
    [Figma link — embed or clickable text link]
    [Note: the core flow must be testable without guidance]

08. Testing & Iteration
    [What you tested (which tasks?), who (participants), when]
    [What changed after testing — show before/after if possible]
    [Evidence of improvement: task completion rate went from X% to Y%]

09. Outcome
    [Metrics improved (if available) — task completion, time-on-task, SUS score]
    [Key lessons — what surprised you, what you'd do differently]
    [What comes next if the project continued]
Portfolio tip: The strongest case studies start with a problem that makes the reader think "yes, I've experienced that frustration" — before showing a single screen. The screens are the evidence, not the story.
