Week 9 — Advanced Research & Strategy
W9

Advanced Research & Strategy

JTBD · Mental Models · Competitor Analysis · Kano Model · HEART · UX ROI

▾

Learning Objectives

  • Write a JTBD statement for a given user scenario
  • Conduct a heuristic competitor analysis on 2 products
  • Categorise features using the Kano model
  • Define UX metrics using the HEART framework
  • Build a business case for UX investment (basic UX ROI)

Jobs To Be Done (JTBD)

  • Created by Clayton Christensen. The insight: people don't buy products — they hire them to do a job
  • Framework: "When I _____ [situation], I want to _____ [motivation], so I can _____ [outcome]"
  • Goes deeper than personas — captures the functional, emotional, and social dimensions of a need
  • Example: "When I'm commuting, I want to listen to curated playlists without choosing each song, so I can arrive at work in a focused headspace"
  • Use JTBD to define what the product must do at its core — not the features, but the job

Mental Models

  • A mental model is how a user expects something to work, based on past experience
  • When the product matches the mental model → easy to learn. When it contradicts it → confusion and errors
  • Example: users expect the Back button to undo their last action. Apps that use Back to go to the home screen instead of the previous screen violate this model
  • Research technique: ask users to draw how they think the system works before using it
  • Your job: map your design to users' existing mental models wherever possible

Competitor Analysis — UX Benchmarking

  • Evaluate 3 competitors using Nielsen's 10 heuristics — gives you a comparative UX score
  • Document: screenshots of the key flow, heuristic violations (severity 0–4), unique strengths
  • Competitive matrix: list features/qualities in rows, competitors in columns — identify gaps
  • Distinguish between "table stakes" (features every competitor has) and "differentiators" (where you can stand out)
  • Share findings with stakeholders: a short competitor UX audit is an extremely persuasive strategic document

Kano Model — Feature Prioritisation

  • Created by Noriaki Kano. Three feature categories:
  • Basic (must-have): Users expect these. Their absence causes dissatisfaction. Their presence causes no delight. Example: a working checkout
  • Performance (more = better): More of this → more satisfaction. Loading speed, number of payment options
  • Delighters (excitement): Unexpected features that create delight. Users didn't ask for them but love them. Example: Swiggy's delivery person real-time map
  • Apply: get basics right first, then invest in performance, then add delighters

HEART Framework — UX Metrics

  • Developed by Google. Five categories for measuring UX at scale:
  • Happiness: Attitudinal — NPS, satisfaction surveys, app store ratings
  • Engagement: Behavioral — sessions per user per week, feature adoption rate
  • Adoption: New users who complete onboarding and use the product at least once
  • Retention: Users who return after 1 day, 7 days, 30 days (D1/D7/D30 retention)
  • Task Success: Task completion rate, time on task, error rate per task

UX ROI — The Business Case

  • Every ₹1 invested in UX returns ₹100 in development savings (Forrester Research)
  • Poor UX costs: support ticket volume, cart abandonment, low activation, negative reviews
  • Build the case: calculate the value of a 10% improvement in task completion on the checkout flow → multiply by conversion value × users per month
  • Stakeholder language: use revenue, retention, and support cost — not "user delight" or "beautiful design"
  • Show before/after data from a completed redesign — one clear data point wins more than ten principles
Practice writing JTBD statements: "When I receive my salary, I want to invest a small amount automatically without having to think about which fund, so I can build savings without financial anxiety." This is richer than "The user wants to invest money" — it captures the trigger (salary), the emotional barrier (anxiety around decisions), and the real desired outcome (peace of mind, not investment returns).

Key Takeaway — Week 9

  • JTBD reframes design from "what does the user want" to "what job are they hiring this product for" — a fundamentally more useful question.
  • Kano model gives you a prioritisation framework: basics first, then performance, then delight.
  • HEART provides the metrics language to talk to executives about UX — use it for every project debrief.
  • A competitor UX audit is the fastest way to earn stakeholder trust with evidence.
Week 10 — Advanced Figma & Prototyping
W10

Advanced Figma & Prototyping

Nested Auto Layout · Figma Variables · Conditional Prototypes · Dev Mode · Design Tokens Export

▾

Learning Objectives

  • Build a carousel component using Figma Variables
  • Create a theme-switching prototype using boolean variables
  • Explain Smart Animate requirements and spring easing
  • Use Figma Dev Mode for a complete developer handoff
  • Export design tokens to JSON using Tokens Studio

Advanced Auto Layout

  • Nested auto layouts: A row of cards (horizontal auto layout) where each card is a vertical auto layout. The parent controls spacing between cards; each card controls its own internal spacing
  • Absolute positioning within auto layout: Pin an element (e.g., a badge) to a corner of an auto layout frame without it affecting the layout flow. Right-click layer → Edit → set position to Absolute
  • Fill container: Make a child element stretch to fill the width of its auto layout parent (useful for full-width input fields inside a form)
  • Mastery of nested auto layouts = components that work at any content length

Figma Variables

  • Variables store values that can be reused and changed across a design: numbers, booleans, strings, colors
  • Number variables: Spacing tokens — define --space-md once, apply everywhere
  • Boolean variables: Show/hide layers in a prototype conditionally. "isLoggedIn = true" shows the dashboard; false shows the login screen
  • String variables: Content overrides — change user name or label text across multiple screens by editing one variable
  • Color variables: Light/dark mode switching — define "surface" as #FFFFFF in light mode and #1A1A2E in dark mode, switch with one click

Complex Components

  • Accordion: Boolean variable controls expanded/collapsed state. Smart Animate transitions the height change. Match layer names in both states
  • Stepper (multi-step form): Number variable tracks current step. Each step is a variant. Prototype arrows advance the variable on button click
  • Carousel: Horizontal frame with overflow hidden. Number variable tracks active index. Navigation arrows increment/decrement the variable
  • Rule: the more realistic the prototype, the earlier you catch real usability issues

Smart Animate — Requirements

  • Smart Animate works by finding layers with matching names in the origin and destination frames and animating the difference
  • Layer must have the same name in both frames — if the name differs, Figma treats it as a different element and dissolves instead of morphing
  • Spring easing: More natural than ease-in-out for UI motion. Mimics physics — the element overshoots slightly and settles back. Use for modals appearing and drawers opening
  • Reduced motion: In your live implementation, always check prefers-reduced-motion and disable non-essential animation for users who have it enabled

Figma Dev Mode

  • A dedicated view for developers (not designers) to inspect designs
  • Shows: exact CSS values for every property, spacing between elements, font details, component annotations
  • Export assets: click any image or icon → Export → choose SVG/PNG/WebP + scale
  • Code Connect (advanced): link Figma components to real code components — Dev Mode shows the actual React/Vue component name instead of CSS
  • Reduce back-and-forth by annotating your frames: use the Annotations feature to add written notes developers can read in Dev Mode

Design Tokens with Tokens Studio

  • Tokens Studio is a free Figma plugin that manages design tokens as JSON
  • Define tokens in the plugin (color, spacing, typography, shadow, border-radius)
  • Export to JSON → run through Style Dictionary (Node.js tool) to generate CSS custom properties, SCSS variables, iOS Swift, Android XML simultaneously
  • Benefit: one source of truth for design and engineering — update the token in Figma, the code updates automatically
  • Token naming convention: semantic names (--color-primary) not value names (--blue-600)

Key Takeaway — Week 10

  • Figma Variables are the most powerful new feature for prototyping — master conditionals and your prototypes become near-functional.
  • Smart Animate requires matching layer names. Name your layers before building any animated prototype.
  • Dev Mode is how design gets to engineering without a meeting. Annotate everything.
  • Design tokens are the bridge between Figma and code. Use semantic names, not value names.
Week 11 — UX Writing & Content Design
W11

UX Writing & Content Design

Microcopy · Button Labels · Error Messages · Tone of Voice · Localization · Content Audit

▾

Learning Objectives

  • Distinguish UX writing from copywriting with 3 examples
  • Rewrite 3 bad button labels and 2 bad error messages
  • Create a tone of voice chart with do/don't examples
  • Explain what designing for RTL languages requires in Figma
  • Annotate a Figma design with UX copy notes for developers
UX Writing vs Copywriting: A copywriter writes the homepage hero: "Reach millions of customers instantly." A UX writer writes what the button says when you hover: "Send message." Copywriting sells the product from the outside. UX writing serves the user inside the product — it is the words on every button, label, tooltip, empty state, error message, and confirmation toast. Both matter. They require different skills.

Principles of Good UX Writing

  • Clear: The user knows exactly what will happen when they tap this button
  • Concise: Remove every word that doesn't serve the user's task. "Click here to proceed to the next step" → "Continue"
  • Useful: Copy helps the user complete a task. It is not decoration. Every word earns its place
  • Conversational: Write the way a helpful colleague would speak, not a legal document
  • Consistent: If you call it "Save" in one screen, never call it "Submit" in another for the same action

Writing for UI Elements

  • Button labels: Use verb + noun. "Save changes" not "OK". "Delete account" not "Confirm". Be specific about what happens
  • Error messages: Three parts: what went wrong + why + how to fix it. "Your session expired. Please log in again to continue."
  • Empty states: Explain why it is empty + invite action. "No orders yet. Browse our menu to place your first order." + CTA button
  • Onboarding: Teach by doing. Don't front-load instructions — surface help at the moment of action

Localization Considerations

  • RTL languages (Arabic, Hebrew, Urdu): Text direction flips; UI layout mirrors. Buttons move from right to left. Back arrows point right. Design RTL-ready from the start if these markets matter
  • Text expansion: German translations average 30% longer than English. Hindi/Bengali average 20% longer. Design flexible layouts (Auto Layout) that handle expansion
  • Indian languages: Devanagari, Tamil, Telugu scripts need larger line heights and correct font support — test your fonts with regional scripts before launch
  • Always design with real translated text — never test with English if the product ships in Hindi

Content Audit

  • An inventory of all UI strings in a product: labels, headings, error messages, tooltips, empty states
  • Audit for: consistency (same action = same label), clarity (no jargon), completeness (every state has copy)
  • Spreadsheet format: Screen | Element | Current copy | Issue | Proposed copy | Priority
  • Run a content audit when: redesigning a product, preparing for localization, onboarding a new UX writer
  • In Figma: use the Annotations feature to add copy notes that appear in Dev Mode

Key Takeaway — Week 11

  • UX writing is not "the words in the design" — it is a design discipline. Treat every label as a design decision.
  • Button labels are verb + noun: "Save draft", "Send message", "Delete account". Never "Submit" for everything.
  • Error messages must explain what went wrong AND how to fix it. "Error" alone is useless.
  • Design with real translated text from day one if you ship in multiple languages. Auto Layout handles expansion.
Week 12 — UX Leadership & Portfolio
W12

UX Leadership & Portfolio

Case Study Structure · What Hiring Managers Look For · Portfolio Platforms · Design Critique · Agile UX · Career Paths

▾

Learning Objectives

  • Structure a complete UX case study (10-part format)
  • Deliver a 5-minute portfolio walkthrough confidently
  • Give a critique using the "I like / I wish / What if" format
  • Explain how design works in an Agile sprint cycle
  • Describe 3 career paths in UX and their progression milestones

UX Case Study Structure

  • 1. Title + Context: What product, what company, when, what was your role
  • 2. Problem statement: "How might we _____ for _____ so they can _____?"
  • 3. Your role + team + timeline
  • 4. Research: Methods used, 3 key findings
  • 5. Synthesis: The insight that drove the design direction
  • 6. Ideation: 3 concepts you considered (not just the final one)
  • 7. Design: Final screens with rationale for key decisions
  • 8. Prototype: Link to interactive prototype
  • 9. Testing: What you tested, what you changed, why
  • 10. Outcome: Metrics (if available) — TCR, NPS, conversion lift

What Hiring Managers Look For

  • Problem framing: Did you solve the right problem? Research evidence, not assumptions
  • Decision rationale: Why did you choose this approach over alternatives? Show your thinking
  • Iteration evidence: Before/after showing how testing changed your design
  • Metrics: Even rough data ("TCR improved from 60% to 87%") beats no data
  • Communication: Can you explain your work clearly to a non-designer? Try your portfolio on a family member
  • Portfolios with 2 great case studies beat portfolios with 10 shallow ones

Design Critique Culture

  • Good critique is specific, actionable, and respectful. It critiques the work, never the person
  • "I like": What is working and why. Specific: "I like how the CTA has high contrast" not just "looks good"
  • "I wish": A gap or problem framed as a wish, not a command. "I wish the empty state explained what users can do next"
  • "What if": Offer an alternative to explore. "What if the error message appeared inline instead of as a toast?"
  • Set up a weekly design crit in your team. Show work early and often — critique at lo-fi saves time at hi-fi

Working in Agile

  • Designers work 1–2 sprints ahead of developers — design sprint n+1 while devs build sprint n
  • Grooming sessions: Review upcoming user stories with developers — clarify design requirements, agree on edge cases
  • Design debt: Quick-and-dirty designs made under time pressure that need rework later. Track it. Pay it down deliberately
  • Definition of Done (UX): Screen has all states (loading/empty/error), annotations are complete, accessibility is checked, developer questions are answered

Key Takeaway — Week 12

  • A portfolio is not a gallery — it is evidence of how you think. Start with the problem, not the pretty screens.
  • Case studies need decision rationale. "I chose this because the research showed..." beats "I chose this because I liked it."
  • Design critique is the culture that separates great teams from average ones. Do it regularly. Do it respectfully.
  • In Agile, design runs one sprint ahead. Never start a sprint without designs that are ready. Design debt compounds.
