Week 9

Condensed Design Sprint

▼

Lab Objective

Experience the GV Design Sprint methodology in a compressed format. Go from problem to tested prototype on a real Indian product in under a week of part-time work.

The Challenge

Redesign the IRCTC train ticket booking flow specifically for a first-time user — someone booking on IRCTC for the very first time, possibly not tech-savvy.

Step-by-Step Sprint

  1. Day 1 — Map (1 hour)
    Draw a simple journey map on paper showing the current IRCTC flow:
    User opens app → searches trains → views results → picks a train → fills passenger details → selects berth → makes payment → gets ticket

    Mark each step with: 😊 Easy / 😐 OK / 😤 Frustrating based on your own experience or by asking 2 friends. Circle the most critical moment — this is typically the "Pick Train" results screen where users face an overwhelming grid of trains, classes, prices, and availability codes (SL, 3A, 2A, 1A). Write a one-sentence "How Might We" statement: "How might we help a first-time user quickly find the right train at the right price without being overwhelmed?"
  2. Day 2 — Crazy 8s (30 minutes)
    Fold an A4 paper into 8 equal sections. Set a timer for 8 minutes. Sketch 8 different UI concepts for the "Pick Train" results screen — one per section. Speed matters more than polish here. Try wildly different layouts: list vs cards vs map, sorting prominence, filter placement, price visualisation. Do not self-censor. After the 8 minutes, pick your top 2 by placing a dot sticker (or pen dot) on them.
  3. Day 3 — Storyboard (45 minutes)
    Pick your best sketch from Crazy 8s. Extend it into a 5-panel storyboard covering the full redesigned booking flow:
    1. Search screen (clear inputs, date picker)
    2. Train results (your redesigned layout)
    3. Seat class selection (visual, with price + availability)
    4. Passenger details (simplified form)
    5. Payment + confirmation
    Sketch at medium fidelity — enough detail that someone else understands the layout and key interactions.
  4. Day 4 — Prototype in Figma (2 hours)
    Build a realistic Figma prototype focusing on:
    • The "Pick Train" results screen: Use real train names (Rajdhani Express, Shatabdi, Jan Shatabdi), real departure times, real class codes with their full names (SL = Sleeper, 3A = 3rd AC). Show availability visually — green for available, red for waitlisted, grey for not available. Display price prominently.
    • The payment screen: Show UPI as the most prominent option (highest adoption in India), followed by cards, net banking.
    Use real content — fake Latin placeholder text undermines the test. Connect these two screens in prototype mode with at least one interactive element (filter tap, train row tap).
  5. Day 5 — Test (1 hour)
    Find 3 participants who have NOT used IRCTC recently (ideally someone from a non-tech background). Give them one task: "Book a 3AC seat from Delhi to Mumbai for next Friday under ₹1500."

    Observe without helping. Note:
    • Do they understand the class codes (3A, SL, etc.)?
    • Can they find a train under ₹1500?
    • Where do they hesitate or tap incorrectly?
  6. Sprint Deliverable
    Submit:
    • Figma file (journey map + prototype) — share view link
    • 200-word sprint summary: What problem did you solve? What did you discover in testing? What would you do differently in a full sprint?
Week 10

Design System Documentation & Tokens

▼

Lab Objective

Elevate your Week 6 design system into a production-ready, token-based system with full developer handoff documentation.

Step-by-Step Instructions

  1. Add a Tokens page to your design system
    Open your Week 6 Figma file. Add a new page named Tokens. Define all spacing values as multiples of 4px. Create a visual swatch for each:
    Token NameValueUse Case
    space-14pxIcon padding, tight gaps
    space-28pxSmall component padding
    space-312pxInput padding vertical
    space-416pxCard inner padding, section gaps
    space-520pxMedium component spacing
    space-624pxButton padding horizontal
    space-832pxSection dividers
    space-1040pxLarge section gaps
    space-1248pxPage-level padding
    space-1664pxHero sections, large gaps
    space-2080pxSection breaks
    space-2496pxPage header height
  2. Install Tokens Studio for Figma
    In Figma: Resources (Shift+I) → Plugins → search "Tokens Studio for Figma" → Install. Open the plugin. You will see a JSON token editor.
  3. Create token sets
    In Tokens Studio, create two token sets:

    global.json — primitive values: 
    {
  "color": {
    "navy": {"value": "#1B3A6B", "type": "color"},
    "gold": {"value": "#C8941A", "type": "color"},
    "cream": {"value": "#F8F6F0", "type": "color"},
    "success": {"value": "#2A9D8F", "type": "color"},
    "warning": {"value": "#E9C46A", "type": "color"},
    "error": {"value": "#E76F51", "type": "color"}
  },
  "spacing": {
    "1": {"value": "4", "type": "spacing"},
    "2": {"value": "8", "type": "spacing"},
    "4": {"value": "16", "type": "spacing"},
    "6": {"value": "24", "type": "spacing"}
  }
}
    semantic.json — mapped semantic tokens: 
    {
  "background": {"value": "{color.cream}", "type": "color"},
  "text-primary": {"value": "{color.navy}", "type": "color"},
  "primary": {"value": "{color.navy}", "type": "color"},
  "accent": {"value": "{color.gold}", "type": "color"},
  "border": {"value": "#D1D5DB", "type": "color"}
}
  4. Export tokens to CSS custom properties
    Use Tokens Studio Export → JSON. Then manually convert to CSS: 
    :root {
  /* Colors */
  --color-primary: #1B3A6B;
  --color-accent:  #C8941A;
  --color-bg:      #F8F6F0;
  --color-success: #2A9D8F;
  --color-warning: #E9C46A;
  --color-error:   #E76F51;

  /* Spacing */
  --space-1: 4px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;
  --space-6: 24px;
  --space-8: 32px;
  --space-12: 48px;
}
  5. Document each component with Anatomy labels
    For each of your 8 components, add a documentation block in Figma showing:
    • Anatomy: Frame with labelled arrows pointing to each part (e.g. for Button: Container, Label, Leading Icon, Trailing Icon)
    • States diagram: All states in a row with labels
    • Accessibility notes: Minimum touch target, colour contrast ratio, keyboard interaction
    • When to use: 2–3 bullet points
  6. Developer handoff test
    Share the design system URL. Pretend you are unavailable for 24 hours. Can a developer build a button from scratch using only your Figma file and the CSS tokens? If not, what is missing? Add it.
Week 11

Implement Your Design in Code

▼

Lab Objective

Bridge the gap between design and development by implementing one of your Figma screens as pixel-accurate HTML/CSS. This is the single most effective exercise for improving your handoff skills.

Step-by-Step Instructions

  1. Choose your screen
    Pick your best-designed mobile screen from any previous lab (the Checkout Cart or Confirmation screen works well). This should be a screen you are proud of — you will use it in your portfolio.
  2. Set up the project
    Create a folder: ux-code-challenge/. Inside: index.html, style.css. No frameworks (no Bootstrap, no Tailwind). The goal is to understand CSS deeply.
  3. Measure from Figma Inspect panel
    Click each element in Figma, open the Inspect tab (right panel). Extract:
    • Exact pixel values for padding and margins
    • Font size, weight, line-height, letter-spacing
    • Hex colour codes (copy directly from colour style)
    • Border radius, border width, box-shadow values
  4. Write CSS custom properties first
    Before writing any layout CSS, declare all values as variables: 
    :root {
  --primary: #1B3A6B;
  --accent:  #C8941A;
  --text:    #1A1A2E;
  --muted:   #6B7280;
  --bg:      #F8F6F0;
  --radius:  8px;
  --shadow:  0 4px 16px rgba(0,0,0,.08);

  --font-h3: 600 18px/28px system-ui, sans-serif;
  --font-body: 400 14px/22px system-ui, sans-serif;
}
  5. Build the layout in HTML/CSS
    Target a 390px wide mobile frame. Use:
    • Flexbox for component interiors (button label + icon, form fields + label)
    • Block flow for vertical page structure
    • Fixed bottom CTA: position: fixed; bottom: 0; left: 0; right: 0;
  6. Side-by-side comparison
    Take a screenshot of your Figma design. Take a screenshot of your browser at 390px width. Put them side-by-side (in Figma, Canva, or literally on your screen). Measure the differences. Common gaps:
    • Letter-spacing: Figma shows it as a number (e.g. 50). CSS uses em: divide by 1000 → letter-spacing: 0.05em
    • Line-height: Figma shows px (28px). CSS prefers unitless ratio: 28/18 = 1.56 → line-height: 1.56
    • Font rendering: slightly different on macOS vs Windows
  7. Add focus states for accessibility
    Add to your CSS: 
    :focus-visible {
  outline: 2px solid #C8941A;
  outline-offset: 2px;
}
button:focus-visible,
a:focus-visible,
input:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
  border-radius: var(--radius);
}
    Test by pressing Tab on your page — every interactive element should show a visible focus indicator.
  8. Lighthouse Accessibility audit
    In Chrome: F12 → Lighthouse tab → select Accessibility → Analyse page load. Fix all issues until score is ≥ 90. Common fixes:
    • Add alt attribute to all <img> tags
    • Add <label> elements for all form inputs
    • Ensure colour contrast ≥ 4.5:1 for body text (check with WebAIM Contrast Checker)
    • Add role="main" or semantic HTML5 elements (<main>, <nav>)
  9. Mobile test in DevTools
    Chrome DevTools (F12) → click the device toggle icon → select 360×800 (Android common) → check the layout. Does text overflow? Is the CTA button accessible? Do images scale correctly? Fix any issues found.

Deliverable Check

  • HTML/CSS implementation of one Figma screen (no framework)
  • Side-by-side comparison screenshot showing close match
  • Focus states visible on Tab navigation
  • Lighthouse Accessibility score ≥ 90
  • Renders correctly at 360×800 px
Week 12

UX Case Study & Portfolio

▼

Lab Objective

Transform your best project into a published UX case study. Build a portfolio that gets you interviews — not just one that looks nice.

Step-by-Step Instructions

  1. Choose your best project
    Select one project from this programme where you have the most complete process documentation: research findings, wireframes, prototypes, and test results. The Mobile Checkout App (Weeks 7–8) or the Design Sprint (Week 9) are strong choices if you have all artefacts.
  2. Write the case study using this exact template

    Title + 1-line summary
    Example: Redesigning IRCTC Checkout — Reducing booking time for first-time users by 40%

    The Problem (1 paragraph)
    Who is the user? What specific problem are they experiencing? Why does it matter?
    Example: First-time IRCTC users spend an average of 12 minutes booking a ticket because the train results screen shows 8 columns of data with no clear prioritisation. Many abandon the process when they encounter waitlist codes they do not understand.

    My Role
    Your responsibilities, team size (solo or pair), timeline (e.g. 5-day sprint in Week 9)

    Research
    Methods: usability observation, think-aloud, journey mapping
    Key Findings (3 bullets): e.g. "6 of 8 participants did not understand 'GNWL' waitlist code"

    Key Insight
    The single finding that reframed your design direction.
    Example: "Users were not overwhelmed by the number of trains — they were overwhelmed by having to decode availability codes while also comparing prices. Separating these two decisions fixed 70% of the confusion."

    Ideation — show 3 concepts you explored (Crazy 8s sketches)

    Design Solution — show final screens with 1-sentence rationale per key decision

    Prototype — embed Figma link (embed code or clickable link)

    Testing & Iteration — what you tested, what changed, evidence of improvement

    Outcome — metrics if available, key lessons, what you would do differently

  3. Publish on Behance
    Create a free account at behance.net. Click "Create Project". Upload your case study as a series of images (export Figma frames as PNG at 2×). Fill in all project details: title, description, tools used, tags (UX Design, Figma, Usability Testing, Mobile, India). Set visibility to Public.
  4. Create a Notion portfolio as backup
    notion.so → New page → "Portfolio". Add a gallery view. For each project: cover image, case study content in collapsible sections, embedded Figma link, Behance link. Share as a public page and get the URL.
  5. Apply the portfolio checklist
    Review your portfolio against these criteria:
    • ✅ First 3 seconds: is it clear who you are and what you do?
    • ✅ At least 2 full case studies showing end-to-end process
    • ✅ Real process shown — research findings, sketches, iterations
    • ✅ Contact information visible on the home page
    • ✅ Each case study starts with the problem, not the final screens
    • ✅ Metrics or user quotes wherever possible (even from classroom testing)
  6. Get feedback and revise
    Share your Behance link with 2 classmates. Ask them: "After reading this, can you tell me what the design problem was? Does my solution make sense? Is anything unclear?" Revise based on at least one piece of their feedback before submitting.

Deliverable Check

  • Published case study on Behance (public URL)
  • Notion portfolio page as backup (public URL)
  • Case study follows the provided template structure
  • Revised based on peer feedback (note what changed)
