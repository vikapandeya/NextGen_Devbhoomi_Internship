Week 9 — Red Team vs Blue Team
W9

Red Team vs Blue Team

Attack simulation · Defense & detection · Purple team · MITRE ATT&CK

▼

Red Team

  • Simulates real attackers using advanced persistent threat (APT) techniques
  • Full attack simulation over weeks — not just a one-day pen test
  • Objective: demonstrate realistic impact (exfiltrate data, reach crown jewels)
  • Uses Living-off-the-Land techniques to avoid detection
  • Provides realistic assessment of whether defenses actually work

Blue Team

  • Defends, monitors, detects, and responds to attacks
  • SOC (Security Operations Center) analysts — 24/7 monitoring
  • Threat hunters — proactively look for signs of compromise
  • IR team — respond when blue team detects something
  • Tunes SIEM detection rules, reviews logs, manages vulnerabilities

Purple Team

  • Red and blue teams working together — not adversarially
  • Red shares TTPs (tactics, techniques, procedures) with blue
  • Blue improves detections based on what red actually did
  • More efficient knowledge transfer than blind red team exercises
  • Growing trend in 2024–2026: most organizations now do purple team workshops

MITRE ATT&CK Framework

  • Knowledge base of adversary tactics and techniques — based on real attacks
  • 14 tactics: Initial Access → Execution → Persistence → … → Exfiltration
  • Hundreds of specific techniques per tactic (e.g., T1566 = Phishing)
  • Use to map your detections to known attack patterns
  • Free at attack.mitre.org — used by every professional SOC globally

Week 9 Key Takeaway

  • MITRE ATT&CK is the industry standard for describing attack techniques.
  • Every SOC and security team uses it to map detections to known adversary behavior.
  • Learn the 14 tactics by name — interviewers ask about this regularly.
Week 10 — Threat Intelligence
W10

Threat Intelligence

Intel Types · IOCs vs TTPs · TI Platforms · CTI Lifecycle

▼

Types of Threat Intelligence

  • Strategic — trends, nation-state activity, for executives and board
  • Tactical — TTPs used by specific threat groups, for security analysts
  • Operational — specific ongoing campaigns targeting your industry
  • Technical — IOCs: IP addresses, file hashes, malicious domains
  • Each level serves a different audience and decision type

IOCs vs TTPs

  • IOCs (Indicators of Compromise) — specific artifacts: IPs, hashes, domains
  • IOCs expire in days or weeks — attackers change infrastructure frequently
  • TTPs (Tactics, Techniques, Procedures) — how attackers operate
  • TTPs last for years — they reflect how a group thinks and behaves
  • Build detections on TTPs for lasting value; IOCs are breadcrumbs

Threat Intelligence Platforms

  • MISP — open source TI sharing platform (used by CERTs globally)
  • VirusTotal — file/URL/IP reputation (free tier available)
  • Shodan — internet-connected device search engine
  • Have I Been Pwned API — check if emails appear in breaches
  • AlienVault OTX — community threat intelligence feeds

CTI Lifecycle

  • Direction — define intelligence requirements from stakeholders
  • Collection — gather raw data from feeds, OSINT, dark web
  • Processing — normalize, deduplicate, structure data
  • Analysis — add context, assess confidence, build narrative
  • Dissemination — share to right audience in right format
  • Feedback — did the intel actually help? Improve the process

Week 10 Key Takeaway

  • IOCs expire in days or weeks — they're breadcrumbs, not paths.
  • TTPs last for years — they reflect how adversaries think. Build detections on TTPs for lasting value.
Week 11 — Cloud Security
W11

Cloud Security

Shared Responsibility · AWS Security Services · IAM · Zero Trust

▼

Shared Responsibility Model

  • AWS responsible: security OF the cloud — hardware, facilities, hypervisor, managed services
  • Customer responsible: security IN the cloud — data, IAM, OS patching, network config
  • Most cloud breaches are customer misconfiguration, not AWS vulnerabilities
  • Common mistakes: public S3 buckets, overprivileged IAM, no MFA on root
  • Understand the model deeply — this is the #1 cloud security concept in interviews

AWS Security Services

  • GuardDuty — ML-based threat detection (crypto mining, unusual API calls)
  • Security Hub — centralized findings from all security services
  • CloudTrail — API audit log for every AWS action (enable day 1)
  • Config — continuously assess configuration compliance
  • Inspector — automated vulnerability scanning for EC2 and containers

IAM Best Practices

  • Never use root account for daily operations — create IAM users
  • Least privilege: grant only what's needed, scope to specific resources
  • Use IAM roles for EC2/Lambda — not access keys embedded in code
  • Rotate access keys every 90 days, review unused keys monthly
  • Enable MFA on all IAM accounts, especially root and admin users

Zero Trust Architecture

  • "Never trust, always verify" — no implicit trust even for internal traffic
  • Every request is authenticated and authorized regardless of network origin
  • Microsegmentation — divide network into small isolated zones
  • Verify device health, user identity, and context on every access request
  • Zero Trust is replacing the perimeter security model across enterprises

Week 11 Key Takeaway

  • 95% of cloud security failures are customer misconfigurations, not AWS/Azure vulnerabilities.
  • Enable CloudTrail and GuardDuty immediately in every AWS account — they're cheap and catch most incidents.
Week 12 — Security Architecture & Compliance
W12

Security Architecture & Compliance

NIST CSF · ISO 27001 · Indian Compliance · Secure SDLC · Architecture Patterns

▼

Security Frameworks

  • NIST CSF: Identify → Protect → Detect → Respond → Recover
  • ISO 27001: Information Security Management System (ISMS) standard; certifiable
  • SOC 2: for SaaS companies; Type I = design, Type II = operating effectiveness
  • PCI DSS: for any company handling payment card data (16 requirements)
  • Frameworks guide what to do; controls define how to do it

Indian Compliance

  • IT Act 2000 + amendments: legal framework for cyber crime in India
  • PDPB 2023: Personal Data Protection Bill — India's GDPR equivalent
  • RBI Cybersecurity Guidelines: mandatory for banks and NBFCs
  • CERT-In Rule: mandatory incident reporting within 6 hours of discovery
  • CERT-In covers 20 incident categories including ransomware and data breaches

Security Architecture Patterns

  • DMZ — demilitarized zone; public-facing servers separated from internal network
  • Bastion host — single hardened SSH entry point into private infrastructure
  • API Gateway — single entry point for all API traffic; authentication, rate limiting
  • WAF — web application firewall; blocks OWASP Top 10 attacks at edge
  • Defense in depth: each layer independent so one breach doesn't cascade

Secure SDLC

  • Design phase: threat modeling (STRIDE) — find flaws before they're built
  • Coding phase: SAST (Static Application Security Testing) in CI/CD pipeline
  • Testing phase: DAST (Dynamic — test running application)
  • Pre-release: penetration test by qualified tester
  • Security built in is 100× cheaper than security bolted on after deployment

Week 12 Key Takeaway

  • Compliance is not security. Following NIST CSF gives you a security program.
  • CERT-In 6-hour reporting is mandatory for Indian companies — know it, and know what triggers it.
  • Threat model during design — fixing architecture later costs 10–100× more.
