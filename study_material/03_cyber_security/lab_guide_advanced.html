Advanced Lab Guide — Cyber Security

4 labs: CTF challenges on TryHackMe, AWS security audit, building a SIEM with ELK Stack, and a capstone red team exercise.

4 Labs Weeks 9–12 TryHackMe · AWS · ELK · Red Team Advanced Level
Lab 9 (Week 9) — HackTheBox / TryHackMe CTF
1

Create TryHackMe Account & Complete Pre-Security Path

~2–3 hours

Create a free account at tryhackme.com. Navigate to Learning Paths → Pre-Security. Complete this path — it covers networking fundamentals, Linux basics, and web fundamentals. This path is required before attacking machines. All rooms run in your browser — no local VM needed.

2

Complete the OWASP Top 10 Room

~3–4 hours

Search for and start the "OWASP Top 10" room on TryHackMe. This guided room walks you through each of the 10 categories with a vulnerable machine for each. Complete all tasks and answer all questions within TryHackMe's guided interface. This reinforces the Week 5 concepts with hands-on practice.

Tip: Connect to TryHackMe via their browser-based AttackBox — no VPN setup required. Or download the OpenVPN configuration from your account and connect from Kali Linux if you have it.
3

Complete "Pickle Rick" CTF Room

~1–2 hours

Search for and start the "Pickle Rick" room — a beginner-friendly CTF. Objective: find 3 flags (text strings) hidden on the machine. No hints for this one — figure it out. Approach: enumerate the web server, find hidden files, use web vulnerabilities to execute commands.

Document each flag you find: which vulnerability you used to find it, what commands or techniques you ran, what you learned from each step.

4

Complete the "Blue" Room (EternalBlue)

~1–2 hours

Start the "Blue" room on TryHackMe — this targets a Windows machine vulnerable to EternalBlue (MS17-010), the exploit used in the WannaCry ransomware outbreak. Follow the guided steps to: scan with Nmap, identify the vulnerability, exploit with Metasploit, escalate privileges, and crack the password hashes.

Note: This exploit is performed on a controlled TryHackMe virtual machine in an isolated environment. This is not your local network or any real machine.
5

Document All Findings

~30 min
Deliverable: Screenshots of each flag found (3 from Pickle Rick, completion badge from OWASP Top 10 room, root flag from Blue). For each flag, a 2–3 sentence explanation: what vulnerability was exploited, what technique/command was used, and what the defender should have done to prevent it.
Lab 10 (Week 10) — AWS Security Audit
Prerequisite: You need an AWS account for this lab. Use a personal account with no production resources. AWS Free Tier is sufficient for all steps. These commands audit YOUR account only.
1

Run AWS Trusted Advisor Security Checks

~15 min

In the AWS Console, search for Trusted Advisor. Navigate to the Security category. Free tier includes checks for: MFA on root account, S3 bucket permissions, security groups with unrestricted access, IAM use, and exposed access keys. Review every finding and note which are flagged as warnings or errors.

2

Generate and Audit the IAM Credential Report

~20 min

Using AWS CLI (configured with your credentials), run:

aws iam generate-credential-report
aws iam get-credential-report \
  --query 'Content' --output text | base64 -d

This produces a CSV report of all IAM users. For each user, audit: has console access (should have MFA), has access keys (should be rotated in last 90 days), last used date (unused accounts should be deleted or disabled).

3

Audit S3 Buckets for Public Access

~15 min

List all S3 buckets and check their public access block settings:

# List all buckets
aws s3api list-buckets --query 'Buckets[].Name'

# Check public access block for each bucket
aws s3api get-public-access-block --bucket YOUR_BUCKET_NAME

Every bucket should have all four Block Public Access settings set to true unless intentionally public (static website hosting). Document any buckets that are not fully blocked.

4

Review CloudTrail and Recent API Activity

~15 min

Verify CloudTrail is enabled in your account. Review recent API calls:

aws cloudtrail lookup-events \
  --max-results 20 \
  --query 'Events[*].{User:Username,Event:EventName,Time:EventTime}'

Look for any unexpected events — API calls from unfamiliar IPs, calls at unusual hours, or high-risk calls like DeleteBucket, CreateUser, or AttachUserPolicy.

5

Enable GuardDuty and Review Findings

~10 min setup, 24 hours monitoring

Enable Amazon GuardDuty in your account — it's free for the first 30 days:

aws guardduty create-detector --enable --finding-publishing-frequency FIFTEEN_MINUTES

Wait 24 hours, then check for findings: AWS Console → GuardDuty → Findings. For each finding, research the finding type on AWS documentation to understand what it means and how severe it is.

Deliverable: Screenshot of IAM credential report showing users and their MFA/access key status + screenshot of GuardDuty findings (or confirmation of "no findings after 24 hours") + a 10-row findings table from your audit with Severity (High/Medium/Low/Info) and Remediation for each finding.
Lab 11 (Week 11) — Building a Simple SIEM with ELK Stack
1

Create Docker Compose File for ELK

~20 min

Create a file named docker-compose.yml with the following ELK stack configuration:

# docker-compose.yml — ELK SIEM stack
version: '3.8'
services:
  elasticsearch:
    image: elasticsearch:8.11.0
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false
    ports: ["9200:9200"]
    volumes: ["esdata:/usr/share/elasticsearch/data"]

  kibana:
    image: kibana:8.11.0
    ports: ["5601:5601"]
    depends_on: [elasticsearch]

  logstash:
    image: logstash:8.11.0
    volumes:
      - ./logstash.conf:/usr/share/logstash/pipeline/logstash.conf
    depends_on: [elasticsearch]

  nginx:
    image: nginx:latest
    ports: ["8081:80"]
    volumes: ["./nginx_logs:/var/log/nginx"]

volumes:
  esdata:

Run: docker compose up -d

2

Configure Logstash Pipeline for Nginx Logs

~20 min

Create logstash.conf to parse Nginx access logs:

# logstash.conf
input {
  file {
    path => "/var/log/nginx/access.log"
    start_position => "beginning"
  }
}

filter {
  grok {
    match => {
      "message" => '%{COMBINEDAPACHELOG}'
    }
  }
  date {
    match => [ "timestamp", "dd/MMM/yyyy:HH:mm:ss Z" ]
  }
  mutate {
    convert => { "response" => "integer" }
    convert => { "bytes" => "integer" }
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "nginx-logs-%{+YYYY.MM.dd}"
  }
}
3

Generate Test Traffic and Explore in Kibana

~20 min

Generate some test traffic to the Nginx container: for i in {1..50}; do curl http://localhost:8081/; curl http://localhost:8081/nonexistent; done

Open Kibana at http://localhost:5601. Go to Stack Management → Index Patterns. Create a pattern for nginx-logs-*. Go to Discover — you should see log entries being parsed with individual fields (response code, client IP, request path, bytes).

4

Create a Kibana Alert for Scanning Activity

~20 min

In Kibana → Alerting → Create rule → Elasticsearch query. Configure an alert that triggers when the count of log entries where response:404 exceeds 10 within a 1-minute window. This simulates detecting a web scanner or directory enumeration attack.

Set the alert action to write to a log file (connector: log) for simplicity — in production this would be Slack/PagerDuty/email.

Deliverable: Screenshot of Kibana Discover showing parsed nginx log fields + screenshot of the alert rule configuration showing the 404 threshold condition + a 1-paragraph explanation of how this detection would help a SOC analyst identify a potential scanning attack.
Lab 12 (Week 12) — Capstone Red Team Exercise (Authorized)
⚠️ Scope: This lab targets ONLY the DVWA Docker container running on localhost on YOUR machine. No other targets. No external IPs. No scanning beyond localhost:80.
1

Plan the Attack

~20 min

Before touching any tools, write your attack plan. Define: target (localhost DVWA container only), objective (gain administrator access and read simulated flag.txt), tools to use (Nmap, Burp Suite, DVWA at Low security), phases (recon → scan → exploit → report). A professional attacker plans before acting — improvised attacks miss things and destroy evidence.

2

Reconnaissance & Scanning

~20 min

Run service version detection against localhost:

nmap -sV -sC -p 80 localhost

Document: open ports, service versions detected, any banners or version information disclosed. Version disclosure itself is an information leakage finding — note the Apache/PHP version in your report.

3

Vulnerability Assessment with Burp Suite

~30 min

With Burp Suite proxy running, browse every page of DVWA. Burp's passive scanner will log all requests and identify input points. Identify: all pages accepting user input, all GET/POST parameters, cookies and session tokens. This is your target map for exploitation.

4

Exploitation Chain

~45 min

Using techniques from Labs 5–7:

  • SQL injection → dump all users and password hashes
  • Stored XSS → inject a script that exfiltrates a simulated session cookie to an alert box
  • File upload → upload a PHP file that reads the directory listing

Screenshot every step with timestamps. A professional pen test must be reproducible — another tester must be able to follow your steps and get the same result.

5

Write the Full Penetration Test Report

~2–3 hours

Using the template from Lab 8, write a complete professional report. Minimum structure: Executive Summary (1 page), Scope & Methodology (1 page), Findings (minimum 5 vulnerabilities, full detail for each with CVSS v3.1 scores), Remediation Roadmap (prioritized by risk), Appendix (all screenshots numbered and referenced).

Final Deliverable: Complete penetration test report (minimum 10 pages) submitted to mentor for evaluation. This is your capstone document for Cyber Security. It will be evaluated as if submitted to a real client. Treat it accordingly.
