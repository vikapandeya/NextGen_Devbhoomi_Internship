Week 5 — OWASP Top 10 Web Vulnerabilities
W5

OWASP Top 10 Web Vulnerabilities

A01 Access Control · A02 Cryptographic Failures · A03 Injection · A04–A10 Overview

▼
Analogy: OWASP Top 10 is like a list of the most common ways houses get broken into: unlocked back doors (misconfigurations), weak locks (broken auth), no alarm system (lack of logging). Knowing this list means you know what attackers look for first.

A01 — Broken Access Control (#1)

  • Users accessing data or functions they shouldn't
  • IDOR example: change /user/123 to /user/124 in URL — see another user's data
  • Authorization must be checked server-side on every single request
  • Client-side checks (JS hiding buttons) are not authorization
  • Most common web vulnerability in 2021, 2023, and 2025

A02 — Cryptographic Failures

  • Sending passwords in plain text over HTTP
  • Using weak hashing: MD5, SHA-1 for passwords (crackable in seconds)
  • Not using HTTPS — all traffic readable on the network
  • Fix: TLS everywhere, bcrypt for passwords (not SHA-256!), AES-256 for data at rest
  • bcrypt includes salt automatically — prevents rainbow table attacks

A03 — Injection (SQL Injection)

  • User input goes directly into an SQL query without sanitization
  • Classic payload: ' OR '1'='1 bypasses authentication entirely
  • Also: command injection, LDAP injection, XPath injection
  • Prevention: parameterized queries (prepared statements) — 100% effective
  • 30+ years old and still in OWASP Top 3 every year

A04–A10 Overview

  • A04 Insecure Design — architectural flaws, threat model at design phase
  • A05 Security Misconfiguration — default creds, open S3 buckets, verbose errors
  • A06 Vulnerable Components — outdated npm/pip packages with known CVEs
  • A07 Auth Failures — weak passwords, no MFA, session fixation
  • A08–A10 — Software Integrity Failures, Logging Failures, SSRF
-- VULNERABLE: user input directly in query
SELECT * FROM users WHERE username='$input';

-- Attack: input = ' OR '1'='1
-- Result: returns ALL users — authentication bypassed

-- FIXED: parameterized query (Python)
cursor.execute(
    "SELECT * FROM users WHERE username = %s",
    (username,)  # user input is NEVER part of the SQL string
)

Week 5 Key Takeaway

  • SQL injection is 30+ years old and still in OWASP Top 3 because developers keep making the same mistake.
  • Parameterized queries prevent SQL injection 100% of the time — use them everywhere, no exceptions.
Week 6 — Penetration Testing Methodology
W6

Penetration Testing Methodology

5 Phases · Recon · Tools Intro · Legal Requirements

▼
Analogy: A pen tester is a hired burglar. You pay them to break into your house before the real criminals do. They use the same techniques attackers use — but report findings to you instead of exploiting them.

5 Phases of Penetration Testing

  • 1. Reconnaissance — gather information about the target
  • 2. Scanning — identify open ports, services, vulnerabilities
  • 3. Gaining Access — exploit discovered vulnerabilities
  • 4. Maintaining Access — simulate persistence (backdoors)
  • 5. Reporting — document all findings, evidence, remediations

Reconnaissance Types

  • Passive — no direct interaction with target
  • Google dorking, LinkedIn, WHOIS, Shodan, Certificate Transparency logs
  • Active — direct interaction with target systems
  • Port scanning (Nmap), ping sweeps, banner grabbing
  • Active recon requires explicit written permission always

Common Tools Introduction

  • Nmap — network port scanner, OS detection
  • Burp Suite — web application proxy and scanner
  • Metasploit — exploitation framework (authorized targets only)
  • Wireshark — network traffic capture and analysis
  • Nikto — web server vulnerability scanner

Legal & Ethical Requirements

  • Written permission (scope of work) is mandatory — no exceptions
  • Define scope clearly: which IPs, which domains, which tests
  • Rules of Engagement: what's allowed, timing, emergency contacts
  • Unauthorized scanning = criminal offense under IT Act 2000 (India)
  • In this program: test ONLY on DVWA, TryHackMe, HackTheBox

Week 6 Key Takeaway

  • NEVER run scanning or exploitation tools without explicit written permission.
  • Unauthorized scanning is a criminal offense under the IT Act 2000 in India.
  • All testing in this program is done on intentionally vulnerable apps: DVWA, HackTheBox, TryHackMe.
Week 7 — Web Application Security Testing
W7

Web Application Security Testing

Burp Suite · XSS Testing · IDOR Testing · Authentication Testing

▼

Burp Suite Basics

  • Proxy — intercept HTTP/HTTPS between browser and server
  • Repeater — modify and resend individual requests manually
  • Intruder — automated fuzzing and brute force
  • Spider/Crawler — map all application endpoints
  • Configure browser to use 127.0.0.1:8080 as proxy to start

Cross-Site Scripting (XSS)

  • Inject malicious JavaScript into a web page viewed by others
  • Test payload: <script>alert(1)</script> in all input fields
  • Reflected XSS — payload in URL, affects only that request
  • Stored XSS — payload saved in DB, runs for every visitor (dangerous)
  • Prevention: output encoding (htmlspecialchars), CSP headers

Testing for IDOR

  • Change numeric IDs in URLs: /user/123 → /user/124
  • If you see another user's data, IDOR vulnerability confirmed
  • Also test: API parameters, POST body JSON, hidden form fields
  • Prevention: server-side authorization check on every data request
  • Never trust the client to enforce authorization — always server-side

Authentication Testing

  • Brute force — try many passwords; test if account lockout exists
  • Credential stuffing — use leaked password lists from HIBP
  • Session fixation — attacker sets session token before login
  • JWT manipulation — try alg:none, weak secret, tampering claims
  • Fix: account lockout, MFA, rotate session tokens on login

Week 7 Key Takeaway

  • Test YOUR application or explicitly authorized targets only.
  • XSS affects millions of sites. Output encoding (htmlspecialchars in PHP, textContent not innerHTML in JS) prevents it.
  • Authorization must always be checked server-side — never trust the client.
Week 8 — Incident Response
W8

Incident Response

IR Lifecycle · Detection Sources · Containment · Post-Incident

▼
Analogy: Incident response is like a hospital emergency room. You need a clear triage process, trained team, documented procedures, and the ability to act fast. A breach discovered 6 months later costs 10× more than one caught immediately.

IR Lifecycle (NIST)

  • 1. Preparation — runbooks, team training, tools ready before an incident
  • 2. Detection & Analysis — identify and confirm the incident
  • 3. Containment — stop the bleeding, isolate affected systems
  • 4. Eradication — remove malware, patch the vulnerability
  • 5. Recovery — restore systems, verify clean
  • 6. Lessons Learned — post-incident review, update runbooks

Detection Sources

  • SIEM — Security Information and Event Management (centralized log alerting)
  • IDS/IPS — Intrusion Detection/Prevention Systems
  • EDR — Endpoint Detection and Response (CrowdStrike, SentinelOne)
  • User reports — often the first to notice something wrong
  • Threat intel feeds — external IOC matching

Containment Strategy

  • Isolate affected systems from the network immediately
  • Preserve evidence first — memory dump, log copies before wiping
  • Do NOT delete logs — they are your investigation trail
  • Block malicious IPs/domains at firewall and DNS level
  • Short-term containment (isolate now), then long-term (rebuild clean)

Post-Incident Requirements

  • Root cause analysis — how did the attacker get in?
  • Patch the exploited vulnerability
  • Update runbooks and detection rules
  • Regulatory notification: GDPR (72 hours), CERT-In (6 hours for Indian cos.)
  • Blameless postmortem — learn, don't punish

Week 8 Key Takeaway

  • The first 15 minutes of incident response determine the outcome.
  • Have a written runbook before an incident — you can't think clearly under pressure.
  • Preserve evidence before containment. Destroyed logs cannot be recovered.
