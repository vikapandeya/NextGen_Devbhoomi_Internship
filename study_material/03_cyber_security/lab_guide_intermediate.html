Intermediate Lab Guide — Cyber Security

4 labs on intentionally vulnerable applications: DVWA setup, web vulnerability testing, Burp Suite practice, and basic pen test report.

4 Labs Weeks 5–8 DVWA · Burp Suite · Docker Intermediate Level
⚠️ Legal Reminder: All labs in this section use intentionally vulnerable practice applications (DVWA). Never use these techniques on any system you do not own or have explicit written permission to test. Unauthorized testing is illegal under IT Act 2000 (India) and Computer Fraud and Abuse Act (USA).
Lab 5 (Week 5) — DVWA Setup & SQL Injection

Lab Objectives

  • Set up a vulnerable practice environment with Docker
  • Perform SQL injection on a low-security target
  • Understand why parameterized queries prevent injection
  • Compare vulnerable vs secure source code side by side
1

Install DVWA via Docker

~10 min

Ensure Docker Desktop is installed. Run the following command to start DVWA (Damn Vulnerable Web Application) — an intentionally vulnerable practice target:

docker run --rm -it -p 80:80 vulnerables/web-dvwa

Docker pulls the image and starts the container. DVWA is now available at http://localhost on your machine only.

2

Configure DVWA at Low Security

~5 min

Navigate to http://localhost. Login with admin / password. Go to Setup / Reset DB — click "Create / Reset Database". Then go to DVWA Security and set the security level to Low.

3

Perform SQL Injection at Low Security

~20 min

Navigate to SQL Injection in the left menu. Enter 1 in the User ID field — observe normal output (shows user details for ID 1).

Now enter the injection payload:

1' OR '1'='1

Observe: all user records are returned. The SQL query is now: SELECT * FROM users WHERE id='1' OR '1'='1' — the condition is always true.

Next, try a UNION-based injection to extract password hashes:

1' UNION SELECT user, password FROM users-- -

You should see all usernames and their MD5-hashed passwords in the output.

4

Compare Low vs High Security Source Code

~15 min

Switch the security level to High. Retry the same injection payloads. They are now blocked. Click View Source in DVWA — it shows you the PHP source code for both security levels side by side.

Key difference: Low uses direct string concatenation into SQL. High uses PDO with prepared statements — the user input is never part of the SQL command string.

Deliverable: Screenshot of successful injection on "Low" (showing all users/hashes) + screenshot of blocked injection on "High" + 2-sentence written explanation of what changed in the source code and why parameterized queries prevent the attack.
Lab 6 (Week 6) — Burp Suite: Intercepting & Modifying Requests
1

Set Up Burp Suite Community Edition

~15 min

Download Burp Suite Community Edition from portswigger.net/burp — it's free. Install and launch it. Click "Temporary project" → "Use Burp defaults" → "Start Burp".

Configure Firefox to use Burp as a proxy: Open Firefox → Settings → Network Settings → Manual proxy → HTTP Proxy: 127.0.0.1, Port: 8080. Install the Burp CA certificate in Firefox (visit http://burpsuite → download CA cert → import in Firefox certificate settings).

2

Intercept a Login Request

~15 min

In Burp Suite → Proxy → Intercept tab — ensure "Intercept is on". In Firefox, navigate to http://localhost and attempt to log in to DVWA. The login POST request will appear in Burp's Intercept tab before it reaches the server.

Examine the request: you can see the POST body including the username and password in plain text. This is exactly what a MITM attacker would see on HTTP traffic.

3

Use Burp Repeater for SQL Injection

~20 min

Navigate to DVWA → SQL Injection. Submit a normal user ID. In Burp Proxy, find that request. Right-click → Send to Repeater.

In the Repeater tab, modify the id parameter value to different SQL payloads and click "Send" for each. Observe the response in the right panel — no need to re-browse. This lets you test many payloads quickly and compare responses precisely.

Tip: Repeater is the most useful tool for manual web application testing. You can modify any part of the HTTP request — headers, cookies, body — and see exactly what the server responds.
4

Use Burp Intruder for Login Brute Force (Practice Only)

~20 min

Intercept the DVWA login POST request. Right-click → Send to Intruder. In Intruder, select the password parameter value and click "Add §" to mark it as a payload position. In the Payloads tab, use a simple list of 10 common passwords (admin, password, 123456, letmein, etc.). Click "Start attack".

Sort results by response length — a different length typically indicates a successful login vs failed. Identify which password produces a different response.

Deliverable: Screenshot of Burp Repeater showing a successful SQL injection request and response + screenshot of Intruder attack results sorted by response length, with the successful entry highlighted.
Lab 7 (Week 7) — XSS and File Upload Vulnerabilities
1

Reflected XSS at Low Security

~15 min

DVWA → XSS (Reflected) at Low security. In the "What's your name?" field, enter:

<script>alert('XSS by YOUR_NAME')</script>

If the alert popup appears with your name, reflected XSS is confirmed. The script executed because the server returned your input directly into the HTML page without encoding it. Take a screenshot of the alert.

2

Stored XSS — More Dangerous

~15 min

DVWA → XSS (Stored) at Low security. In the message/comment field, post:

<script>document.body.style.background='red'</script>

Reload the page. The background turns red for every user who visits this page — because the script is stored in the database and served to everyone. This demonstrates why stored XSS is far more dangerous than reflected: it persists and affects all users automatically.

3

File Upload Vulnerability at Low Security

~20 min

DVWA → File Upload at Low security. Create a text file named test.php containing:

<?php echo "Uploaded: " . $_SERVER['PHP_SELF']; ?>

Upload it. DVWA shows the path where it was saved. Navigate to that path in your browser — the PHP code executes on the server. This is a remote code execution vector — an attacker could upload a PHP web shell and control the server.

4

Test File Upload at Medium Security & Bypass Attempt

~20 min

Switch to Medium security for File Upload. Rename your test.php to evil.php.jpg and try to upload. Note the result — Medium security checks the file extension more carefully. View the source code (DVWA "View Source" button) to see exactly what validation is happening and why the extension trick may or may not work at this level.

5

Research CSP Headers as Prevention

~15 min

Look up Content Security Policy (CSP) on MDN Web Docs (developer.mozilla.org/en-US/docs/Web/HTTP/CSP). Identify which CSP directives would prevent the XSS attacks you performed in steps 1 and 2. Write them down as HTTP response headers.

Deliverable: Screenshot of reflected XSS alert popup + screenshot of stored XSS red background effect + screenshot of file upload test on Medium security + written explanation (100 words) of which CSP directives would prevent the XSS attacks and how.
Lab 8 (Week 8) — Writing a Basic Penetration Test Report
1

Write the Executive Summary

~20 min

Write 1 paragraph describing: the overall security posture of DVWA at Low security level, the most critical finding discovered, and the business risk it represents. Write this for a non-technical manager — avoid jargon, focus on risk and impact.

2

Define Scope and Methodology

~15 min

Document what was tested: application name (DVWA), URL tested (http://localhost), testing period (dates), tools used (Burp Suite Community Edition, browser DevTools), testing approach (grey box — source code available for comparison), and any scope limitations (did not test database server directly, no automated scanner used).

3

Create the Findings Table

~30 min

Create a findings table with columns: ID | Vulnerability Name | Severity | CVSS 3.1 Score | Description | Evidence (screenshot reference) | Remediation. Include at minimum:

  • VULN-001: SQL Injection — Critical — CVSS 9.8
  • VULN-002: Stored Cross-Site Scripting — High — CVSS 8.0
  • VULN-003: Unrestricted File Upload — High — CVSS 8.8

Calculate CVSS scores at nvd.nist.gov/vuln-metrics/cvss/v3-calculator — select the appropriate impact metrics for each finding.

4

Write Per-Finding Detailed Sections

~30 min

For each finding, write a full section containing: description of the vulnerability, steps to reproduce (numbered, precise), screenshot evidence, and specific remediation recommendation (not generic — e.g. "Use PDO prepared statements in the getUserData() function on line 42" not just "use prepared statements").

5

Conclusion, Recommendations, and Appendix

~20 min

Write a conclusion with a prioritized fix list (Critical first, then High), suggested re-test timeline (re-test Critical findings within 2 weeks of remediation), and overall security recommendation. Add an appendix with all screenshots referenced in the findings.

Deliverable: Professional pen test report in PDF or Word format (minimum 5 pages), submitted to mentor portal before end of Week 8. This document will be reviewed as if submitted to a real client.
