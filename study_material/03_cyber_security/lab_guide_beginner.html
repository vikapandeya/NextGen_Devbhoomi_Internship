Beginner Lab Guide — Cyber Security

4 labs: safe security practices, analyzing phishing emails, network traffic basics with Wireshark, and password security audit.

4 Labs Weeks 1–4 No hacking tools required Beginner Level
Lab 1 (Week 1) — Security Mindset & Threat Modeling

Lab Objectives

  • Think like an attacker to find weaknesses
  • Identify and classify digital assets by sensitivity
  • Apply the STRIDE threat modeling framework
  • Calculate and communicate risk scores to stakeholders
1

Define Your Web Application & List Assets

~20 min

Create a threat model for a simple web application: a user login system with a personal notes storage feature. Begin by listing every asset in this system.

Assets to identify: user passwords (stored in database), notes content (user's private data), session tokens (issued after login), database connection credentials, the server itself.

2

Apply STRIDE to Each Asset

~30 min

For each asset, identify threats using the STRIDE model. Create a table with columns: Asset | Threat Type | Threat Description.

  • Spoofing — pretending to be another user or system
  • Tampering — modifying data without authorization
  • Repudiation — denying that an action was taken
  • Information Disclosure — data exposed to unauthorized parties
  • Denial of Service — making the system unavailable
  • Elevation of Privilege — gaining more access than permitted
3

Score Each Threat (Risk = Likelihood × Impact)

~20 min

For each threat in your table, rate on a 1–5 scale: Likelihood (how probable?) and Impact (how damaging?). Calculate Risk Score = Likelihood × Impact (max = 25).

Example: "Attacker brute-forces weak password" — Likelihood: 4, Impact: 5, Risk: 20 (Critical).

Thresholds: ≥ 20 = Critical, 12–19 = High, 6–11 = Medium, < 6 = Low.

4

Propose Mitigations for High-Risk Threats

~15 min

For every threat with a risk score ≥ 12, propose one concrete mitigation. Example: brute force risk → implement account lockout after 5 failed attempts + require MFA.

Add a Residual Risk column: after your mitigation is applied, what is the new risk score? Mitigation should reduce likelihood or impact.

5

Write the Threat Model Document

~25 min

Create a 1-page document containing: (1) Application description, (2) Asset list, (3) Completed STRIDE threat table with risk scores, (4) Mitigations for all High/Critical threats, (5) Your overall risk assessment conclusion.

Deliverable: Threat model document (Word/PDF/Markdown) submitted to mentor portal before end of Week 1.
Lab 2 (Week 2) — Analyzing Phishing Emails
⚠️ Safety Notice: All analysis is performed on SAMPLE emails provided by the course portal. Never interact with, forward, or click links in real phishing emails. Never enter credentials into any site discovered through a sample email.
1

Download Sample Phishing Emails

~5 min

Download the sample phishing email set from the course portal (3 sample .eml files provided). These are sanitized examples — all links have been defanged (e.g. hxxps://) and cannot be clicked.

2

Analyze Email Content for Red Flags

~30 min

For each of the 3 sample emails, identify and document:

  • Sender display name vs the actual email address (often different)
  • Urgency language in subject line or body
  • Suspicious links — note display text vs actual URL
  • Grammar and spelling errors
  • Mismatched domain names (e.g. paypa1.com instead of paypal.com)
3

Analyze Email Headers via MXToolbox

~20 min

Open each sample email's raw headers (provided as text in the course portal). Paste into mxtoolbox.com/EmailHeaders. Look for:

  • SPF — does the sender IP match authorized senders for that domain?
  • DKIM — is the email cryptographically signed by the sender domain?
  • DMARC — does the domain policy match what was received?
  • SPF Fail + DKIM Fail = almost certainly spoofed or phishing

Screenshot the MXToolbox results for each email.

4

Check Domains on VirusTotal

~10 min

The sample emails contain defanged domains in the body. Re-fang only for VirusTotal analysis (add back https://, do NOT visit the URL). At virustotal.com, paste the domain name (not the full URL) and check the Security Vendors' Analysis results.

Note: how many vendors flag the domain as malicious? Record the category (phishing, malware distribution, etc.).

5

Write the Analysis Report

~20 min

Write a 1-page analysis covering all 3 emails: what social engineering techniques were used in each, how you identified each as phishing, and what a user could have done to avoid being tricked.

Deliverable: Analysis report (1 page minimum) + screenshots of email header analysis from MXToolbox. Submit via mentor portal by end of Week 2.
Lab 3 (Week 3) — Wireshark Network Traffic Analysis
1

Install Wireshark

~10 min

Download and install Wireshark from wireshark.org. Install WinPcap/Npcap (included in installer) to allow packet capture. Run as Administrator on Windows.

2

Capture HTTP Traffic

~15 min

Select your active network interface (Wi-Fi or Ethernet). Start a capture. Open Firefox and browse to http://httpbin.org/get — this is a safe plain HTTP test endpoint. Browse around for 1–2 minutes then stop the capture.

Tip: httpbin.org intentionally serves over HTTP for testing purposes. Never send sensitive data over HTTP in real use.
3

Filter and Examine HTTP Packets

~20 min

In Wireshark's filter bar, type http and press Enter. Find an HTTP GET request. Click on it and examine: source/destination IP address, HTTP method and path, request headers (User-Agent, Host, Accept), and response headers and body.

You can read everything — the URL, your IP, all headers. This is what an attacker on your network can see when you use HTTP.

4

Capture and Compare HTTPS Traffic

~15 min

Start a new capture. Browse to https://example.com. Stop capture. Filter by tcp.port == 443. You can see that traffic exists (IP addresses, timing, packet sizes) but the payload is completely encrypted. The content is unreadable.

Take a screenshot of each — the readable HTTP packet content, and the encrypted HTTPS TLS Application Data.

5

Write Comparison and Reflection

~15 min
Deliverable: Two labeled screenshots — one showing readable HTTP data (with annotations pointing to sensitive fields), one showing encrypted HTTPS data. Written comparison (200 words minimum): what could an attacker on public WiFi see with each protocol, and what does this mean for your daily browsing habits?
Lab 4 (Week 4) — Password Security Audit
1

Check Your Emails on Have I Been Pwned

~10 min

Go to haveibeenpwned.com. Enter your personal email addresses. This service is safe, privacy-respecting (Troy Hunt), and free. It checks your email against 13+ billion compromised accounts from known data breaches.

Note: how many breaches appear? Which ones include passwords? This is how attackers find valid credential combos for credential stuffing attacks.

2

Install Bitwarden Password Manager

~15 min

Install Bitwarden from bitwarden.com — it is free, open source, and independently audited. Install the browser extension for Chrome/Firefox and the desktop app. Create an account with a strong master passphrase (4+ random words, e.g. "mountain-river-keyboard-cloud").

Tip: Your master password is the one password you do need to remember. Make it a passphrase, not a password. Write it on paper and store it physically safely — not in a file on your computer.
3

Audit 10 Existing Passwords

~20 min

Add 10 of your current account credentials to Bitwarden. Go to Tools → Vault Health Reports. Identify:

  • Reused passwords (same password on multiple sites)
  • Weak passwords (under 12 characters)
  • Exposed passwords (found in known breach databases)

Take a screenshot of the health report showing the current state (number of issues found).

4

Generate Strong Passwords for All Weak Accounts

~20 min

For every weak/reused/exposed password found, use Bitwarden's password generator (Settings → Generator) to create new passwords: 20+ characters, include uppercase, lowercase, numbers, and symbols. Update each account and save the new credential in Bitwarden.

5

Enable TOTP MFA on 3 Critical Accounts

~20 min

Install Aegis Authenticator (Android, free, open source) or Google Authenticator on your phone. Enable MFA on: your primary email (Gmail/Outlook), your GitHub account, and one social media account. Choose the "Authenticator App" option, scan the QR code.

Store the backup codes in Bitwarden as a secure note — they are your emergency access if you lose your phone.

Deliverable: Two screenshots of Bitwarden Health Report — before and after remediation — showing improvement in password hygiene. Plus confirmation screenshot of MFA enabled on 3 accounts.
