Week 5 Firewalls, VPN & Security ▾

iptables — Linux Netfilter Firewall

iptables is the classic Linux firewall tool. It works with tables (filter, nat, mangle) and chains (INPUT, OUTPUT, FORWARD, PREROUTING, POSTROUTING).

ChainPurpose
INPUTTraffic destined for this host
OUTPUTTraffic originating from this host
FORWARDTraffic passing through this host (routing/NAT)
 
# View current rules
sudo iptables -L -n -v           # List all rules with packet counters
sudo iptables -t nat -L -n -v    # View NAT table rules

# Allow SSH (port 22)
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Allow established/related connections (critical for stateful behaviour)
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
sudo iptables -A OUTPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow HTTP and HTTPS
sudo iptables -A INPUT -p tcp -m multiport --dports 80,443 -j ACCEPT

# Allow loopback interface (always needed)
sudo iptables -A INPUT -i lo -j ACCEPT
sudo iptables -A OUTPUT -o lo -j ACCEPT

# Drop everything else on INPUT (set default policy to DROP)
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT    # Usually allow all outbound

# Delete a rule (use -D instead of -A)
sudo iptables -D INPUT -p tcp --dport 22 -j ACCEPT

# Save rules (Debian/Ubuntu)
sudo iptables-save > /etc/iptables/rules.v4
sudo ip6tables-save > /etc/iptables/rules.v6

nftables — Modern iptables Replacement

# nftables uses a cleaner, consistent syntax
sudo nft list ruleset                    # View all rules

# Create a basic ruleset
sudo nft add table inet filter
sudo nft add chain inet filter input { type filter hook input priority 0 \; policy drop \; }
sudo nft add rule inet filter input iif lo accept
sudo nft add rule inet filter input ct state established,related accept
sudo nft add rule inet filter input tcp dport 22 accept
sudo nft add rule inet filter input tcp dport { 80, 443 } accept

UFW — Uncomplicated Firewall

TLS/SSL Handshake

1. ClientHello      → Client sends: supported TLS versions, cipher suites, random bytes
2. ServerHello      ← Server selects: TLS version, cipher suite, sends its Certificate
3. Certificate verify → Client validates server certificate against trusted CAs
4. Key Exchange     ← Server sends key exchange data (DH/ECDH parameters)
5. Client Finished  → Client derives session keys, sends Finished message (encrypted)
6. Server Finished  ← Server confirms, both sides now use symmetric encryption
7. Encrypted data ↔  Application data exchanged over encrypted channel

# View certificate info:
openssl s_client -connect google.com:443 -showcerts
# Generate self-signed cert:
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes

VLANs — Virtual Local Area Networks

VLANs allow a single physical switch to be divided into multiple logical networks. Hosts in different VLANs cannot communicate without a Layer 3 device (router or L3 switch).

# VLAN tagging uses IEEE 802.1Q standard
# A 4-byte tag is inserted into the Ethernet frame header:
# TPID (0x8100) | PCP (3 bits) | DEI (1 bit) | VLAN ID (12 bits, 1-4094)

# Port types:
# Access port: carries ONE VLAN (untagged) — connects to end devices
# Trunk port:  carries MULTIPLE VLANs (tagged) — connects switches/routers

Linux VLAN Configuration

# Create VLAN 10 on interface eth0
sudo ip link add link eth0 name eth0.10 type vlan id 10
sudo ip addr add 192.168.10.1/24 dev eth0.10
sudo ip link set eth0.10 up

# Create VLAN 20
sudo ip link add link eth0 name eth0.20 type vlan id 20
sudo ip addr add 192.168.20.1/24 dev eth0.20
sudo ip link set eth0.20 up

# Verify
ip -d link show eth0.10          # Shows VLAN ID and parent interface
ip addr show eth0.10

# Persistent (Netplan)
# /etc/netplan/00-vlans.yaml
network:
  version: 2
  vlans:
    eth0.10:
      id: 10
      link: eth0
      addresses: [192.168.10.1/24]
    eth0.20:
      id: 20
      link: eth0
      addresses: [192.168.20.1/24]

STP — Spanning Tree Protocol

STP prevents Layer 2 loops in redundant switch topologies. Without STP, a broadcast storm would instantly saturate all links.

# STP process:
1. Elect Root Bridge (lowest Bridge ID = priority + MAC)
2. Each non-root switch finds its Root Port (best path to root)
3. Each segment elects a Designated Port (one per segment)
4. All other ports become Blocking ports

# Port states: Blocking → Listening → Learning → Forwarding
# 802.1D (original STP): ~30-50 seconds to converge
# 802.1w (RSTP): ~1-2 seconds to converge (preferred)
# 802.1s (MSTP): Multiple instances, one per group of VLANs

Link Aggregation — LACP & Bonding

# Linux bonding — combines multiple NICs for redundancy or speed
# /etc/netplan/00-bonding.yaml
network:
  version: 2
  bonds:
    bond0:
      interfaces: [eth0, eth1]
      parameters:
        mode: 802.3ad           # LACP (requires switch support)
        mii-monitor-interval: 100
      addresses: [192.168.1.10/24]
      routes:
        - to: 0.0.0.0/0
          via: 192.168.1.1

# Bonding modes:
# active-backup (mode 1): only one active, other is standby — failover only
# balance-rr   (mode 0): round-robin across all interfaces — throughput
# 802.3ad      (mode 4): LACP — requires switch config, provides LB + redundancy
# balance-tlb  (mode 5): adaptive TX load balancing, no switch support needed

Network Namespaces & Virtual Networking

# Network namespaces — isolated network stack (used by containers/VMs)
sudo ip netns add ns1                        # Create namespace
sudo ip netns add ns2
sudo ip netns list                           # List namespaces
sudo ip netns exec ns1 ip addr              # Run command inside ns1

# Connect two namespaces with a virtual ethernet pair (veth)
sudo ip link add veth0 type veth peer name veth1   # Create pair
sudo ip link set veth0 netns ns1            # Move veth0 to ns1
sudo ip link set veth1 netns ns2            # Move veth1 to ns2
sudo ip netns exec ns1 ip addr add 192.168.100.1/24 dev veth0
sudo ip netns exec ns2 ip addr add 192.168.100.2/24 dev veth1
sudo ip netns exec ns1 ip link set veth0 up
sudo ip netns exec ns2 ip link set veth1 up
sudo ip netns exec ns1 ping 192.168.100.2   # Test connectivity

# Virtual bridge — connects multiple interfaces at Layer 2
sudo ip link add br0 type bridge
sudo ip link set eth1 master br0
sudo ip link set eth2 master br0
sudo ip addr add 192.168.1.1/24 dev br0
sudo ip link set br0 up

# tun vs tap:
# tun: Layer 3 (IP packets) — used by VPNs like WireGuard/OpenVPN (tun mode)
# tap: Layer 2 (Ethernet frames) — used by VM bridging, OpenVPN (tap mode)

ECMP — Equal-Cost Multi-Path

# When multiple routes to same destination have equal metric,
# traffic is distributed across all next-hops (load balancing)
sudo ip route add 10.0.0.0/8 nexthop via 192.168.1.1 weight 1 \
                               nexthop via 192.168.2.1 weight 1

# Verify
ip route show 10.0.0.0/8
# Output: 10.0.0.0/8
#   nexthop via 192.168.1.1 dev eth0 weight 1
#   nexthop via 192.168.2.1 dev eth1 weight 1

rsync — Efficient File Transfer

# rsync: only transfers changed files/blocks (delta sync)
rsync -avz /local/dir/ user@remote:/remote/dir/   # Sync to remote (trailing / matters!)
rsync -avz user@remote:/remote/dir/ /local/dir/   # Sync from remote

# Key flags:
# -a: archive mode (preserve permissions, timestamps, symlinks, owner, group)
# -v: verbose
# -z: compress during transfer
# --progress: show file transfer progress
# --delete: delete files on dest that don't exist on source (mirror mode)
# --exclude: exclude patterns
# --dry-run / -n: show what would happen without doing it

rsync -avz --progress --exclude='*.log' --exclude='node_modules/' \
  /data/app/ user@backup-server:/backups/app/

# Common backup pattern with date stamping:
DEST="/backups/app-$(date +%Y%m%d)"
rsync -avz --link-dest=/backups/app-latest /data/app/ $DEST/
ln -sfn $DEST /backups/app-latest    # Update symlink to latest

LVM — Logical Volume Management

# LVM hierarchy: Physical Volume (PV) → Volume Group (VG) → Logical Volume (LV)
# Benefits: resize LVs without downtime, snapshots, span multiple disks

# Setup (assuming new disk /dev/sdb)
sudo pvcreate /dev/sdb                           # Initialise PV
sudo vgcreate datavg /dev/sdb                    # Create Volume Group
sudo lvcreate -L 10G -n datalv datavg            # Create 10GB LV
sudo mkfs.ext4 /dev/datavg/datalv                # Format
sudo mkdir /data && sudo mount /dev/datavg/datalv /data
echo '/dev/datavg/datalv /data ext4 defaults 0 2' | sudo tee -a /etc/fstab

# Info commands
sudo pvdisplay; sudo vgdisplay; sudo lvdisplay
sudo vgs; sudo lvs                               # Compact summary

# Extend an LV (no downtime needed for ext4/xfs)
sudo lvextend -L +5G /dev/datavg/datalv          # Add 5GB
sudo resize2fs /dev/datavg/datalv                # Grow ext4 filesystem
# For XFS: sudo xfs_growfs /data

# Snapshot (for backups)
sudo lvcreate -L 2G -s -n datalv_snap /dev/datavg/datalv
sudo mount -o ro /dev/datavg/datalv_snap /mnt/snap
sudo lvremove /dev/datavg/datalv_snap            # Remove when done

Prometheus Architecture

Prometheus is a pull-based time-series monitoring system. It scrapes metrics from HTTP endpoints at configurable intervals and stores them in its own TSDB (Time Series Database).

# /etc/prometheus/prometheus.yml
global:
  scrape_interval: 15s          # How often to scrape targets
  evaluation_interval: 15s      # How often to evaluate alerting rules
  external_labels:
    datacenter: 'dc1'

rule_files:
  - "alerts.yml"
  - "recording_rules.yml"

alerting:
  alertmanagers:
    - static_configs:
        - targets: ['localhost:9093']

scrape_configs:
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  - job_name: 'node'
    static_configs:
      - targets:
          - 'server1.lab.local:9100'
          - 'server2.lab.local:9100'
    scrape_interval: 10s     # Override global for this job

  - job_name: 'nginx'
    static_configs:
      - targets: ['server1.lab.local:9113']  # nginx-prometheus-exporter

PromQL — Query Language

# Instant vector — current value
node_cpu_seconds_total                              # All CPU metric time series
node_cpu_seconds_total{mode="idle"}                 # Filter by label
node_cpu_seconds_total{instance="server1:9100"}     # Filter by instance

# Range vector — values over time window
node_cpu_seconds_total[5m]                          # Last 5 minutes of data

# Rate — per-second rate of increase over time window (use for counters)
rate(node_cpu_seconds_total{mode="idle"}[5m])

# CPU usage % per instance
100 - (avg by(instance)(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# Sum all HTTP requests per job
sum by(job)(rate(http_requests_total[5m]))

# Memory usage
node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes

# Disk free percentage
node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"} * 100

# Functions:
# rate()       — per-second rate (counters)
# irate()      — instantaneous rate (last two samples)

SLI / SLO / SLA / Error Budget

TermDefinitionExample
SLI — Service Level IndicatorA measured metric that reflects service qualityp99 request latency = 145ms; error rate = 0.01%
SLO — Service Level ObjectiveInternal target for an SLIp99 latency < 200ms, 99.9% of the time over 30 days
SLA — Service Level AgreementCustomer-facing contract with penalties for breach"We guarantee 99.9% uptime; credits if we fail"
Error BudgetAllowed downtime/failures = 1 − SLO availability99.9% SLO → 0.1% = 43.8 min/month error budget

# Availability SLI:
sum(rate(http_requests_total{status!~"5.."}[30d])) /
sum(rate(http_requests_total[30d]))

# Error budget remaining (%):
