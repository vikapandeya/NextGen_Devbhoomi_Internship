Week 1 OSI & TCP/IP Models ▾

The OSI Model (7 Layers)

The Open Systems Interconnection (OSI) model is a conceptual framework that standardises how different network systems communicate. It divides communication into 7 distinct layers, each with a specific responsibility.

# Layer PDU Key Protocols / Technologies Responsibility
7ApplicationDataHTTP, HTTPS, FTP, DNS, SMTP, IMAP, SSH, TelnetEnd-user services, network APIs
6PresentationDataSSL/TLS, JPEG, MPEG, ASCII, UTF-8Data translation, encryption, compression
5SessionDataNetBIOS, RPC, PPTP, SIPEstablishes, manages, terminates sessions
4TransportSegmentTCP (reliable), UDP (fast/unreliable)End-to-end delivery, flow control, error correction
3NetworkPacketIP (IPv4/IPv6), ICMP, OSPF, RIP, BGPLogical addressing, routing between networks
2Data LinkFrameEthernet (802.3), Wi-Fi (802.11), ARP, VLAN (802.1Q)Physical addressing (MAC), error detection on local link
1PhysicalBitsEthernet cable, fibre, radio waves, USB, BluetoothTransmits raw bits over physical medium
Memory trick: "All People Seem To Need Data Processing" (Application → Physical, top-down) or "Please Do Not Throw Sausage Pizza Away" (bottom-up).

TCP/IP Model (4 Layers)

The TCP/IP model is the practical model used in real networks (the internet runs on TCP/IP). It maps to the OSI model but collapses some layers.

TCP/IP LayerEquivalent OSI LayersExample Protocols
Application7, 6, 5 (Application + Presentation + Session)HTTP, FTP, DNS, SMTP, SSH
Transport4 (Transport)TCP, UDP
Internet3 (Network)IP, ICMP, ARP (disputed)
Network Access2, 1 (Data Link + Physical)Ethernet, Wi-Fi, MAC addressing

Encapsulation & Decapsulation

As data travels down the OSI stack (sender side), each layer wraps it with its own header (and sometimes trailer). This is encapsulation. On the receiver side, each layer unwraps its header — this is decapsulation.

Application data: "GET / HTTP/1.1\r\nHost: google.com"
         ↓  Transport layer adds TCP header (src/dst port, seq#)
       [TCP Header | Data]          ← Segment
         ↓  Network layer adds IP header (src/dst IP)
    [IP Header | TCP Header | Data] ← Packet
         ↓  Data Link adds Ethernet header + FCS trailer
[ETH Hdr | IP Hdr | TCP Hdr | Data | FCS] ← Frame
         ↓  Physical layer sends as bits
   10110011 00101010 ...            ← Bits

IPv4 Address Structure

An IPv4 address is a 32-bit number written in dotted-decimal notation — four octets (0–255) separated by dots.

Example:  192  .  168  .   1  .  100
Binary:  11000000.10101000.00000001.01100100
         ←──── Network ────→←── Host ──→ (for /24)

IPv4 Address Classes

ClassFirst Octet RangeDefault MaskUse
A1 – 126/8 (255.0.0.0)Large organisations
B128 – 191/16 (255.255.0.0)Medium organisations
C192 – 223/24 (255.255.255.0)Small networks
D224 – 239N/AMulticast
E240 – 255N/AExperimental / Reserved

Private IP Ranges (RFC 1918)

These ranges are not routable on the public internet — they are reserved for internal/private use:

ClassPrivate RangeCIDR
A10.0.0.0 – 10.255.255.25510.0.0.0/8
B172.16.0.0 – 172.31.255.255172.16.0.0/12
C192.168.0.0 – 192.168.255.255192.168.0.0/16
Special addresses: Loopback: 127.0.0.0/8 (127.0.0.1 = localhost — always refers to your own machine). Link-local: 169.254.0.0/16 (APIPA — assigned when DHCP fails).

TCP vs UDP

FeatureTCPUDP
ConnectionConnection-oriented (3-way handshake)Connectionless
ReliabilityGuaranteed delivery, acknowledgementsBest-effort, no ACK
OrderOrdered (sequence numbers)No ordering
SpeedSlower (overhead)Faster (low overhead)

Week 2 Subnetting & IP Addressing ▾

CIDR Notation

CIDR (Classless Inter-Domain Routing) notation writes an IP address followed by a slash and the number of network bits: IP/prefix-length.

192.168.1.0/24
              ↑
     24 bits are NETWORK (fixed), 8 bits are HOST (variable)
     Subnet mask: 255.255.255.0
     Total addresses: 2^8 = 256
     Usable host addresses: 256 - 2 = 254  (subtract network + broadcast)

Common Subnet Masks

Subnetting Calculation

Formula: To create n subnets, borrow log₂(n) bits from the host portion.

# Example: Split 192.168.10.0/24 into 4 equal subnets
# Borrow 2 bits → 2^2 = 4 subnets; new prefix = /26 (24+2)
# Each subnet: 2^6 = 64 addresses, 62 usable hosts

Subnet 1: 192.168.10.0/26   → hosts .1 – .62,  broadcast .63
Subnet 2: 192.168.10.64/26  → hosts .65 – .126, broadcast .127
Subnet 3: 192.168.10.128/26 → hosts .129 – .190, broadcast .191
Subnet 4: 192.168.10.192/26 → hosts .193 – .254, broadcast .255

VLSM — Variable Length Subnet Masking

VLSM allows using different prefix lengths within the same address space — allocate just the right size for each segment, avoiding waste.

# Example: Allocate 192.168.1.0/24 for:
#   HQ LAN (50 hosts)  → need /26 (62 hosts) → 192.168.1.0/26
#   Branch LAN (20 hosts) → need /27 (30 hosts) → 192.168.1.64/27
#   Point-to-point link (2 hosts) → need /30 (2 hosts) → 192.168.1.96/30
#   Remaining: 192.168.1.100/30 onwards unused

IPv6 Addressing

IPv6 uses 128-bit addresses written as 8 groups of 4 hexadecimal digits separated by colons.

Full:       2001:0db8:85a3:0000:0000:8a2e:0370:7334
Compressed: 2001:db8:85a3::8a2e:370:7334
            (leading zeros dropped; :: replaces one run of all-zero groups)

Loopback:   ::1            (equivalent to 127.0.0.1 in IPv4)
Link-local: fe80::1/10     (automatically assigned, not routable)
Global:     2001::/16 and 2000::/3 (public, routable)
Multicast:  ff00::/8

NAT — Network Address Translation

NAT allows multiple devices with private IP addresses to share one (or few) public IP addresses when accessing the internet.

TypeDescription
Static NATOne-to-one mapping of private IP to public IP (used for servers)
Dynamic NATMany-to-many from a pool of public IPs
PAT (NAT Overload)Many private IPs → one public IP, differentiated by port number (most common home/office NAT)

DHCP — DORA Process

DHCP automatically assigns IP configuration to clients. The four-step exchange is called DORA:

1. DISCOVER  – Client broadcasts "I need an IP" (src: 0.0.0.0:68, dst: 255.255.255.255:67)
2. OFFER     – Server broadcasts an available IP lease offer
3. REQUEST   – Client broadcasts "I accept the offered IP from that server"
4. ACKNOWLEDGE – Server confirms the lease (includes IP, mask, gateway, DNS, lease time)

DNS — Domain Name System

DNS translates human-readable domain names (google.com) into IP addresses. The hierarchy works top-down:

Root servers (.) → TLD servers (.com, .org, .in) → Authoritative NS → Resolver → Client

Query for www.google.com:
  1. OS checks local cache / /etc/hosts
  2. Asks recursive resolver (ISP or 8.8.8.8)
  3. Resolver asks root server → points to .com TLD
  4. .com TLD → points to google.com authoritative NS
  5. Authoritative NS → returns A record (IP: 142.250.x.x)
  6. Resolver caches and returns to client

File Permissions

# Permission string: -rwxr-xr--
# Format: [type][owner rwx][group rwx][other rwx]
# - = file, d = directory, l = symlink

# r = 4, w = 2, x = 1
chmod 755 script.sh       # owner=rwx(7), group=r-x(5), other=r-x(5)
chmod 644 config.txt      # owner=rw-(6), group=r--(4), other=r--(4)
chmod u+x deploy.sh       # Add execute for owner only
chmod go-w /etc/cron.d/   # Remove write from group and other
chown alice:developers file.txt  # Change owner and group
chown -R www-data:www-data /var/www/  # Recursive ownership change

# View current permissions:
ls -la /etc/shadow         # -rw-r----- root shadow
stat /etc/passwd           # Detailed inode info including permissions

# Default permissions: umask 022 means:
#   Files: 666 - 022 = 644  (rw-r--r--)
#   Dirs:  777 - 022 = 755  (rwxr-xr-x)

Bash Scripting Basics

#!/bin/bash
# shebang line MUST be first line, specifies interpreter

# Variables
NAME="Alice"
COUNT=10
CURRENT_DATE=$(date +%Y-%m-%d)   # Command substitution

echo "Hello, $NAME. Today is $CURRENT_DATE."

# Conditional
if [ $COUNT -gt 5 ]; then
    echo "Count is greater than 5"
elif [ $COUNT -eq 5 ]; then
    echo "Count is exactly 5"
else
    echo "Count is less than 5"
fi

# For loop
for server in web01 web02 db01; do
    echo "Checking $server..."
    ping -c 1 $server > /dev/null && echo "$server: UP" || echo "$server: DOWN"
done

# While loop
i=1
while [ $i -le 3 ]; do
    echo "Iteration $i"
    ((i++))
done

# Function
check_service() {
    local SERVICE=$1
    if systemctl is-active --quiet $SERVICE; then
        echo "$SERVICE is RUNNING"
    else
        echo "$SERVICE is STOPPED"
    fi
}
check_service nginx
check_service sshd
Week 4 Network Troubleshooting Basics ▾

Core Diagnostic Tools

ping — Test ICMP Reachability

ping -c 4 8.8.8.8          # Send 4 ICMP echo requests to Google DNS
ping -c 4 google.com       # Tests DNS resolution + reachability
ping -i 0.2 192.168.1.1    # Ping every 0.2 seconds (faster)

# Output: round-trip time (RTT) in ms, TTL value
# TTL ~64 = Linux/macOS hop away; TTL ~128 = Windows hop away
# Packet loss % indicates reliability issues

traceroute / tracert — Path Discovery

traceroute google.com          # Linux: shows each hop (router) to destination
tracert google.com             # Windows equivalent
traceroute -n google.com       # -n: don't resolve hostnames (faster)

# * * * means the hop didn't respond (firewall or ICMP disabled)
# High latency at a specific hop = bottleneck at that router

nslookup & dig — DNS Queries

nslookup google.com            # Basic DNS lookup
dig google.com A               # Query A records for google.com
dig google.com MX              # Query mail exchanger records
dig @8.8.8.8 example.com       # Use specific DNS server (8.8.8.8)
dig +short google.com          # Clean output — just the IP
dig -x 142.250.80.46           # Reverse DNS lookup (PTR record)

# dig output sections: QUESTION, ANSWER, AUTHORITY, ADDITIONAL

ss / netstat — Socket Statistics

ss -tulpn                      # TCP/UDP Listening Ports with process names
ss -s                          # Summary of socket counts
ss -t state established        # Active TCP connections

netstat -an                    # All connections (older systems)
netstat -tulpn                 # Same as ss -tulpn

# Common port numbers:
# 22=SSH, 80=HTTP, 443=HTTPS, 25=SMTP, 53=DNS, 3306=MySQL, 5432=PostgreSQL

ip — Network Interface & Routing

ip addr show                   # All interfaces with IP addresses
ip addr show eth0              # Specific interface
ip link show                   # Link layer info (MAC, UP/DOWN state)
ip link set eth0 up            # Bring interface up
ip link set eth0 down          # Bring interface down

ip route show                  # Routing table
ip route add default via 192.168.1.1          # Add default gateway
ip route add 10.0.0.0/8 via 192.168.1.1       # Add specific route
ip route del 10.0.0.0/8                       # Delete route

arp -n                         # ARP table (IP to MAC mappings)

curl & wget — HTTP Testing

curl -I https://example.com           # Headers only (HTTP status code, server type)
curl -o /tmp/page.html https://example.com  # Download to file
curl -v https://example.com          # Verbose: show TLS handshake, headers
curl -u admin:password http://api/   # Basic auth

wget https://example.com/file.tar.gz  # Download file
wget -q -O /dev/null --server-response https://example.com  # Check HTTP status silently

tcpdump — Packet Capture

sudo tcpdump -i eth0                   # Capture all traffic on eth0
sudo tcpdump -i eth0 port 80           # Only HTTP traffic
sudo tcpdump -i eth0 host 8.8.8.8      # Traffic to/from specific IP
sudo tcpdump -i eth0 -w /tmp/cap.pcap  # Save to file for Wireshark analysis
sudo tcpdump -r /tmp/cap.pcap          # Read saved capture file
sudo tcpdump -i eth0 'tcp and port 443 and host 192.168.1.100'  # Complex filter

Wireshark Filters

http                           # All HTTP traffic
tcp.port == 443                # HTTPS traffic
ip.addr == 192.168.1.100       # Traffic involving specific IP
tcp.flags.syn == 1             # TCP SYN packets only (connections being established)
dns                            # All DNS queries/responses
icmp                           # Ping traffic only

Systematic Troubleshooting Workflow

When a server cannot reach an external service, work through the OSI model bottom-up:

Step 1: Check physical/link layer
  ip link show                 # Is interface UP? Any errors?

Step 2: Check IP configuration
  ip addr show                 # Do we have the correct IP + subnet mask?

Step 3: Check default gateway
  ip route show                # Is there a default route (0.0.0.0/0)?

Step 4: Ping the gateway
  ping -c 2 192.168.1.1        # Can we reach our router?

Step 5: Ping an external IP
  ping -c 2 8.8.8.8            # Can we reach the internet? (tests routing)

Step 6: Test DNS resolution
  dig @8.8.8.8 google.com      # Does DNS work with explicit server?
  dig google.com               # Does DNS work with configured server?

Step 7: Test the service
  curl -I http://service:port  # Does the application respond?
  ss -tulpn | grep :80         # Is the service actually listening?
