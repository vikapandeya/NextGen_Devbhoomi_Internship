Week 1 Lab VM Setup, Static IP & OSI/Subnetting Practice ▾

Objectives

  • Create and configure an Ubuntu Server 22.04 VM in VirtualBox
  • Configure a static IP address using Netplan
  • Understand how each OSI layer applies to a real HTTP request
  • Practice subnetting by hand — split a /24 into 4 equal subnets

Prerequisites

  • VirtualBox installed (version 7.x recommended)
  • Ubuntu Server 22.04 LTS ISO downloaded
  • At least 4 GB RAM available on host machine

Step 1 — Create Ubuntu Server VM

VirtualBox Settings for the VM:
  Name:      lab-server1
  Type:      Linux / Ubuntu (64-bit)
  RAM:       2048 MB
  Disk:      20 GB (VDI, dynamically allocated)
  Network:
    Adapter 1: NAT         (internet access)
    Adapter 2: Host-Only   (host-to-VM communication, 192.168.56.x range)

Installation:
  - Boot from ISO
  - Choose: "Install Ubuntu Server"
  - Network: leave as DHCP for now (we configure static next)
  - Storage: use entire disk
  - Create user: devuser / YourPassword123
  - Install OpenSSH server: YES
  - Wait for installation to complete and reboot

Step 2 — Configure Static IP via Netplan

# After booting, log in as devuser
# Find the name of your second interface (Host-Only NIC)
ip addr show
# Look for: enp0s8 (or enp0s3 for second adapter)

# Edit the Netplan configuration
sudo nano /etc/netplan/00-installer-config.yaml

# Replace the content with:
network:
  version: 2
  ethernets:
    enp0s3:                    # First NIC (NAT) — leave as DHCP
      dhcp4: true
    enp0s8:                    # Second NIC (Host-Only) — static IP
      dhcp4: false
      addresses:
        - 192.168.56.10/24
      nameservers:
        addresses:
          - 8.8.8.8
          - 1.1.1.1

# Apply the configuration
sudo netplan apply
YAML tip: Netplan files use YAML — indentation MUST use spaces (not tabs). Two spaces per level. A single incorrect space will cause an error.

Step 4 — OSI Layer Exercise

For an HTTP GET request from your browser to http://google.com, document what happens at each OSI layer:

Layer 7 — Application:
  Browser creates HTTP request:
  "GET / HTTP/1.1\r\nHost: google.com\r\nUser-Agent: Firefox/120\r\n\r\n"

Layer 6 — Presentation:
  If HTTPS: TLS negotiation, data encrypted
  Character encoding: UTF-8

Layer 5 — Session:
  TCP session established (SYN/SYN-ACK/ACK handshake)
  Session tracks request-response pairs

Layer 4 — Transport:
  TCP segment created:
  Source port: 54321 (ephemeral, randomly chosen)
  Destination port: 80 (HTTP) or 443 (HTTPS)
  Sequence number: tracks byte ordering

Layer 3 — Network:
  IP packet created:
  Source IP: 192.168.56.10 (your VM's IP)
  Destination IP: 142.250.80.46 (Google's IP — resolved by DNS)
  TTL: 64 (decremented by each router)

Layer 2 — Data Link:
  Ethernet frame created:
  Source MAC: your VM's MAC address
  Destination MAC: default gateway's MAC (ARP lookup)
  EtherType: 0x0800 (IPv4)

Layer 1 — Physical:
  Frame converted to electrical signals (copper) or
  light pulses (fibre) or radio waves (WiFi)
  Transmitted at physical medium's speed

Step 5 — Subnetting Worksheet

Exercise: Split 192.168.10.0/24 into 4 equal subnets

Step 1: How many bits to borrow?
  4 subnets = 2^2, so borrow 2 bits
  New prefix: /24 + 2 = /26
  New subnet mask: 255.255.255.192

Step 2: How many hosts per subnet?
  Host bits: 32 - 26 = 6 bits
  Total IPs: 2^6 = 64
  Usable hosts: 64 - 2 = 62 (subtract network + broadcast)

Step 3: Calculate the 4 subnets
  Block size: 256 / 4 = 64

Subnet 1: 192.168.10.0/26
  Network:   192.168.10.0
  First host: 192.168.10.1
  Last host:  192.168.10.62
  Broadcast:  192.168.10.63

Subnet 2: 192.168.10.64/26
  Network:    192.168.10.64
  First host: 192.168.10.65
  Last host:  192.168.10.126
  Broadcast:  192.168.10.127

Subnet 3: 192.168.10.128/26
  Network:    192.168.10.128
  First host: 192.168.10.129
  Last host:  192.168.10.190
  Broadcast:  192.168.10.191

Subnet 4: 192.168.10.192/26
  Network:    192.168.10.192
  First host: 192.168.10.193
  Last host:  192.168.10.254
  Broadcast:  192.168.10.255

Verification: 4 × 64 = 256 ✓ (accounts for entire /24)

Week 2 Lab DNS Server (BIND9) & DHCP Server Setup ▾

Objectives

  • Install and configure BIND9 as an authoritative DNS server for lab.local
  • Create forward and reverse DNS zones
  • Install and configure ISC DHCP Server
  • Test DNS resolution and DHCP lease assignment

Part A — BIND9 DNS Server

# Install BIND9
sudo apt update && sudo apt install bind9 bind9utils bind9-doc -y

# Verify installation
named -v
# Expected: BIND 9.18.x

# Edit named.conf.local to add your zone
sudo nano /etc/bind/named.conf.local
// Add to /etc/bind/named.conf.local:

// Forward zone
zone "lab.local" {
    type master;
    file "/etc/bind/db.lab.local";
};

// Reverse zone for 192.168.56.x
zone "56.168.192.in-addr.arpa" {
    type master;
    file "/etc/bind/db.192.168.56";
};
# Create the forward zone file
sudo nano /etc/bind/db.lab.local
; /etc/bind/db.lab.local — Forward zone for lab.local
$TTL    604800
@       IN      SOA     ns1.lab.local. admin.lab.local. (
                        2024010101      ; Serial (increment when you change)
                        3600            ; Refresh
                        1800            ; Retry
                        604800          ; Expire
                        86400 )         ; Negative TTL

; Name servers
@       IN      NS      ns1.lab.local.

; A Records
ns1     IN      A       192.168.56.10
server1 IN      A       192.168.56.10
server2 IN      A       192.168.56.11
gateway IN      A       192.168.56.1
www     IN      CNAME   server1         ; www is an alias for server1
# Create the reverse zone file
sudo nano /etc/bind/db.192.168.56
; /etc/bind/db.192.168.56 — Reverse zone for 192.168.56.x
$TTL    604800
@       IN      SOA     ns1.lab.local. admin.lab.local. (
                        2024010101
                        3600
                        1800
                        604800
                        86400 )

@       IN      NS      ns1.lab.local.

; PTR Records (last octet only)
10      IN      PTR     server1.lab.local.
11      IN      PTR     server2.lab.local.
1       IN      PTR     gateway.lab.local.
# Validate configuration files
sudo named-checkconf                              # Check named.conf syntax
sudo named-checkzone lab.local /etc/bind/db.lab.local       # Check forward zone
sudo named-checkzone 56.168.192.in-addr.arpa /etc/bind/db.192.168.56  # Check reverse

# Expected output: "OK"

# Restart BIND9
sudo systemctl restart bind9
sudo systemctl status bind9                       # Verify running

Test DNS Resolution

# Test forward lookup (A record)
dig @192.168.56.10 server1.lab.local A
# Expected ANSWER SECTION:
# server1.lab.local.  604800  IN  A  192.168.56.10

# Test reverse lookup (PTR record)
dig @192.168.56.10 -x 192.168.56.10
# Expected ANSWER SECTION:
# 10.56.168.192.in-addr.arpa. 604800 IN PTR server1.lab.local.

# Test CNAME
dig @192.168.56.10 www.lab.local
# Should show CNAME + A record chain

# Test from the VM itself (configure to use local DNS)
sudo resolvectl dns enp0s8 192.168.56.10
ping server1.lab.local                            # Should resolve locally

Part B — ISC DHCP Server

# Install DHCP server
sudo apt install isc-dhcp-server -y

# Configure DHCP
sudo nano /etc/dhcp/dhcpd.conf
# /etc/dhcp/dhcpd.conf
default-lease-time 600;       # 10 minute default lease
max-lease-time 7200;          # 2 hour maximum lease
authoritative;                # This server is authoritative for the subnet

# DNS update settings
ddns-update-style none;

subnet 192.168.56.0 netmask 255.255.255.0 {
    range 192.168.56.100 192.168.56.200;          # Dynamic IP pool
    option routers 192.168.56.1;                  # Default gateway
    option domain-name-servers 192.168.56.10;     # Our DNS server
    option domain-name "lab.local";               # Default search domain
    option subnet-mask 255.255.255.0;
    option broadcast-address 192.168.56.255;
}

# Static reservation for server2 (optional)
host server2 {
    hardware ethernet 08:00:27:AB:CD:EF;          # Replace with actual MAC
    fixed-address 192.168.56.11;
}
# Specify which interface to listen on
sudo nano /etc/default/isc-dhcp-server

# Set:
INTERFACESv4="enp0s8"

# Restart DHCP server
sudo systemctl restart isc-dhcp-server
sudo systemctl status isc-dhcp-server

Test DHCP

# On a second VM (lab-client1) with Host-Only adapter and DHCP enabled:
# After boot, check received IP:
ip addr show
# Should show IP in range 192.168.56.100-200

# View DHCP lease file on the server
cat /var/lib/dhcp/dhcpd.leases
# Should show the client's lease entry

# View client's lease
cat /var/lib/dhcp/dhclient.leases    # On the client VM

# Monitor DHCP activity
sudo journalctl -u isc-dhcp-server -f

Exercises 11–15: Disk Monitor Script

# Exercise 11: Create the monitoring script
sudo tee /usr/local/bin/disk_monitor.sh <<'EOF'
#!/bin/bash
# Disk usage monitoring script
THRESHOLD=80
LOG_FILE="/var/log/disk_monitor.log"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')

df -H | grep -vE '^Filesystem|tmpfs|cdrom' | awk '{ print $5 " " $1 }' | while read output; do
    usep=$(echo "$output" | awk '{ print $1}' | cut -d'%' -f1)
    partition=$(echo "$output" | awk '{ print $2 }')

    if [ "$usep" -ge "$THRESHOLD" ]; then
        echo "$TIMESTAMP: ALERT - Partition $partition is ${usep}% full" | tee -a "$LOG_FILE"
        logger "DISK ALERT: $partition is ${usep}% full"
    else
        echo "$TIMESTAMP: OK - $partition is ${usep}% full" >> "$LOG_FILE"
    fi
done
EOF

# Exercise 12: Make it executable and test it
sudo chmod +x /usr/local/bin/disk_monitor.sh
sudo /usr/local/bin/disk_monitor.sh
# Expected: OK messages for each partition (unless disk is >80% full)

# Exercise 13: View the log output
cat /var/log/disk_monitor.log
# Expected: timestamped OK or ALERT messages for each partition

# Exercise 14: Schedule with cron (runs every 5 minutes)
(crontab -l 2>/dev/null; echo "*/5 * * * * /usr/local/bin/disk_monitor.sh") | crontab -
crontab -l
# Expected: the cron entry is listed

# Exercise 15: Verify cron ran (wait 5+ minutes)
tail -f /var/log/disk_monitor.log
# Ctrl+C to stop watching
# Check system cron log:
sudo grep CRON /var/log/syslog | tail -5

Week 4 Lab Network Diagnostics — 5 Scenario Troubleshooting ▾

Objectives

  • Apply a systematic diagnostic process to identify and fix 5 real network problems
  • Use tcpdump to capture and analyze a TCP 3-way handshake
  • Understand common misconfiguration patterns and their symptoms
Lab Setup: An instructor script breaks different configs on the VM. Students must diagnose and fix them without being told what was broken. The scenarios below describe the symptoms you will encounter and the fix.

Scenario 1 — Wrong Default Gateway

# SYMPTOM: Can ping 192.168.56.1 (local gateway) but cannot reach 8.8.8.8

# STEP 1: Confirm the symptom
ping -c 2 192.168.56.1        # Should succeed (local)
ping -c 2 8.8.8.8             # Should FAIL (external)

# STEP 2: Check the routing table
ip route show
# Broken output example:
# 192.168.56.0/24 dev enp0s8 proto kernel scope link src 192.168.56.10
# (Missing: default via X.X.X.X)
# OR the default route points to a wrong gateway

# STEP 3: Fix — add correct default route temporarily
sudo ip route add default via 192.168.56.1
ping -c 2 8.8.8.8             # Should now succeed

# STEP 4: Make the fix permanent (Netplan)
sudo nano /etc/netplan/00-installer-config.yaml
# Add under enp0s8:
#   routes:
#     - to: 0.0.0.0/0
#       via: 192.168.56.1
sudo netplan apply

# VERIFICATION:
ip route show | grep default
# Expected: default via 192.168.56.1 dev enp0s8

Scenario 2 — DNS Misconfiguration

# SYMPTOM: ping 8.8.8.8 works (IP reachable) but ping google.com fails
# Error: "ping: google.com: Name or service not known"

# STEP 1: Confirm DNS is the problem
ping -c 2 8.8.8.8             # Succeeds — internet works
ping -c 2 google.com          # FAILS — DNS broken

# STEP 2: Test DNS with explicit server
dig @8.8.8.8 google.com       # Should SUCCEED (DNS works if we specify server)
dig google.com                # FAILS — configured DNS server is bad

# STEP 3: Check configured DNS server
cat /etc/resolv.conf
# Broken output:
# nameserver 192.168.56.99    (wrong/nonexistent DNS server)

# STEP 4: Fix using systemd-resolved
sudo resolvectl dns enp0s8 8.8.8.8
resolvectl status              # Verify DNS server set

# OR edit Netplan to set correct nameservers
sudo nano /etc/netplan/00-installer-config.yaml
# Ensure nameservers.addresses: [8.8.8.8, 1.1.1.1]
sudo netplan apply

# VERIFICATION:
dig google.com | grep "ANSWER SECTION" -A 1
# Expected: google.com shows an IP address

Scenario 3 — Firewall Blocking Port 80

# SYMPTOM: nginx is running but curl from other VM times out

# STEP 1: Verify nginx is actually running and listening
sudo systemctl status nginx           # Should show: active (running)
ss -tulpn | grep :80
# Expected: tcp LISTEN 0 511 0.0.0.0:80 nginx

# STEP 2: Test locally (same machine)
curl -s http://localhost              # Should return HTML

# STEP 3: Check from other VM — times out
# (On server2): curl -m 5 http://192.168.56.10/
# Expected: curl: (28) Connection timed out

# STEP 4: Check iptables for blocking rules
sudo iptables -L -n -v
# Look for: REJECT or DROP rule before any ACCEPT rule for port 80

# STEP 5: Fix — allow port 80
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
# OR if there's a specific REJECT rule, remove it:
sudo iptables -D INPUT -p tcp --dport 80 -j REJECT

# STEP 6: Test again from server2
# (On server2): curl -s http://192.168.56.10/
# Expected: nginx default HTML page

# VERIFICATION:
sudo iptables -L INPUT -n -v | grep "dpt:80"
# Expected: ACCEPT line for port 80

Scenario 4 — Wrong Subnet Mask

# SYMPTOM: Two VMs on "same network" cannot communicate
# Both have 192.168.56.x IPs but pings fail

# STEP 1: Check IP configuration
ip addr show enp0s8
# Broken output:
# inet 192.168.56.10/32 brd 192.168.56.10 scope global enp0s8
#                   ^^^  /32 means "this host only" — no LAN connectivity!

# STEP 2: Understand the problem
# /32 mask = 255.255.255.255 — the host thinks it's isolated
# No ARP requests sent, cannot reach any other host

# STEP 3: Check ARP table
arp -n
# Empty or missing 192.168.56.1

# STEP 4: Fix — remove wrong IP and add with correct mask
sudo ip addr del 192.168.56.10/32 dev enp0s8
sudo ip addr add 192.168.56.10/24 dev enp0s8
# Bring interface back up if needed:
sudo ip link set enp0s8 up

# STEP 5: Make permanent in Netplan
sudo nano /etc/netplan/00-installer-config.yaml
# Ensure addresses: [192.168.56.10/24]  (not /32)
sudo netplan apply

# VERIFICATION:
ip addr show enp0s8 | grep inet
# Expected: inet 192.168.56.10/24 brd 192.168.56.255
ping -c 2 192.168.56.11       # Should now succeed

Scenario 5 — DHCP Wrong Interface

# SYMPTOM: DHCP server running but clients not getting IP addresses

# STEP 1: Check if DHCP server is running
sudo systemctl status isc-dhcp-server
# May show: running but with warnings

# STEP 2: Check DHCP server logs
sudo journalctl -u isc-dhcp-server -n 30
# Look for: "Not configured to listen on any interfaces"
# OR:       "can't open /var/lib/dhcp/dhcpd.leases"

# STEP 3: Check interface configuration
cat /etc/default/isc-dhcp-server
# Broken: INTERFACESv4=""   (empty or wrong interface name)

# STEP 4: Fix — set the correct interface
sudo nano /etc/default/isc-dhcp-server
# Change to: INTERFACESv4="enp0s8"
sudo systemctl restart isc-dhcp-server

# STEP 5: Monitor for client leases
sudo journalctl -u isc-dhcp-server -f
# Then on client VM, request new DHCP lease:
# sudo dhclient -v enp0s3

# VERIFICATION:
cat /var/lib/dhcp/dhcpd.leases
# Should show a new lease entry with client MAC and IP
sudo systemctl status isc-dhcp-server
# Expected: active (running), no errors

tcpdump Lab — Capture a TCP Handshake

# Start capture in background (captures port 80 traffic)
sudo tcpdump -i enp0s8 -w /tmp/http.pcap port 80 &
TCPDUMP_PID=$!

# Wait a moment then generate some HTTP traffic
sleep 1
curl -s http://server2.lab.local/ > /dev/null   # Make HTTP request to server2

# Stop capture
sleep 1
kill $TCPDUMP_PID
echo "Capture stopped"

# Analyze the capture
sudo tcpdump -r /tmp/http.pcap -n
# Expected output shows 3-way handshake:
# XX:XX:XX.XXXXX IP 192.168.56.10.54321 > 192.168.56.11.80: Flags [S]  ← SYN
# XX:XX:XX.XXXXX IP 192.168.56.11.80 > 192.168.56.10.54321: Flags [S.] ← SYN-ACK
# XX:XX:XX.XXXXX IP 192.168.56.10.54321 > 192.168.56.11.80: Flags [.]  ← ACK
# ... HTTP data exchange follows ...
# XX:XX:XX.XXXXX Flags [F.] ← FIN (connection close)

TCP Flag Reference

Flagtcpdump symbolMeaning
SYN[S]Synchronise — start connection
SYN-ACK[S.]Server accepts connection
ACK[.]Acknowledge received data
FIN[F]Finish — close connection
RST[R]Reset — abort connection (port closed/error)
PSH[P]Push — send data immediately (no buffering)
