⚙️ Week 9 Lab — Ansible Automation with Roles

▼

🎯 Objectives

  • Write a complete Ansible playbook using roles to configure web servers
  • Implement nginx install, template deployment, UFW, user creation, and service management
  • Use Ansible Vault to protect sensitive variables
  • Verify idempotency — second run must show zero changes

1 Project Directory Structure

Create the following layout before writing any YAML. This structure follows the standard Ansible roles convention.

ansible-project/ ├── inventory ├── playbook.yml ├── ansible.cfg ├── group_vars/ │ └── webservers.yml └── roles/ └── nginx/ ├── tasks/ │ └── main.yml ├── handlers/ │ └── main.yml ├── templates/ │ ├── nginx.conf.j2 │ └── index.html.j2 ├── files/ └── vars/ └── main.yml
# Create the structure in one command:
mkdir -p ansible-project/{group_vars,roles/nginx/{tasks,handlers,templates,files,vars}}
cd ansible-project

2 inventory file

Define your three web server targets. Adjust ansible_host IPs to match your VirtualBox network.

[webservers]
web1 ansible_host=192.168.56.11 ansible_user=ubuntu
web2 ansible_host=192.168.56.12 ansible_user=ubuntu
web3 ansible_host=192.168.56.13 ansible_user=ubuntu

[webservers:vars]
ansible_ssh_private_key_file=~/.ssh/lab_key
ansible_python_interpreter=/usr/bin/python3

3 ansible.cfg

Project-level configuration that overrides global defaults. Place in the project root.

[defaults]
inventory = inventory
remote_user = ubuntu
host_key_checking = False
roles_path = roles
retry_files_enabled = False

4 group_vars/webservers.yml

Group variables are automatically loaded for any host in the [webservers] group. These variables are available in all tasks and templates.

nginx_port: 80
admin_user: webadmin
site_name: "Lab Web Server"

  - 22
  - 80
  - 443

5 roles/nginx/vars/main.yml

Role variables that reference Vault-encrypted values. Vault variables use the vault_ prefix convention.

---
db_password: "{{ vault_db_password }}"
app_secret_key: "{{ vault_app_secret_key }}"

6 roles/nginx/tasks/main.yml

The main task file for the nginx role. Tasks run top-to-bottom. Notice how notify triggers handlers, and loop iterates over the allowed_ports list.

---
- name: Update apt cache
  apt:
    update_cache: yes
    cache_valid_time: 3600

- name: Install nginx
  apt:
    name: nginx
    state: present
  notify: restart nginx

- name: Deploy nginx config
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/sites-available/default
    owner: root
    group: root
    mode: '0644'
  notify: reload nginx

- name: Deploy index page
  template:
    src: index.html.j2
    dest: /var/www/html/index.html
    owner: www-data
    group: www-data
    mode: '0644'

- name: Create admin user
  user:
    name: "{{ admin_user }}"
    groups: sudo
    shell: /bin/bash
    create_home: yes
    state: present

- name: Set up SSH key for admin user
  authorized_key:
    user: "{{ admin_user }}"
    state: present
    key: "{{ lookup('file', '~/.ssh/lab_key.pub') }}"

- name: Install UFW
  apt:
    name: ufw
    state: present

- name: Configure UFW rules
  ufw:
    rule: allow
    port: "{{ item }}"
    proto: tcp
  loop: "{{ allowed_ports }}"

- name: Enable UFW
  ufw:
    state: enabled
    policy: deny

- name: Ensure nginx is enabled and started
  service:
    name: nginx
    state: started
    enabled: yes

7 roles/nginx/handlers/main.yml

Handlers run once at the end of a play, only if notified. Multiple tasks can notify the same handler — it still runs only once.

---
- name: restart nginx
  service:
    name: nginx
    state: restarted

- name: reload nginx
  service:
    name: nginx
    state: reloaded

8 templates/nginx.conf.j2

Jinja2 template. Variables in {{ }} are replaced at deploy time with values from inventory, group_vars, and role vars.

server {
    listen {{ nginx_port }};
    server_name {{ inventory_hostname }};
    root /var/www/html;
    index index.html;
    access_log /var/log/nginx/access.log;
    error_log /var/log/nginx/error.log;

    location / {
        try_files $uri $uri/ =404;
    }
}

11 Ansible Vault — Encrypting Secrets

Never commit plaintext passwords. Ansible Vault encrypts secrets files at rest. The vault password is supplied at runtime.

# Create and encrypt the secrets file:
ansible-vault encrypt group_vars/secrets.yml

# Inside the file, add vault-prefixed variables:
vault_db_password: "SecurePass123"
vault_app_secret_key: "MyAppSecret456"

# Run playbook with vault decryption:
ansible-playbook playbook.yml --ask-vault-pass

# Or use a vault password file (recommended for CI):
echo "MyVaultPass" > .vault_pass
chmod 600 .vault_pass
ansible-playbook playbook.yml --vault-password-file .vault_pass
⚠️ Add .vault_pass to your .gitignore. Never commit the vault password file to version control.

12 Run and Verify Idempotency

Idempotency is the hallmark of well-written Ansible. The first run makes changes; subsequent runs should show changed=0.

# First run (verbose):
ansible-playbook playbook.yml -i inventory -v

# Expected output (first run):
# PLAY RECAP ****
# web1 : ok=12 changed=8 unreachable=0 failed=0
# web2 : ok=12 changed=8 unreachable=0 failed=0
# web3 : ok=12 changed=8 unreachable=0 failed=0

# Second run (idempotency test):
ansible-playbook playbook.yml -i inventory -v

# Expected output (second run — this is what we want!):
# PLAY RECAP ****
# web1 : ok=12 changed=0 unreachable=0 failed=0
# web2 : ok=12 changed=0 unreachable=0 failed=0
# web3 : ok=12 changed=0 unreachable=0 failed=0

13 Ad-Hoc Commands — Quick Checks

Ad-hoc commands let you run single tasks without writing a playbook. Useful for quick checks and one-off operations.

# Ping all webservers:
ansible webservers -m ping

# Check nginx version on all servers:
ansible webservers -m command -a "nginx -v"

# Restart nginx on all servers (with privilege escalation):
ansible webservers -m service -a "name=nginx state=restarted" -b

# Check disk usage on all servers:
ansible webservers -m shell -a "df -h" -b

# Gather all facts about web1:
ansible web1 -m setup
✅ Week 9 Checkpoint — All of these must be true:
  • All 3 web servers (web1, web2, web3) serve a custom index page via nginx
  • UFW is enabled on all servers with only ports 22, 80, 443 allowed
  • Admin user webadmin exists with sudo access and SSH key authentication
  • Second playbook run shows changed=0 on all hosts (idempotency verified)
  • Ad-hoc ansible webservers -m ping returns SUCCESS for all 3 hosts

⚖️ Week 10 Lab — HAProxy + Keepalived High Availability

▼

🎯 Objectives

  • Deploy HAProxy as a load balancer across two backend web servers
  • Configure Keepalived with VRRP for a floating VIP (Virtual IP)
  • Demonstrate automatic failover when the MASTER load balancer fails
  • Enable and use the HAProxy statistics page for monitoring
🗺️ Lab Topology
web1192.168.56.11
Backend (nginx)
web2192.168.56.12
Backend (nginx)
lb1 — MASTER192.168.56.20
HAProxy + Keepalived
lb2 — BACKUP192.168.56.21
HAProxy + Keepalived
VIP192.168.56.100
Floating (VRRP)

1 Prepare Backend Web Servers

Each backend server must serve a unique page so you can confirm round-robin is working. Run on each respective server.

# On web1 (192.168.56.11):
echo "<h1>Backend Server 1 - $(hostname)</h1>" | sudo tee /var/www/html/index.html

# On web2 (192.168.56.12):
echo "<h1>Backend Server 2 - $(hostname)</h1>" | sudo tee /var/www/html/index.html

# Verify both are serving:
curl http://192.168.56.11
curl http://192.168.56.12

2 Install HAProxy on lb1 and lb2

Run this on both lb1 and lb2.

sudo apt update && sudo apt install haproxy -y
haproxy -v  # Verify version: 2.4+

3 HAProxy Configuration — /etc/haproxy/haproxy.cfg

Apply this configuration to both lb1 and lb2. The configuration is identical on both nodes — Keepalived handles which one is active.

global
    log /dev/log local0
    maxconn 50000
    user haproxy
    group haproxy
    daemon

defaults
    log global
    mode http
    option httplog
    option dontlognull
    timeout connect 5s
    timeout client 30s
    timeout server 30s
    retries 3

frontend http_front
    bind *:80
    default_backend http_back
    option forwardfor
    http-request set-header X-Forwarded-Proto http

backend http_back
    balance roundrobin
    option httpchk GET /
    http-check expect status 200
    server web1 192.168.56.11:80 check inter 2s fall 3 rise 2
    server web2 192.168.56.12:80 check inter 2s fall 3 rise 2
    cookie SERVERID insert nocache

listen stats
    bind *:8404
    stats enable
    stats uri /stats
    stats refresh 10s
    stats auth admin:haproxy123
    stats show-legends
Key parameters explained:
check inter 2s — health check every 2 seconds
fall 3 — mark server DOWN after 3 consecutive failures
rise 2 — mark server UP after 2 consecutive successes
balance roundrobin — distribute requests sequentially across backends

4 Keepalived on lb1 (MASTER)

Install and configure Keepalived on lb1. Replace enp0s8 with your actual interface name (ip addr to check).

sudo apt install keepalived -y

Edit /etc/keepalived/keepalived.conf on lb1:

global_defs {
   router_id LB1
}

vrrp_script chk_haproxy {
    script "pidof haproxy"
    interval 2
    weight 2
}

vrrp_instance VI_1 {
    state MASTER
    interface enp0s8
    virtual_router_id 51
    priority 101
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass secret123
    }
    virtual_ipaddress {
        192.168.56.100/24
    }
    track_script {
        chk_haproxy
    }
}

5 Keepalived on lb2 (BACKUP)

The BACKUP configuration is identical except for state and priority. lb2 takes over if lb1 fails.

global_defs {
   router_id LB2
}

vrrp_script chk_haproxy {
    script "pidof haproxy"
    interval 2
    weight 2
}

vrrp_instance VI_1 {
    state BACKUP
    interface enp0s8
    virtual_router_id 51
    priority 100
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass secret123
    }
    virtual_ipaddress {
        192.168.56.100/24
    }
    track_script {
        chk_haproxy
    }
}
Priority 101 (MASTER) > 100 (BACKUP). The node with the highest priority wins the VIP. Both must have the same virtual_router_id and auth_pass.

6 Enable Services on Both lb1 and lb2

sudo systemctl enable --now haproxy keepalived
sudo systemctl status haproxy keepalived

# Verify VIP is on lb1:
ip addr show enp0s8 | grep 192.168.56.100
# Should show the VIP on lb1 only

7 Test Load Balancing

From a client machine, send 10 requests through the VIP. You should see responses alternating between Server 1 and Server 2.

for i in {1..10}; do
  curl -s http://192.168.56.100 | grep "Backend Server"
done

# Expected output (alternating):
# <h1>Backend Server 1 - web1</h1>
# <h1>Backend Server 2 - web2</h1>
# <h1>Backend Server 1 - web1</h1>
# <h1>Backend Server 2 - web2</h1>
# ...

8 Failover Test — The Critical Demo

This is the most important test. Simulate lb1 failure and verify the VIP migrates to lb2 within 3 seconds.

# === On lb1: simulate haproxy failure ===
sudo systemctl stop haproxy

# === On lb2: verify it acquired the VIP ===
ip addr show enp0s8
# Should now show: inet 192.168.56.100/24

# === On client: continue accessing via VIP — should still work ===
for i in {1..5}; do
  curl -s http://192.168.56.100 | grep "Backend"
  sleep 1
done
# No interruption after 1-3 seconds (VRRP advertisement interval)

# === Restore lb1 ===
sudo systemctl start haproxy
# After ~2-3 seconds, lb1 reclaims the VIP (preemptive mode)
ip addr show enp0s8  # VIP back on lb1

9 HAProxy Statistics Page

Access the stats dashboard to monitor backend health in real time.

# In browser:
http://192.168.56.100:8404/stats
# Login: admin / haproxy123

# Via curl (check backend status):
curl -u admin:haproxy123 http://192.168.56.100:8404/stats;csv | grep "http_back"
The stats page shows: green/red server status, current sessions, request rates, bytes in/out, and health check results. A green row means the backend is UP.

10 Sticky Sessions Test

Verify that SERVERID cookie causes the client to always hit the same backend.

# First request — sets the sticky cookie:
curl -c cookies.txt -b cookies.txt http://192.168.56.100 -I
# Look for: Set-Cookie: SERVERID=web1

# Subsequent requests — same backend every time:
for i in {1..5}; do
  curl -c cookies.txt -b cookies.txt -s http://192.168.56.100 | grep "Backend Server"
done
# All responses should say "Backend Server 1" (or whichever web1 serves)
✅ Week 10 Checkpoint — All of these must be true:
  • Round-robin load balancing working: alternate responses from web1 and web2
  • VIP (192.168.56.100) is held by lb1 in normal operation
  • Stopping HAProxy on lb1 causes lb2 to acquire the VIP within 3 seconds
  • Traffic continues via VIP during failover (no connection refused)
  • HAProxy stats page accessible at :8404/stats showing both backends green

5 security_groups.tf — ALB and EC2 Security Groups

The EC2 security group only accepts HTTP from the ALB security group (not from the internet directly). This is a key security pattern.

resource "aws_security_group" "alb" {
  name   = "${var.project_name}-alb-sg"
  vpc_id = aws_vpc.main.id

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
  tags = { Name = "${var.project_name}-alb-sg" }
}

resource "aws_security_group" "ec2" {
  name   = "${var.project_name}-ec2-sg"
  vpc_id = aws_vpc.main.id

  # Only accept HTTP from the ALB (not the internet)
  ingress {
    from_port       = 80
    to_port         = 80
    protocol        = "tcp"
    security_groups = [aws_security_group.alb.id]
  }
  # SSH only from within VPC (bastion or VPN)
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["10.0.0.0/16"]
  }
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
  tags = { Name = "${var.project_name}-ec2-sg" }
}

8 Remote State Backend — main.tf

Remote state prevents conflicts when multiple team members run Terraform. DynamoDB provides state locking.

terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  backend "s3" {
    bucket         = "ndfi-terraform-state"
    key            = "networking/staging/terraform.tfstate"
    region         = "ap-south-1"
    dynamodb_table = "ndfi-terraform-locks"
    encrypt        = true
  }
}
Create the S3 bucket and DynamoDB table manually before running terraform init, or use a bootstrap script. The state bucket itself cannot be managed by Terraform in the same config.

9 Route 53 Health Check + Failover

5 Ingress — Path-Based Routing

# ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ndfi-ingress
  namespace: ndfi-app
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/ssl-redirect: "false"
spec:
  ingressClassName: nginx
  rules:
  - host: ndfi.lab.local
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: backend
            port:
              number: 3000
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80

6 NetworkPolicy — Isolate Database

This policy ensures only the backend pods can connect to PostgreSQL on port 5432. All other ingress is denied.

# networkpolicy-db.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: postgres-isolation
  namespace: ndfi-app
spec:
  podSelector:
    matchLabels:
      app: postgres
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: backend
    ports:
    - protocol: TCP
      port: 5432
  egress:
  - {}  # Allow all egress (for DNS resolution)

7 NetworkPolicy — Restrict Backend Ingress

The backend only accepts connections from frontend pods and the ingress-nginx namespace. Blocks direct internet access to the API.

# networkpolicy-backend.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-policy
  namespace: ndfi-app
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: ingress-nginx
    ports:
    - protocol: TCP
      port: 3000
