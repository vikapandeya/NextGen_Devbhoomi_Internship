Project Briefs — Networking & System Administration

Three capstone projects from beginner Linux admin to cloud-native infrastructure

3 Projects · Weeks 4, 8, 12
Apply your Networking & System Administration skills through three progressively complex real-world projects. Each project mimics actual infrastructure engineering tasks you'll encounter in the industry — from hardening a single Linux server to building fully automated, cloud-native infrastructure on AWS that real startups use in production.
🖥️

Project 1 — Fully Configured Linux Server

Set up a production-ready Ubuntu Server from scratch with DNS, DHCP, firewall, and web serving

Beginner Solo Project Deadline: End of Week 4
📖 Project Overview You will set up a production-ready Ubuntu Server from scratch. This mirrors the first task given to a junior Linux administrator at any IT company: take a bare server and make it fully operational, secure, and documented. By the end, a real user should be able to reach your web server, get a DHCP lease, and resolve hostnames through your DNS server — all running on one machine you configured entirely by hand.

📦 Deliverables Checklist

  • ✓Ubuntu Server 22.04 LTS installed (VirtualBox or cloud VM), snapshots taken before each major change
  • ✓Static IP configured via Netplan: 192.168.56.10/24, gateway and DNS set, persists across reboots
  • ✓Hostname set to server1.lab.local, /etc/hosts updated with all lab hosts
  • ✓SSH hardened: key-based authentication only, PasswordAuthentication no, PermitRootLogin no, non-default port optional
  • ✓UFW firewall: allow SSH (22), HTTP (80); deny all other incoming; ufw status verbose output included in report
  • ✓BIND9 DNS: authoritative zone lab.local with at least 5 A records and corresponding PTR records; passes named-checkzone lab.local /etc/bind/db.lab.local
  • ✓ISC DHCP server: serving the 192.168.56.100–200 range; lease tested on a client VM (screenshot of ip addr showing assigned IP)
  • ✓nginx: serving a custom HTML page on port 80, accessible from the host machine browser
  • ✓Cron job: disk_monitor.sh runs every 5 minutes, logs disk usage to /var/log/disk_monitor.log with timestamp
  • ✓Documentation: 1-page setup guide with all commands used, architecture diagram showing all services and their ports

📤 Submission Instructions

How to submit: GitHub repository containing: all configuration files (/etc/netplan/, /etc/bind/, /etc/dhcp/, /etc/nginx/), screenshot folder (10+ screenshots), commands log file, and your 1-page setup guide PDF or Markdown. Present via a live 5-minute demo where you demonstrate each service working from a separate client VM. Mentors will test by SSHing into your server with a key they provide.

Project 2 — Infrastructure Monitoring Stack

Build a production-grade monitoring system for a 3-node cluster with Prometheus, Grafana, Alertmanager, and HA load balancing

Intermediate Team of 2 Deadline: End of Week 8
📖 Project Overview Build a production-grade monitoring infrastructure for a 3-node cluster. This is the exact stack used by DevOps and SRE teams at companies like Freshworks, Razorpay, and CRED. You will install and configure Prometheus for metrics collection, Grafana for visualization, Alertmanager for notifications, and HAProxy + Keepalived for high availability. By the end, a simulated node crash should automatically trigger an alert email within 2 minutes.

📦 Deliverables Checklist

  • ✓3-node Ubuntu cluster (node1, node2, node3) with Ansible-provisioned base configuration (hostname, packages, users, UFW) — playbook must be idempotent
  • ✓Prometheus (v2.45+) on node1 scraping all 3 nodes via Node Exporter; all 3 targets show UP in Prometheus UI at /targets
  • ✓Grafana on node1 with 5 dashboards: CPU Usage, Memory Usage, Disk I/O, Network Throughput, Active Alerts — each with at least 3 panels
  • ✓Alertmanager: 3 alert rules configured — CPU >80% for 5 minutes, Disk <20% free for 10 minutes, Node Down for 1 minute — email notification received and demonstrated
  • ✓HAProxy load balancing Grafana across node1 (if replicated) with stats page enabled and accessible
  • ✓Keepalived VIP (192.168.56.200) on 2 HAProxy nodes with <3 second failover demonstrated live during demo
  • ✓Automation: systemd service unit + watchdog script that automatically restarts Prometheus if the process crashes
  • ✓Loki + Promtail: log aggregation from all 3 nodes, logs searchable and filterable in Grafana Explore view

Project 3 — Cloud-Native Infrastructure with IaC

Build complete AWS infrastructure using Terraform, deploy a 3-tier app on EKS, implement monitoring, and automate deployments via GitHub Actions

Advanced Team of 3 Deadline: End of Week 12
📖 Project Overview Build a complete cloud-native infrastructure on AWS using Terraform, deploy a 3-tier application on Kubernetes (EKS), implement full observability, and automate deployments via GitHub Actions. This is equivalent to a production infrastructure setup at a funded Indian startup — the same stack used at companies like Zepto, Groww, and Slice. This project is team-based because real infrastructure work is always collaborative.

🗺️ Architecture Diagram

Internet
   │
   ▼
CloudFront (CDN — static asset caching)
   │
   ▼
Route 53 (ndfi-lab.example.com — DNS + health checks)
   │
   ▼
ALB (Application Load Balancer — public subnets: ap-south-1a + ap-south-1b)
   │
   ▼
EKS Cluster (private subnets — 3x t3.medium nodes)
├── Frontend Pods (nginx — 3 replicas)
├── Backend Pods (Node.js API — 3 replicas)
└── Database (RDS PostgreSQL db.t3.micro — separate private subnet)
   │
NAT Gateway ──→ Internet (for pod outbound traffic: image pulls, updates)
   │
Monitoring Namespace:
  Prometheus + Grafana (kube-prometheus-stack via Helm)
  Alertmanager (PagerDuty + email)
   │
CI/CD:
  GitHub Actions ──→ ECR (container images) ──→ EKS (rolling deploy)
   │
Remote State:
  S3 bucket (terraform.tfstate) + DynamoDB table (state locking)

📦 Deliverables Checklist

  • ✓Terraform IaC: VPC (10.0.0.0/16), 2 public + 2 private subnets, IGW + NAT GW, EKS cluster (3x t3.medium), RDS PostgreSQL (db.t3.micro) in private subnet, ALB, CloudFront distribution, remote state in S3 + DynamoDB locking — all in modular HCL files
  • ✓Kubernetes Manifests: Deployments (3 replicas, RollingUpdate strategy), Services (ClusterIP for backend/DB, LoadBalancer for frontend), Ingress with nginx controller (path-based routing for / and /api), ConfigMap for non-secret config
  • ✓NetworkPolicy: DB isolated — only backend pods can connect on port 5432; Backend isolated — only frontend + ingress-nginx namespace can reach port 3000; tested and verified with nc
  • ✓HPA (Horizontal Pod Autoscaler): CPU threshold 70%, min 2 replicas, max 10 replicas — demonstrated scaling up under load using k6 or ab
  • ✓Monitoring: kube-prometheus-stack via Helm, 3 custom Grafana dashboards (Pod CPU/Memory, HTTP request rate & error rate, DB connection count), 2 alert rules (pod CrashLoopBackOff, HTTP error rate >5%)
  • ✓CI/CD Pipeline (.github/workflows/deploy.yml): on push to main → Docker build → push to ECR with commit SHA tag → kubectl set image to update deployment → kubectl rollout status to verify zero-downtime deploy
  • ✓Security: all secrets in AWS Secrets Manager or Kubernetes Secrets (no hardcoded credentials anywhere in code), SGs follow least-privilege principle, all NetworkPolicies applied and tested, terraform plan shows no security group rules open to 0.0.0.0/0 for SSH
  • ✓Ansible: playbook for any EC2 bastion host configuration (hardening, tool installation) or EKS worker node bootstrap customization — idempotent

📤 Submission Instructions

How to submit: GitHub monorepo with the following structure: terraform/ (all .tf files), k8s/ (all manifests), ansible/ (playbooks), .github/workflows/ (CI/CD pipeline), docs/ (architecture diagram PNG, cost estimate). Present a 10-minute live demo showing: terraform apply output, end-to-end app working via browser, a GitHub Actions pipeline run completing successfully, Grafana dashboards, and one alert firing. Include a cost estimate screenshot from AWS Pricing Calculator.
⚠️ AWS Cost Warning: This infrastructure costs approximately $120–180/month if left running continuously. Use AWS Free Tier resources where possible (t3.micro for RDS falls under free tier for 12 months). Always run terraform destroy after completing demos to stop all charges. The main cost drivers are: EKS cluster (~$72/month), NAT Gateway (~$32/month), and ALB (~$16/month). Use --auto-approve only in CI — always review the plan output before applying manually.
