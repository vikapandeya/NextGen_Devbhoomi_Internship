Week 5 — React.js Fundamentals
W5

React.js — Component-Based UI

JSX · Components · Props · State · Hooks

▼
Analogy: Think of React components like LEGO bricks. Each brick is a small, reusable piece. You assemble them to build complex UIs. Change one brick's design, and everywhere it's used updates automatically.

Why React?

  • Built by Facebook (Meta) in 2013
  • #1 most-used frontend framework in 2026
  • Component-based: reuse UI pieces across pages
  • Virtual DOM: only updates what changed (fast)
  • Massive job market demand

JSX — JavaScript + HTML

  • JSX lets you write HTML-like syntax inside JavaScript
  • className not class (class is a JS keyword)
  • htmlFor not for
  • Every JSX expression must return one root element
  • Use <>...</> (Fragment) to wrap without a div

Components & Props

  • Component = a function that returns JSX
  • Component names must start with capital letter
  • Props = data passed from parent to child
  • Props are read-only — never modify them
  • Use { } to embed JavaScript in JSX

useState Hook

  • const [count, setCount] = useState(0)
  • count is the current value
  • setCount updates the value
  • When state changes, React re-renders the component
  • Never mutate state directly — always use setter
// A complete React component with state
import { useState } from 'react';

function Counter({ title }) {
  const [count, setCount] = useState(0);

  return (
    <>
      <h2>{title}</h2>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>-</button>
    </>
  );
}

export default Counter;

// Using the component with props
<Counter title="My Counter" />

useEffect Hook

  • Runs code after the component renders
  • Use for: API calls, timers, subscriptions
  • Second argument = dependency array
  • [] empty array = run only once (on mount)
  • [id] = run when id changes

Rendering Lists

  • Use .map() to render arrays
  • Each item needs a unique key prop
  • Use the item's ID as key, not the index
  • Keys help React update the DOM efficiently

Week 5 Key Takeaways

  • Components are just functions that return JSX — start simple
  • useState is your most-used hook — master it first
  • Never mutate state directly: always use the setter function
  • useEffect(fn, []) = run once on mount — perfect for initial API calls
W6

Node.js & Express — Backend Basics

Node runtime · Express server · Routing · Middleware

▼
Analogy: A restaurant: React is the dining room (what customers see), Node/Express is the kitchen (where orders are processed). The kitchen never talks directly to customers — it sends food (data) to the dining room through a defined menu (REST API).

What is Node.js?

  • JavaScript runtime built on Chrome's V8 engine
  • Run JavaScript on the server (not just browsers)
  • Non-blocking, event-driven I/O — handles many requests
  • npm: world's largest package registry
  • Same language on front and back end

Express.js

  • Minimal web framework for Node.js
  • Handles HTTP routes: GET, POST, PUT, DELETE
  • Middleware: functions that run between request and response
  • app.use() registers middleware globally
  • Most popular Node.js framework by far

HTTP Methods

  • GET — retrieve data (safe, no body)
  • POST — create new data (sends body)
  • PUT — replace existing data
  • PATCH — update partial data
  • DELETE — remove data
  • REST uses these methods + URLs together

Middleware Concept

  • Middleware = function(req, res, next)
  • Runs before your route handler
  • Must call next() to pass to next middleware
  • Common: body-parser, cors, auth, logging
  • Order matters — middleware runs top to bottom
// Minimal Express server
const express = require('express');
const app = express();

app.use(express.json()); // parse JSON bodies

const tasks = [
  { id: 1, title: 'Learn React', done: false },
  { id: 2, title: 'Build API', done: false },
];

// GET /api/tasks — return all tasks
app.get('/api/tasks', (req, res) => {
  res.json(tasks);
});

// POST /api/tasks — create new task
app.post('/api/tasks', (req, res) => {
  const { title } = req.body;
  const task = { id: Date.now(), title, done: false };
  tasks.push(task);
  res.status(201).json(task); // 201 = Created
});

app.listen(3001, () => console.log('Server on port 3001'));

Week 6 Key Takeaways

  • Express routes map HTTP method + URL to a handler function
  • Always use app.use(express.json()) before reading req.body
  • Return correct HTTP status codes: 200=OK, 201=Created, 404=Not Found, 500=Server Error
  • Test every API endpoint in Postman before connecting your frontend

REST APIs & Database Integration

MySQL · SQL CRUD · API + DB connection · Async/Await

▼

REST API Design Principles

  • Resources as nouns in URLs: /users, /tasks
  • HTTP methods as verbs (GET, POST, DELETE)
  • Stateless: each request is self-contained
  • Return JSON with consistent structure
  • Use plural nouns: /tasks not /task
  • Nest for relationships: /users/5/tasks

MySQL Basics

  • Relational database — data in tables with rows/columns
  • SQL: Structured Query Language
  • SELECT * FROM tasks WHERE done = 0
  • INSERT INTO tasks (title) VALUES ('Learn SQL')
  • UPDATE tasks SET done = 1 WHERE id = 5
  • DELETE FROM tasks WHERE id = 5

Async/Await

  • Database calls are asynchronous (take time)
  • async function enables await inside
  • await pauses until the promise resolves
  • Wrap in try/catch for error handling
  • Cleaner than nested .then().then() chains

Environment Variables

  • Never hardcode passwords in your code
  • Store secrets in .env file
  • Add .env to .gitignore — never commit it
  • Use dotenv package: require('dotenv').config()
  • Access: process.env.DB_PASSWORD
// Express route with MySQL + async/await
const mysql = require('mysql2/promise');

const db = await mysql.createPool({
  host: process.env.DB_HOST,
  user: process.env.DB_USER,
  password: process.env.DB_PASS,
  database: process.env.DB_NAME,
});

app.get('/api/tasks', async (req, res) => {
  try {
    const [rows] = await db.query('SELECT * FROM tasks ORDER BY id DESC');
    res.json(rows);
  } catch (err) {
    console.error(err);
    res.status(500).json({ error: 'Database error' });
  }
});

Week 7 Key Takeaways

  • NEVER hardcode database credentials — always use .env + process.env
  • Always add .env to .gitignore before your first commit
  • Use parameterised queries (? placeholders) to prevent SQL injection
  • Return meaningful error messages to the client, but log full errors on the server
W8

React + API Integration & Deployment

Fetch API · CORS · Vercel + Render deployment · Auth intro

▼

Fetching Data in React

  • Use fetch() or axios library
  • Call API in useEffect with []
  • Track three states: loading, data, error
  • Show spinner while loading, error UI on failure
  • axios: cleaner syntax, auto JSON parse, interceptors

CORS

  • Browser blocks requests between different origins
  • Enable in Express: app.use(cors())
  • Install: npm install cors
  • In production: restrict to your frontend's domain only
  • CORS is a browser security feature, not a server bug

Deploying Frontend (Vercel)

  • Push code to GitHub
  • Import repo at vercel.com
  • Vercel auto-detects React/Vite, builds, deploys
  • Every push = auto-redeploy
  • Free tier is enough for internship projects

Deploying Backend (Render)

  • Push Express app to GitHub
  • Create Web Service at render.com
  • Add environment variables in Render dashboard
  • Free tier: sleeps after 15 min inactivity
  • Add start script: "start": "node index.js"
// React: fetch tasks from Express API
import { useState, useEffect } from 'react';

function TaskList() {
  const [tasks, setTasks] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch('https://your-api.onrender.com/api/tasks')
      .then(res => {
        if (!res.ok) throw new Error('API error');
        return res.json();
      })
      .then(data => { setTasks(data); setLoading(false); })
      .catch(err => { setError(err.message); setLoading(false); });
  }, []);

  if (loading) return <p>Loading...</p>;
  if (error)   return <p>Error: {error}</p>;

  return (
    <ul>
      {tasks.map(t => <li key={t.id}>{t.title}</li>)}
    </ul>
  );
}

Week 8 Key Takeaways

  • Always handle three states in your data-fetching components: loading, success, error
  • Add cors() middleware to Express before any routes — frontend will silently fail without it
  • Store your API base URL in a React environment variable (VITE_API_URL) not hardcoded
  • Set up Vercel + Render deployments early — deploying often catches bugs that local dev hides
