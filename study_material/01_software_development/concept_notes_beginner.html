Week 1 — HTML5 Foundations
W1

HTML5 — Structure of the Web

Semantic tags · Document structure · Forms · Accessibility basics

▼
Real-world analogy: Think of an HTML document like a house blueprint. HTML provides the structure — walls, rooms, doors — but no decoration yet. CSS adds the paint and furniture later. JavaScript makes the doors open and lights turn on.

What is HTML?

  • HyperText Markup Language
  • Describes the structure of web pages
  • Uses tags like <p>, <h1>, <div>
  • NOT a programming language — it's a markup language
  • Current version: HTML5 (2014, still evolving)

Document Structure

  • <!DOCTYPE html> — tells browser this is HTML5
  • <html lang="en"> — root element, language declared
  • <head> — metadata, title, CSS links (not visible)
  • <body> — everything the user sees
  • Always close every tag you open

Semantic HTML5 Tags

  • <header> — top section of a page or section
  • <nav> — navigation links
  • <main> — primary page content (use once)
  • <section> — a thematic grouping
  • <article> — self-contained content
  • <footer> — bottom of page/section
  • <aside> — sidebar content

Why Semantic Tags Matter

  • Screen readers use them for accessibility
  • Google uses them for SEO ranking
  • Easier for other developers to read your code
  • <div> has no meaning — use it only for layout
  • Industry standard — companies require it
<!-- A well-structured HTML5 page -->
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>My Portfolio</title>
</head>
<body>
  <header>
    <nav>
      <a href="#about">About</a>
      <a href="#projects">Projects</a>
    </nav>
  </header>
  <main>
    <section id="about">
      <h1>Hi, I am Priya</h1>
      <p>A frontend developer from Uttarakhand.</p>
    </section>
  </main>
  <footer>
    <p>© 2026 Priya Sharma</p>
  </footer>
</body>
</html>

HTML Forms

  • <form action="" method="post">
  • <input type="text"> — text input
  • <input type="email"> — validates email format
  • <textarea> — multi-line input
  • <button type="submit"> — submit button
  • Always add label for accessibility

Important HTML Attributes

  • id — unique identifier (one per page)
  • class — reusable group identifier
  • href — link destination on <a>
  • src — file source on <img>
  • alt — image description (required for accessibility)
  • placeholder — hint text in inputs

Week 1 Key Takeaways

  • Always declare <!DOCTYPE html> at the very top
  • Use semantic tags (<header>, <main>, <footer>) instead of <div> for structure
  • Every <img> must have an alt attribute
  • The <head> is for the browser, <body> is for the user
  • Validate your HTML at validator.w3.org before submitting any work
Further Reading
  • MDN Web Docs — HTML basics (developer.mozilla.org)
  • W3C HTML5 specification overview
  • WebAIM — Introduction to Web Accessibility
W2

CSS3 — Styling & Layout

Box model · Flexbox · Colors · Typography · CSS variables

▼
Real-world analogy: If HTML is the house's blueprint (walls and rooms), CSS is the interior decorator — choosing paint colors, furniture placement, and the size of everything. The same HTML can look completely different with different CSS.

The CSS Box Model

  • Every element is a rectangular box
  • Content — the actual text/image inside
  • Padding — space inside the border
  • Border — line around padding
  • Margin — space outside the border
  • Use box-sizing: border-box always — it makes sizing predictable

CSS Selectors

  • p — selects all paragraphs
  • .card — selects elements with class "card"
  • #hero — selects element with id "hero"
  • nav a — selects links inside nav
  • button:hover — selects hovered buttons
  • Be specific but not over-specific

CSS Flexbox

  • display: flex on parent activates flex
  • justify-content — horizontal alignment
  • align-items — vertical alignment
  • gap — space between items
  • flex-wrap: wrap — items wrap on small screens
  • Most useful for navigation bars and card grids

CSS Variables (Custom Properties)

  • Defined at :root { --color-navy: #1B3A6B; }
  • Used as color: var(--color-navy);
  • Change once at :root, updates everywhere
  • Industry standard — all professional codebases use them
  • Dark mode = just update the variables
/* Professional CSS structure */
:root {
  --color-primary: #1B3A6B;
  --color-accent:  #C8941A;
  --font-body:     'Segoe UI', sans-serif;
}

* {
  margin: 0;
  padding: 0;
  box-sizing: border-box; /* always include this */
}

.card {
  background: white;
  border-radius: 12px;
  padding: 24px;
  border: 1px solid #e0e0e0;
  box-shadow: 0 2px 8px rgba(0, 0, 0, 0.08);
}

.nav {
  display: flex;
  justify-content: space-between;
  align-items: center;
  gap: 16px;
}

Typography in CSS

  • font-family — typeface name
  • font-size — use rem (relative to root, accessible)
  • font-weight — 400 = normal, 700 = bold
  • line-height — 1.5–1.7 for body text
  • letter-spacing — for headings and labels
  • Import Google Fonts with @import url(...)

Colors in CSS

  • Hex: #1B3A6B — most common
  • RGB: rgb(27, 58, 107)
  • RGBA: rgba(27, 58, 107, 0.5) — with transparency
  • HSL: hsl(220, 60%, 26%) — easiest to adjust
  • Always check contrast ratio ≥ 4.5:1 for text

Week 2 Key Takeaways

  • Always put *, *::before, *::after { box-sizing: border-box; } at the top of every stylesheet
  • Use CSS variables (--variable-name) from day one — it's the professional way
  • Flexbox solves 80% of layout problems — master justify-content and align-items
  • Use rem for font sizes, not px — it respects the user's browser settings
  • Chrome DevTools (F12 → Elements) is your best debugging tool for CSS
W3

Responsive Design & CSS Grid

Media queries · CSS Grid · Mobile-first · Viewport units

▼
Real-world analogy: Responsive design is like water — it takes the shape of its container. Your page should look great whether viewed on a 5-inch phone or a 27-inch monitor. In 2026, 60% of web traffic is mobile. Non-responsive sites are unprofessional.

Mobile-First Approach

  • Start by writing CSS for the smallest screen
  • Then use @media (min-width: ...) to scale up
  • Why? It forces you to focus on essential content first
  • Mobile-first = better performance by default
  • Industry standard since 2012 — all companies use it

Media Query Breakpoints

  • 480px — small phones
  • 640px — large phones
  • 768px — tablets
  • 1024px — small laptops
  • 1200px — desktops
  • Use logical breakpoints, not device names

CSS Grid

  • display: grid on parent
  • grid-template-columns: repeat(3, 1fr) — 3 equal columns
  • gap — space between rows and columns
  • Use Grid for 2-dimensional layouts (rows + columns)
  • Use Flexbox for 1-dimensional (only rows OR columns)
  • auto-fill + minmax() = responsive without queries

Viewport Units

  • vw — percentage of viewport width
  • vh — percentage of viewport height
  • 100vh = full screen height (hero sections)
  • min(), max(), clamp() for fluid sizing
  • clamp(1rem, 4vw, 2rem) — fluid typography
/* Mobile-first responsive card grid */

.card-grid {
  display: grid;
  grid-template-columns: 1fr; /* 1 column on mobile */
  gap: 16px;
}

@media (min-width: 640px) {
  .card-grid {
    grid-template-columns: repeat(2, 1fr); /* 2 on tablets */
  }
}

@media (min-width: 1024px) {
  .card-grid {
    grid-template-columns: repeat(3, 1fr); /* 3 on desktop */
  }
}

/* Or let CSS handle it automatically: */
.card-grid-auto {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(280px, 1fr));
  gap: 20px;
}

Week 3 Key Takeaways

  • Always add <meta name="viewport" content="width=device-width, initial-scale=1.0"> — without it, responsive CSS won't work on mobile
  • Write mobile styles first (no media query), then add min-width queries for larger screens
  • CSS Grid for 2D layouts (cards, page structure), Flexbox for 1D (nav, button groups)
  • repeat(auto-fill, minmax(250px, 1fr)) creates a responsive grid with zero media queries
  • Test every page in Chrome DevTools device mode before submitting
W4

JavaScript Basics & DOM

Variables · Functions · Events · DOM manipulation · Deployment

▼
Real-world analogy: If HTML is the house structure and CSS is the decoration, JavaScript is the electricity — it makes things interactive. Lights that switch on, doors that open, appliances that run. Without JavaScript, a web page is just a static document.

JavaScript Variables

  • const — value that won't change (use by default)
  • let — value that will change
  • var — old way, avoid in modern code
  • Data types: string, number, boolean, array, object
  • Template literals: `Hello ${name}`
  • Always declare variables before using them

Functions

  • Traditional: function greet(name) { ... }
  • Arrow: const greet = (name) => { ... }
  • Arrow functions are shorter and common in modern JS
  • Parameters go in (), code runs in { }
  • return sends back a value
  • Functions are reusable blocks of logic

DOM Manipulation

  • DOM = Document Object Model (the page as a JS object)
  • document.querySelector('.btn') — find one element
  • document.querySelectorAll('li') — find all matching
  • element.textContent = 'Hello' — change text
  • element.classList.add('active') — add CSS class
  • element.style.display = 'none' — hide element

Event Listeners

  • element.addEventListener('click', handler)
  • Common events: click, input, submit, keydown
  • Handler = the function that runs when event fires
  • event.preventDefault() — stops form from reloading page
  • event.target — the element that was clicked
  • Never use onclick="" in HTML — use addEventListener
// Real-world example: interactive navigation menu

const menuBtn = document.querySelector('.menu-btn');
const navLinks = document.querySelector('.nav-links');

menuBtn.addEventListener('click', () => {
  navLinks.classList.toggle('open'); // add or remove class
});

// Form validation example
const form = document.querySelector('#contact-form');

form.addEventListener('submit', (event) => {
  event.preventDefault(); // stop page refresh

  const name = document.querySelector('#name').value.trim();

  if (name === '') {
    alert('Please enter your name');
    return;
  }

  console.log('Form submitted:', name);
});

Deploying to GitHub Pages

  • Create a GitHub account at github.com
  • Create a new repository (name it your-username.github.io)
  • Upload your HTML/CSS/JS files
  • Go to Settings → Pages → Source: main branch
  • Your site is live at https://your-username.github.io
  • Free, instant, professional URL

Chrome DevTools for JS

  • Press F12 → Console tab
  • console.log(value) — print to console (debugging)
  • Red errors = your JS has a bug
  • Sources tab → set breakpoints to pause execution
  • Network tab → see what files load
  • Never use alert() in real projects — use console

Week 4 Key Takeaways

  • Use const by default, let only when you need to reassign — never use var
  • Arrow functions (() => {}) are the modern standard — use them for event handlers
  • Always use addEventListener — never write onclick="" directly in HTML
  • querySelector is more flexible than getElementById — learn it well
  • Deploy to GitHub Pages before your Week 4 submission — a live URL is part of your grade
