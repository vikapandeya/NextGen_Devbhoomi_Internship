TypeScript — Type-Safe JavaScript

Types · Interfaces · Generics · TypeScript with React

▼
Analogy: TypeScript is like a spell-checker for code. JavaScript lets you write anything and only shows errors when you run the program. TypeScript catches errors while you type, before the app even runs. Companies love it because bugs discovered early cost 10× less to fix than bugs found in production.

Why TypeScript?

  • JavaScript + static typing = TypeScript
  • Catch type errors at compile time, not runtime
  • 70% of codebases at large companies use TS
  • Better IDE autocomplete (IntelliSense)
  • Self-documenting code — types explain what functions expect

Basic Types

  • string, number, boolean
  • string[] or Array<string> — array of strings
  • null, undefined, void
  • any — avoid, defeats the purpose
  • unknown — safer alternative to any

Interfaces

  • Define shape of an object
  • Optional properties: name?: string
  • Readonly: readonly id: number
  • Extend with extends
  • Use for API response shapes, component props

TypeScript in React

  • Create: npm create vite@latest -- --template react-ts
  • Props interface: interface CardProps { title: string; }
  • State: useState<User | null>(null)
  • React.FC for function component type
  • Event types: React.ChangeEvent<HTMLInputElement>
// TypeScript interfaces and typed components

interface Task {
  id: number;
  title: string;
  done: boolean;
  dueDate?: string; // optional
}

interface TaskCardProps {
  task: Task;
  onToggle: (id: number) => void;
}

const TaskCard: React.FC<TaskCardProps> = ({ task, onToggle }) => {
  return (
    <div className={`card ${task.done ? 'done' : ''}`}>
      <h3>{task.title}</h3>
      <button onClick={() => onToggle(task.id)}>
        {task.done ? 'Undo' : 'Complete'}
      </button>
    </div>
  );
};

Week 9 Key Takeaways

  • TypeScript is a superset of JavaScript — valid JS is valid TS, you just add types
  • Define an interface for every API response and component props object
  • Avoid any — if you reach for it, stop and think about the correct type
  • Enable strict: true in tsconfig.json from day one
W10

Testing — Unit & Integration Tests

Jest · React Testing Library · Test pyramid · Coverage

▼

Why Testing?

  • Catch bugs before they reach users
  • Refactor confidently — tests prove nothing broke
  • Code with tests is 3× cheaper to maintain
  • Companies require test coverage in job descriptions
  • Untested code = technical debt that grows

Test Pyramid

  • Unit tests (bottom, most) — one function at a time
  • Integration tests (middle) — multiple pieces together
  • E2E tests (top, fewest) — whole user flow (slow)
  • Aim for 70% unit, 20% integration, 10% E2E
  • High unit coverage + some integration = sweet spot

Jest Basics

  • describe('group', () => { }) — group tests
  • test('name', () => { }) — individual test
  • expect(value).toBe(expected) — assertion
  • toBe — strict equality
  • toEqual — deep equality (objects/arrays)
  • toThrow — expects an error thrown

React Testing Library

  • Test from user's perspective, not implementation
  • render(<Component />) — renders component
  • screen.getByText('Submit') — find by visible text
  • fireEvent.click(button) — simulate user click
  • waitFor() — wait for async updates
// Unit test with Jest
function calculateTotal(items) {
  return items.reduce((sum, item) => sum + item.price, 0);
}

describe('calculateTotal', () => {
  test('returns 0 for empty array', () => {
    expect(calculateTotal([])).toBe(0);
  });
  test('sums prices correctly', () => {
    const items = [{ price: 100 }, { price: 250 }];
    expect(calculateTotal(items)).toBe(350);
  });
});

// React component test
import { render, screen, fireEvent } from '@testing-library/react';
import Counter from './Counter';

test('increments count when button clicked', () => {
  render(<Counter title="Test" />);
  const btn = screen.getByText('+');
  fireEvent.click(btn);
  expect(screen.getByText('Count: 1')).toBeInTheDocument();
});

Week 10 Key Takeaways

  • Write tests alongside code — not as an afterthought at the end
  • Test behaviour (what it does), not implementation (how it does it)
  • Run npm test -- --coverage to see your coverage report
  • Aim for 70%+ coverage; 100% is often impractical and counterproductive

CI/CD Pipelines with GitHub Actions

Automation · Build pipelines · Auto-deploy · Quality gates

▼

What is CI/CD?

  • CI = Continuous Integration: auto-run tests on every push
  • CD = Continuous Deployment: auto-deploy when tests pass
  • Removes "works on my machine" problem
  • Every commit is validated before it reaches production
  • Industry standard at every professional company

GitHub Actions

  • Workflows defined as YAML files in .github/workflows/
  • on: [push, pull_request] — triggers
  • jobs — groups of steps (run in parallel)
  • steps — individual commands
  • Free minutes for public repos; private repos have limits

Quality Gates

  • Block merges if tests fail
  • Block merges if coverage drops below threshold
  • Run linting (ESLint) in pipeline
  • Run type checking (tsc --noEmit) in pipeline
  • Branch protection rules enforce this

Docker Basics

  • Container: isolated environment with all dependencies
  • Dockerfile: instructions to build the container image
  • docker build -t myapp . — build image
  • docker run -p 3000:3000 myapp — run container
  • Same container runs locally AND in production
# .github/workflows/ci.yml — GitHub Actions pipeline
name: CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Use Node.js 20
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Type check
        run: npx tsc --noEmit

      - name: Run tests
        run: npm test -- --coverage --watchAll=false

      - name: Build
        run: npm run build

Week 11 Key Takeaways

  • A CI pipeline that runs on every push is non-negotiable at professional companies
  • Use npm ci in CI (not npm install) — it's faster and uses exact lockfile versions
  • Add branch protection rules on GitHub so main can't be pushed to directly
  • A failing CI badge on your repo tells employers you care about code quality

System Design & Architecture Principles

SOLID · Design patterns · Scalability · Performance

▼

SOLID Principles

  • Single Responsibility — one class/function, one job
  • Open/Closed — open for extension, closed for modification
  • Liskov Substitution — subclasses must be usable as parent
  • Interface Segregation — small specific interfaces
  • Dependency Inversion — depend on abstractions

Common Design Patterns

  • Singleton — one instance shared globally
  • Observer — publish/subscribe event model
  • Repository — abstract data access layer
  • Factory — create objects without specifying class
  • MVC — Model View Controller separation

Performance Best Practices

  • React: useMemo, useCallback, React.memo
  • Lazy loading: React.lazy() + Suspense
  • Image optimization: compress, use WebP format
  • API: pagination, caching, debouncing search
  • Measure before optimizing — use Chrome Lighthouse

Scalability Concepts

  • Horizontal scaling: add more servers
  • Vertical scaling: bigger server (has limits)
  • Caching: Redis for frequently read data
  • Database indexing: indexes speed up reads
  • Rate limiting: protect API from abuse

Week 12 Key Takeaways

  • SOLID principles prevent spaghetti code — start applying S (Single Responsibility) immediately
  • Don't over-engineer early — patterns are solutions to known problems, not requirements
  • Run Lighthouse on every deployed project — aim for Performance score ≥ 90
  • Architecture is a conversation, not a rule — explain your choices in your portfolio
