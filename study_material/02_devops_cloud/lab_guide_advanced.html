Lab 9 — Deploy to Kubernetes with minikube
LAB 09 · WEEK 9
Goal: Deploy the Express app to a local Kubernetes cluster, expose it via a Service, and scale it up to 4 replicas while observing the pods being created.
1

Start minikube and Verify the Cluster

~20 min

Install minikube and kubectl, then start a local Kubernetes cluster. minikube runs a single-node K8s cluster inside a VM on your machine.

# Install minikube (Ubuntu/WSL)
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Start the cluster
minikube start --driver=docker
minikube status              # all components should show "Running"

# Verify kubectl can talk to the cluster
kubectl get nodes            # should show 1 node, STATUS=Ready
kubectl get namespaces       # default, kube-system, kube-public
Tip: minikube requires Docker Desktop to be running on Windows/WSL. Start Docker Desktop before running minikube start.
2

Write deployment.yaml

~25 min

A Deployment tells Kubernetes to always keep 2 replicas of your app running. If one dies, K8s restarts it automatically.

# k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  labels:
    app: myapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: YOUR_DOCKERHUB_USERNAME/myapp:v1
          ports:
            - containerPort: 3000
          resources:
            requests:
              memory: "64Mi"
              cpu: "100m"
            limits:
              memory: "128Mi"
              cpu: "250m"
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 10
3

Write service.yaml

~15 min

A NodePort Service makes your app accessible from outside the cluster. minikube provides a URL to access it.

# k8s/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  type: NodePort
  selector:
    app: myapp
  ports:
    - protocol: TCP
      port: 80
      targetPort: 3000
      nodePort: 30080
4

Apply Manifests and Verify

~20 min 
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

kubectl get pods             # wait for STATUS=Running on both pods
kubectl get svc              # should show myapp-service
kubectl describe deployment myapp   # full deployment details

# Get the service URL from minikube
minikube service myapp-service --url
curl <the-url>/health       # should return JSON
5

Scale Up and Watch Pods

~15 min

Scale the deployment to 4 replicas and watch Kubernetes create new pods in real time using the -w flag.

# Open two terminal windows for this step

# Terminal 1 — watch pods in real time
kubectl get pods -w

# Terminal 2 — scale up
kubectl scale deployment myapp --replicas=4

# Back in Terminal 1: watch 2 new pods appear (Pending → Running)
# This is Kubernetes' self-healing in action

kubectl get pods             # should now show 4 pods running
Deliverable: Screenshot of kubectl get pods showing all 4 replicas with STATUS=Running. Bonus: include the kubectl get pods -w output showing the 2 new pods transitioning from Pending to Running.
Lab 10 — Prometheus + Grafana Monitoring
LAB 10 · WEEK 10
Goal: Instrument the Node.js app to expose metrics, scrape them with Prometheus, visualize them in Grafana, and generate real load to see the metrics change live.
1

Add Prometheus Metrics to the App

~25 min

Install the prom-client library and expose a /metrics endpoint. Prometheus will scrape this endpoint every 15 seconds.

npm install prom-client

# Add to server.js
const client = require('prom-client');
const register = new client.Registry();

client.collectDefaultMetrics({ register });

# Custom counter for request tracking
const httpRequests = new client.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'route', 'status'],
  registers: [register]
});

app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});

# Increment counter on each request
app.use((req, res, next) => {
  res.on('finish', () => {
    httpRequests.inc({ method: req.method, route: req.path, status: res.statusCode });
  });
  next();
});
2

Add Prometheus and Grafana to docker-compose

~30 min 
# docker-compose.monitoring.yml
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"

  prometheus:
    image: prom/prometheus:latest
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml

  grafana:
    image: grafana/grafana:latest
    ports:
      - "3001:3000"
    environment:
      GF_SECURITY_ADMIN_PASSWORD: devbhoomi2026
    volumes:
      - grafana-data:/var/lib/grafana

volumes:
  grafana-data:
3

Configure Prometheus to Scrape the App

~15 min 
# prometheus.yml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

scrape_configs:
  - job_name: 'myapp'
    static_configs:
      - targets: ['app:3000']
    metrics_path: '/metrics'

# Start everything
docker-compose -f docker-compose.monitoring.yml up -d

# Verify Prometheus can reach the app
# Open http://localhost:9090 → Status → Targets
# "myapp" should show State=UP
4

Configure Grafana Dashboard

~25 min

Open Grafana, connect it to Prometheus, and import a pre-built Node.js dashboard.

Navigate to http://localhost:3001 → Login: admin / devbhoomi2026 → Configuration → Data Sources → Add → Prometheus → URL: http://prometheus:9090 → Save & Test.

Then: + → Import → enter Dashboard ID 11159 (Node.js Application Dashboard) → Import.

5

Generate Load and Watch Metrics

~20 min

Use Apache Bench to send 1000 requests and watch your Grafana dashboard respond in real time.

# Install apache bench
sudo apt install -y apache2-utils

# Send 1000 requests, 10 concurrent
ab -n 1000 -c 10 http://localhost:3000/health

# Switch to Grafana — watch request rate graph spike up
# The graphs update every 30 seconds by default
# Change the refresh to 5s (top right) during the load test

# Write a PromQL query in Prometheus to see it yourself:
# rate(http_requests_total[1m])

Lab 11 — Security Scanning in CI Pipeline
LAB 11 · WEEK 11
Goal: Add three layers of automated security scanning to your CI pipeline: container image scanning (Trivy), dependency audit (npm audit), and secret detection (Gitleaks).
1

Add Trivy Container Image Scan

~20 min

Add a security-scan job to your GitHub Actions workflow. It runs after the image is built and fails on any HIGH or CRITICAL CVE.

# Add to .github/workflows/ci.yml
  security-scan:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4

      - name: Build image for scanning
        run: docker build -t myapp:scan .

      - name: Scan image with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'myapp:scan'
          format: 'table'
          severity: 'HIGH,CRITICAL'
          exit-code: '1'
          ignore-unfixed: true
2

Add Dependency Security Audit

~15 min

npm audit checks all dependencies in package.json against the Node.js security advisory database.

# Add as a step inside the test job
      - name: Audit npm dependencies
        run: npm audit --audit-level=high
        # Fails if any HIGH or CRITICAL vulnerability found
        # Run locally first: npm audit
        # Fix issues: npm audit fix (or npm audit fix --force)
3

Add Secret Detection with Gitleaks

~15 min

Gitleaks scans the entire Git history for accidentally committed API keys, passwords, and tokens.

# Add as a separate job — runs on every push
  secret-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0    # full history needed for scanning

      - name: Detect secrets with Gitleaks
        uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
4

Introduce a Vulnerability and Confirm Block

~20 min

Deliberately use a known-vulnerable package version to verify your pipeline actually blocks deployments with vulnerabilities.

# Install an old, vulnerable version of a package
npm install lodash@4.17.4    # has known prototype pollution CVE
git add package.json package-lock.json
git commit -m "Add vulnerable lodash version (intentional test)"
git push

# Watch the pipeline — npm audit should fail the build
# Check Actions tab to confirm security-scan job is red
Warning: This is intentional for lab purposes only. Never ship code with known high-severity vulnerabilities in a real project.
5

Fix the Vulnerability

~10 min 
# Fix the vulnerable dependency
npm install lodash@latest
npm audit                    # confirm no more HIGH/CRITICAL issues

git add package.json package-lock.json
git commit -m "Fix: upgrade lodash to patched version"
git push

# Watch all security jobs turn green in GitHub Actions
Deliverable: Two screenshots: (1) the failing GitHub Actions run showing the security scan blocking the build due to the vulnerable lodash version, (2) the passing run after the fix with all security jobs green.
Lab 12 — Chaos Engineering Experiment
LAB 12 · WEEK 12
Goal: Prove your Kubernetes deployment self-heals by deliberately killing pods and watching Kubernetes automatically restore the desired state — all while traffic continues flowing.
1

Scale to 3 Replicas

~10 min 
kubectl scale deployment myapp --replicas=3
kubectl get pods             # wait for all 3 to show Running

# Get the service URL
minikube service myapp-service --url
2

Start Continuous Traffic

~5 min

Open a dedicated terminal window for this. Keep it running through steps 3–5 to prove the app stays available during chaos.

# Terminal 1 — continuous traffic loop
SERVICE_URL=$(minikube service myapp-service --url)
while true; do
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" $SERVICE_URL/health)
  echo "$(date +%H:%M:%S) — HTTP $STATUS"
  sleep 1
done
# Should print "HTTP 200" every second
# Any failures will show as non-200 codes — watch for them during chaos
3

Delete a Pod — Watch Self-Healing

~15 min

Open a second terminal. Delete one pod and watch Kubernetes immediately create a replacement — proving the Deployment controller is always working.

# Terminal 2 — watch pods in real time
kubectl get pods -w

# Terminal 3 — kill a pod
POD_NAME=$(kubectl get pods -l app=myapp -o jsonpath='{.items[0].metadata.name}')
kubectl delete pod $POD_NAME

# In Terminal 2: watch the deleted pod Terminating
# Watch a new pod appear as Pending → ContainerCreating → Running
# In Terminal 1: confirm traffic stays green (200) the whole time
4

Kill a Container Inside a Pod

~15 min

SSH into a running pod and kill the main process. Kubernetes detects the container failed and restarts it automatically.

# Get a pod name
POD=$(kubectl get pods -l app=myapp -o jsonpath='{.items[1].metadata.name}')

# Open a shell inside the pod
kubectl exec -it $POD -- /bin/sh

# Inside the pod — kill the node process
kill 1
# The shell session will close — the container exited

# Back on your machine — watch K8s restart it
kubectl get pods -w
# RESTARTS column on that pod should increment from 0 to 1
5

Simulate Node Failure

~20 min

In a multi-node cluster, this simulates a server going down. With minikube's multi-node add-on, we can demonstrate the concept.

# Add a second node to minikube
minikube node add
kubectl get nodes            # should now show 2 nodes

# Verify pods spread across both nodes
kubectl get pods -o wide     # NODE column shows which node each pod runs on

# Stop one node (simulates server failure)
minikube node stop m02       # stop the second node

# Watch K8s reschedule pods from the failed node to the remaining node
kubectl get pods -w
# Pods that were on m02 will be rescheduled to minikube (node 1)

# Restart the node when done
minikube node start m02
Tip: Write down what you observe in each step. This forms your "chaos experiment report" — expected outcome, actual outcome, recovery time. SRE teams document this in a Game Day report.
Deliverable: Screenshot of kubectl get pods -w showing a pod being Terminated and a new pod immediately entering ContainerCreating → Running. Also include 2–3 lines from Terminal 1 showing HTTP 200 responses throughout the experiment — proving the app never went down.
