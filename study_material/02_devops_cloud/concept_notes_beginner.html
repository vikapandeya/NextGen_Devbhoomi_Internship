Beginner Level — DevOps & Cloud Concept Notes

4 weeks: Linux fundamentals, Git version control, networking basics, and an introduction to cloud computing. These are the bedrock skills every DevOps engineer builds on.

4 Weeks Linux · Git · Networking · Cloud Intro Beginner Level

What You'll Learn

  • Linux command line basics
  • File permissions & processes
  • Git branching strategies
  • TCP/IP and DNS fundamentals

Tools Used

  • Ubuntu (WSL or VM)
  • Git + GitHub
  • curl, ping, nslookup
  • AWS Free Tier (signup)

Learning Approach

  • Use terminal daily
  • Practice each command
  • Build muscle memory
  • Screenshot every task

End Goal

  • Comfortable in Linux terminal
  • Git workflow fluency
  • Basic network knowledge
  • AWS account configured
Week 1 — Linux Command Line Fundamentals
W1

Linux Terminal — Navigate, Files, Permissions

bash · ls · cd · chmod · sudo · man

▼
Analogy: Linux is like knowing how to drive a manual (stick shift) car. Most servers in the world run Linux, and knowing the terminal is the "manual gearbox" of DevOps — more powerful, more precise, and required for any serious infrastructure work.

Why Linux for DevOps?

  • 90%+ of cloud servers run Linux
  • All Docker containers are Linux-based
  • Kubernetes nodes run Linux
  • Free and open source
  • More control than GUI tools

Navigation Commands

  • pwd — print current directory
  • ls -la — list all files with details
  • cd /path — change directory
  • cd .. — go up one level
  • cd ~ — go to home directory

File Operations

  • mkdir foldername — create directory
  • touch file.txt — create empty file
  • cp source dest — copy file
  • mv old new — move or rename
  • rm file.txt — delete (permanent!)
  • cat file.txt — display file content

File Permissions

  • chmod 755 script.sh — rwxr-xr-x
  • r=4, w=2, x=1 (owner/group/others)
  • chmod +x file — make executable
  • sudo command — run as superuser
  • chown user:group file — change owner
# Common Linux commands in practice
pwd                         # /home/ubuntu
ls -la /etc                 # list /etc with permissions
mkdir -p projects/myapp     # create nested directories
cat /etc/os-release         # see Linux version
grep "error" /var/log/syslog  # search in log file
tail -f /var/log/nginx/access.log  # watch live log

# Process management
ps aux                      # list all processes
ps aux | grep nginx         # filter for nginx
kill 1234                   # kill process by PID
top                         # live process monitor
htop                        # better version of top

Week 1 Key Takeaways

  • The terminal is not scary — it's just a more powerful way to work with files
  • sudo = administrator mode; never run as root by default on production servers
  • man commandname shows the manual for any command — use it freely
  • Practice: do all your file operations from terminal for the rest of this week, not GUI
W2

Git — Version Control & Branching

Commit history · branches · merge · pull requests

▼
Analogy: Git is like Google Docs' version history, but for code. Every save (commit) is a snapshot you can return to. Branches are like parallel "what if" versions that can later be merged together. Teams of 1000 engineers collaborate through Git without stepping on each other's code.

Core Git Concepts

  • Repository: folder tracked by Git
  • Commit: a saved snapshot of your code
  • Branch: an independent line of development
  • Remote: the copy on GitHub/GitLab
  • HEAD: the commit you're currently on

Git Workflow (Feature Branch)

  • git checkout -b feature/login
  • Make changes, write code
  • git add . && git commit -m "Add login"
  • git push origin feature/login
  • Open Pull Request on GitHub
  • Review → Merge → Delete branch

Commit Best Practices

  • Present tense: "Add feature" not "Added feature"
  • Short first line (<72 chars)
  • One commit per logical change
  • Never commit .env files or secrets
  • Commit often — small commits are safer

Resolving Merge Conflicts

  • Conflict = two branches changed the same line
  • Git marks conflicts with <<<<< and >>>>>
  • Edit the file to keep what you want
  • git add file.txt then git commit
  • Prevention: pull often, work on separate files

Week 2 Key Takeaways

  • Always branch from main — never commit directly to main on team projects
  • git pull before you start any new work — always start from latest code
  • Write meaningful commit messages — your future self will thank you
  • Never use git push --force on shared branches — it overwrites others' work
W3

Networking Fundamentals

TCP/IP · DNS · HTTP · Ports · Load balancing

▼

IP Addresses & Subnets

  • IP address: unique identifier for a device
  • IPv4: 192.168.1.1 (4 octets, 0–255)
  • IPv6: longer, for modern internet
  • Private: 192.168.x.x, 10.x.x.x (not internet)
  • Public: visible on the internet
  • Subnet mask: defines network size

DNS

  • DNS = Domain Name System
  • Translates domain names to IP addresses
  • google.com → 142.250.x.x
  • A record: domain → IPv4 address
  • CNAME: domain → another domain
  • nslookup google.com to test DNS

Common Ports

  • 22 — SSH (secure remote login)
  • 80 — HTTP (web, unencrypted)
  • 443 — HTTPS (web, encrypted)
  • 3306 — MySQL database
  • 5432 — PostgreSQL database
  • 6379 — Redis cache
  • 3000/8080 — common app dev ports

HTTP Basics

  • Stateless request-response protocol
  • Client sends request → server sends response
  • Headers carry metadata
  • HTTPS = HTTP + TLS encryption
  • Status codes: 2xx=success, 4xx=client error, 5xx=server error

Week 3 Key Takeaways

  • Everything on the internet is just computers sending packets over TCP/IP
  • Memorize common port numbers — you'll see them in firewalls, security groups, and config files
  • When a website loads slowly, DNS resolution is often the culprit — use dig to diagnose
  • HTTP = unencrypted. Never serve production apps on HTTP — always HTTPS
W4

Cloud Computing Introduction

IaaS · PaaS · SaaS · AWS core services · Cloud concepts

▼
Analogy: Cloud computing is like electricity from the grid. You don't build a power plant — you plug in and pay for what you use. Before the cloud, companies built their own data centers (their own power plants). Now they rent compute, storage, and networking on demand.

Service Models

  • IaaS (Infrastructure): raw VMs, storage, networking. You manage OS up. Examples: AWS EC2, GCP Compute Engine
  • PaaS (Platform): managed runtime. You manage code only. Examples: Heroku, Render, Railway
  • SaaS (Software): ready-made software. Examples: Gmail, Slack, GitHub

Key Cloud Concepts

  • Region: geographic location of data centers
  • Availability Zone (AZ): isolated data centers in a region
  • Elasticity: scale up/down on demand
  • Pay-as-you-go: only pay for what you use
  • Managed services: AWS handles maintenance

AWS Core Services (Beginner)

  • EC2: virtual server (compute)
  • S3: object storage (files, images)
  • RDS: managed relational database
  • IAM: user permissions management
  • VPC: private network in AWS

AWS Free Tier

  • EC2 t2.micro: 750 hours/month free
  • S3: 5 GB storage free
  • RDS: 750 hours/month free
  • Lambda: 1 million requests/month free
  • Free for 12 months from signup
  • Set billing alerts immediately!

Week 4 Key Takeaways

  • Create your AWS account immediately and set a billing alarm at $5 — unexpected charges are common for beginners
  • Never commit AWS access keys to GitHub — use IAM roles and environment variables
  • The Big 3 cloud providers are AWS (largest), Azure (Microsoft), GCP (Google) — AWS is most in-demand in India
  • Start with the simplest services (S3 for storage, EC2 for compute) before advanced ones
