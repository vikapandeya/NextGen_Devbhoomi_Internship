Project Briefs — DevOps & Cloud Engineering

Three capstone projects demonstrating container deployment, CI/CD automation, and cloud infrastructure management. Each builds on the last — complete them in order.

3 Projects Beginner · Intermediate · Advanced Portfolio Projects
Project 1 — Beginner Level

Dockerized Portfolio Deployment

Docker · Nginx · AWS EC2 · Security Groups

Beginner · Due End Week 4
Stack
Docker · Nginx · EC2
Deadline
End of Week 4
Pass Mark
60 / 100
Submission
GitHub Repo + EC2 URL

Project Description

Take your existing portfolio website and containerize it using Nginx as a web server. Deploy it both locally using Docker and on AWS EC2. The goal is to demonstrate that the same container image works identically in both environments — this is the core promise of Docker.

Requirements

  • Write a production-ready Dockerfile using nginx:alpine as the base image
  • Include a .dockerignore file excluding dev files
  • Container must serve the website on port 80
  • docker run command documented in README with explanation of each flag
  • Deploy to AWS EC2 t2.micro (Free Tier) in ap-south-1
  • Website accessible via the EC2 public IP address
  • Security Group restricts SSH (port 22) to your IP only — not 0.0.0.0/0

Evaluation Rubric

CriteriaPointsWhat We Look For
Dockerfile quality20nginx:alpine base, COPY is clean, no unnecessary layers
Security Group configuration15Port 22 restricted to your IP, port 80 open to all
EC2 deployment20Container running on EC2, website loads via public IP
Documentation20README explains Docker commands and EC2 setup steps clearly
Running container proof25Screenshot of docker ps showing running container + browser screenshot
Total100Pass mark: 60
Tip: Build the image locally first with docker build -t portfolio:v1 . and test it before touching EC2. The most common mistake is copying the build step to EC2 before verifying it works locally.
Project 2 — Intermediate Level

Full CI/CD Pipeline for a Microservice

Node.js · Docker · GitHub Actions · AWS ECR · EC2

Intermediate · Due End Week 8

End of Week 8

GitHub Repo + Live API URL

Project Description

Build a REST API microservice with a health check and two data endpoints, containerize it, push to AWS Elastic Container Registry (ECR) via a GitHub Actions CI/CD pipeline, and configure the pipeline to auto-deploy to your EC2 instance on every push to the main branch.

Requirements

  • Express API with /health and /api/data endpoints responding with JSON
  • Unit tests with Jest — minimum 60% code coverage
  • Dockerfile using node:20-alpine + .dockerignore
  • GitHub Actions workflow: test → docker build → push to ECR → SSH deploy to EC2
  • All secrets (AWS keys, SSH key) stored in GitHub Secrets — never hardcoded
  • Deployed API accessible via EC2 public URL (http://<ip>:3000/health)
  • README includes architecture diagram (ASCII is acceptable) explaining the full pipeline

Evaluation Rubric

CriteriaPointsWhat We Look For
API + Tests20Both endpoints work, tests pass, coverage ≥60%
Dockerfile quality15Alpine base, .dockerignore present, no secrets in image
CI/CD pipeline30All 4 stages work: test → build → push to ECR → deploy to EC2
ECR + EC2 deployment25Image in ECR, container running on EC2, API responds live
Documentation10Architecture diagram, setup instructions, pipeline explanation
Total100Pass mark: 65
Security rule: If your GitHub Actions YAML contains an AWS access key or SSH private key as plain text, the project will be failed regardless of other scores. Use ${{ secrets.SECRET_NAME }} syntax exclusively.
Project 3 — Advanced Level

Production-Grade Kubernetes Infrastructure

Terraform · Kubernetes (EKS / minikube) · Prometheus · Grafana · GitHub Actions

End of Week 12

Project Description

Provision an EKS cluster with Terraform (or minikube for local development), deploy a multi-service application (frontend + backend + database), set up monitoring with Prometheus and Grafana, and implement a complete CI/CD pipeline that deploys to Kubernetes on every merge to main.

Requirements

  • Terraform: VPC + EKS cluster provisioned as code (or minikube as documented alternative)
  • Kubernetes: Deployment + Service + ConfigMap + Ingress for each service
  • Horizontal Pod Autoscaler configured and demonstrated with load test
  • Prometheus scraping application metrics via /metrics endpoint
  • Grafana dashboard showing live request rate, error rate, and pod count
  • GitHub Actions deploys to Kubernetes via kubectl apply on push to main
  • All secrets in Kubernetes Secrets or AWS Secrets Manager — zero plain text secrets in code
  • Runbook in README: how to deploy, rollback, and scale the application

Evaluation Rubric

CriteriaPointsWhat We Look For
Terraform IaC20VPC + cluster as code, variables used, state committed
Kubernetes manifests25All resource types present, HPA configured, health probes set
Monitoring setup20Prometheus collecting metrics, Grafana showing real traffic data
CI/CD pipeline25Automated deploy to K8s on push to main with rollback tested
Documentation + runbook10Architecture diagram, operational runbook, decision rationale
Total100Pass mark: 70
Career note: This project is the one that gets you interviews. A GitHub repo with a working EKS deployment, Grafana dashboards, and a CI/CD badge on the README tells a hiring manager more than any certification. Document every architecture decision you made and why.
