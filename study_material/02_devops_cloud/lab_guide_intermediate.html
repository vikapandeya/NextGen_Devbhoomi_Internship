Lab 5 — Dockerize a Node.js App
LAB 05 · WEEK 5
Goal: Take a Node.js Express app, write a production-ready Dockerfile, and run it locally alongside a Redis cache using docker-compose.
1

Create the Express Application

~20 min

Write a minimal Express server with a health check endpoint. This is the application you'll containerize.

# server.js
const express = require('express');
const app = express();
const PORT = process.env.PORT || 3000;

app.get('/health', (req, res) => {
  res.json({ status: 'ok', time: new Date(), version: '1.0' });
});

app.get('/', (req, res) => {
  res.json({ message: 'Devbhoomi App running in Docker!' });
});

app.listen(PORT, () => console.log(`Server on port ${PORT}`));

# package.json — add start script
npm init -y
npm install express
# In package.json: "start": "node server.js"
2

Write the Dockerfile

~20 min

Write a production-quality Dockerfile. Every line has a reason — understand each one before moving on.

FROM node:20-alpine
# alpine = small image (~50MB vs ~900MB for full node)

WORKDIR /app
# separate COPY for package.json enables Docker layer caching
COPY package*.json ./
RUN npm ci --production
# ci installs exact versions from package-lock.json

COPY . .
# Don't run as root in containers
USER node

EXPOSE 3000
CMD ["node", "server.js"]
3

Create .dockerignore

~5 min

Prevent unnecessary files from being copied into the image. This keeps images small and secure.

# .dockerignore
node_modules
.env
*.md
.git
.gitignore
npm-debug.log
4

Build and Run the Container

~20 min 
# Build the image
docker build -t myapp:v1 .
docker images                # confirm it appears in list

# Run the container in detached mode
docker run -d -p 3000:3000 --name myapp myapp:v1

# Test the health endpoint
curl http://localhost:3000/health

# Check logs
docker logs myapp
docker ps                    # should show container as running
5

Multi-Container with docker-compose

~30 min

Add Redis as a second service. The two containers communicate by service name inside Docker's network.

# docker-compose.yml
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      REDIS_URL: redis://redis:6379
    depends_on:
      - redis
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

# Start everything
docker-compose up -d
docker-compose ps            # both services should show "Up"
docker-compose logs -f app   # follow app logs
docker-compose down          # stop and remove containers
Deliverable: Screenshot of curl http://localhost:3000/health showing the JSON response + screenshot of docker ps showing both app and redis containers running.
Lab 6 — GitHub Actions CI/CD Pipeline
LAB 06 · WEEK 6
Goal: Build an automated CI pipeline that tests your code, builds the Docker image, and confirms everything passes before any merge is allowed.
1

Push Docker App to GitHub

~15 min 
git init
git add .
git commit -m "Initial Express app with Docker setup"
git remote add origin https://github.com/YOUR_USERNAME/devbhoomi-docker-app.git
git push -u origin main
2

Create the CI Workflow

~30 min

Create the workflows directory and write the CI pipeline. It runs on every push to any branch.

# .github/workflows/ci.yml
name: CI Pipeline
on:
  push:
    branches: ['**']
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test

  build:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - uses: actions/checkout@v4
      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .

  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npx eslint . --ext .js || true
Tip: The needs: test line means the build job only starts if test passes. Jobs without needs run in parallel.
3

Add a Failing Test to Verify the Pipeline

~20 min

Install Jest and add a test that intentionally fails. Push it and confirm the pipeline blocks the build.

npm install --save-dev jest supertest

# server.test.js — intentionally failing test
const request = require('supertest');
const app = require('./server');

test('health endpoint returns status ok', async () => {
  const res = await request(app).get('/health');
  expect(res.body.status).toBe('wrong-value'); // this will fail
});

# In package.json: "test": "jest"
git add . && git commit -m "Add failing test"
git push                     # watch pipeline fail in GitHub Actions tab
4

Fix the Test and Verify Pipeline Passes

~15 min 
# Fix server.test.js
expect(res.body.status).toBe('ok'); // correct value

git add . && git commit -m "Fix: correct expected value in health test"
git push

Go to the Actions tab on GitHub and watch all three jobs turn green.

5

Add GitHub Secrets for AWS

~10 min

These secrets will be used by the deploy job in a later lab. Add them now so the pipeline is ready.

Go to: Repository → Settings → Secrets and variables → Actions → New repository secret.

Add: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION (value: ap-south-1), EC2_HOST (leave empty for now).

Deliverable: Link to a GitHub Actions run showing all three jobs (test, build, lint) with green checkmarks. Also screenshot the run that failed when you pushed the broken test.
Lab 7 — Deploy to AWS EC2
LAB 07 · WEEK 7
Goal: Launch an EC2 instance, SSH into it, install Docker, run your container, and confirm the app is reachable from the public internet.
1

Launch EC2 Instance

~20 min

In the AWS Console: EC2 → Launch Instance. Configure as below — every setting matters for this lab.

  • Name: devbhoomi-app-server
  • AMI: Ubuntu Server 22.04 LTS (Free Tier)
  • Instance type: t2.micro (Free Tier)
  • Key pair: Create new → name it devbhoomi-key → download the .pem file
  • Security Group: Allow SSH (22) from My IP only, HTTP (80) from anywhere (0.0.0.0/0)
Warning: Move the downloaded .pem key file to a safe location. chmod 400 devbhoomi-key.pem immediately — SSH will refuse to use keys that are too open.
2

SSH into the Instance

~10 min 
chmod 400 ~/Downloads/devbhoomi-key.pem
ssh -i ~/Downloads/devbhoomi-key.pem ubuntu@<EC2-PUBLIC-IP>
# Find the public IP in EC2 Console → Instances → your instance
# Accept the fingerprint prompt with "yes" the first time
3

Install Docker on EC2

~15 min

Run these commands inside your EC2 SSH session. The last command adds your user to the docker group so you don't need sudo each time.

sudo apt update
sudo apt install -y docker.io
sudo systemctl enable docker
sudo systemctl start docker
sudo usermod -aG docker ubuntu
# Log out and back in for group change to take effect
exit
ssh -i ~/Downloads/devbhoomi-key.pem ubuntu@<EC2-PUBLIC-IP>
docker --version             # confirm it works without sudo
4

Run Your Docker Image on EC2

~20 min

Push your image to Docker Hub first, then pull and run it on EC2. Port 80 on EC2 maps to port 3000 inside the container.

# On your local machine — push image to Docker Hub
docker login
docker tag myapp:v1 YOUR_DOCKERHUB_USERNAME/myapp:v1
docker push YOUR_DOCKERHUB_USERNAME/myapp:v1

# On EC2 — pull and run
docker pull YOUR_DOCKERHUB_USERNAME/myapp:v1
docker run -d -p 80:3000 --name myapp YOUR_DOCKERHUB_USERNAME/myapp:v1
docker ps                    # confirm it's running
5

Verify the App is Live

~10 min

Open your browser and navigate to your EC2 instance's public IP. Your containerized app is now running on cloud infrastructure.

http://<EC2-PUBLIC-IP>/health
# Should return: {"status":"ok","time":"...","version":"1.0"}

# Also test from your local terminal
curl http://<EC2-PUBLIC-IP>/health
Tip: Stop your EC2 instance in the AWS Console when not in use — stopped instances don't incur compute charges (you only pay for the attached EBS storage).
Deliverable: Screenshot of your browser showing the /health response from your EC2 public IP address. The IP in the browser bar must be visible in the screenshot.
Lab 8 — Terraform — Provision AWS Infrastructure
LAB 08 · WEEK 8
Goal: Write Terraform code to provision an S3 bucket and EC2 instance. Practice the init/plan/apply/destroy workflow and understand infrastructure as code.
1

Install Terraform

~15 min 
# On Ubuntu/WSL
wget -O- https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor \
  -o /usr/share/keyrings/hashicorp-archive-keyring.gpg

echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \
  https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/hashicorp.list

sudo apt update && sudo apt install terraform
terraform --version          # should show Terraform v1.x.x
2

Write main.tf

~25 min

Create a new directory called terraform-lab and write your infrastructure files.

# main.tf
terraform {
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }
}

provider "aws" {
  region = var.region
}

resource "aws_s3_bucket" "lab_bucket" {
  bucket = var.bucket_name
  tags = {
    Project     = "devbhoomi-internship"
    Environment = var.environment
    Batch       = "2026-27"
  }
}

resource "aws_s3_bucket_versioning" "lab_bucket_versioning" {
  bucket = aws_s3_bucket.lab_bucket.id
  versioning_configuration {
    status = "Enabled"
  }
}
3

Create variables.tf and outputs.tf

~15 min 
# variables.tf
variable "region" {
  description = "AWS region"
  default     = "ap-south-1"
}

variable "bucket_name" {
  description = "Globally unique S3 bucket name — include your name"
}

variable "environment" {
  description = "dev / staging / prod"
  default     = "dev"
}

# outputs.tf
output "bucket_arn" {
  value = aws_s3_bucket.lab_bucket.arn
}

output "bucket_name" {
  value = aws_s3_bucket.lab_bucket.bucket
}
4

Run the Terraform Workflow

~20 min 
terraform init
# Downloads the AWS provider plugin — run once per new project

terraform plan -var="bucket_name=YOUR_NAME-devbhoomi-tf-lab"
# Read the output carefully — it shows exactly what will be created
# + indicates a new resource, ~ indicates a change, - indicates deletion

terraform apply -var="bucket_name=YOUR_NAME-devbhoomi-tf-lab"
# Type "yes" when prompted
# Wait for the "Apply complete!" message

# Verify bucket appears in AWS Console (S3 → Buckets)
aws s3 ls | grep devbhoomi
5

Destroy and Clean Up

~10 min

Always destroy lab resources when you're done. S3 buckets themselves are free, but data stored in them is billed.

terraform destroy -var="bucket_name=YOUR_NAME-devbhoomi-tf-lab"
# Type "yes" to confirm
# Terraform will delete every resource it created

# Verify it's gone in AWS Console
# Note: terraform.tfstate still exists locally — this is expected
Warning: Never run terraform destroy on a production environment without a formal approval process. Add a confirmation gate to your pipeline for any destructive Terraform operations.
Deliverable: Screenshot of the terminal showing the terraform apply output with "Apply complete! Resources: 2 added" + screenshot of the AWS Console showing the S3 bucket before you destroyed it.
