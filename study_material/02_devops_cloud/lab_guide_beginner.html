Beginner Lab Guide — DevOps & Cloud

4 hands-on labs: set up your Linux environment, practice Git branching, explore networking tools, and create your first AWS resources.

4 Labs Linux · Git · Networking · AWS Beginner Level

Before You Start

  • Windows 10 or 11 (64-bit)
  • 8 GB RAM minimum
  • Stable internet connection
  • GitHub account created

What You'll Build

  • Functional Linux environment
  • First GitHub Pull Request
  • Network diagnostic report
  • First S3 bucket on AWS

Time Estimate

  • Lab 1: ~2 hours
  • Lab 2: ~1.5 hours
  • Lab 3: ~1 hour
  • Lab 4: ~2 hours

Submission

  • GitHub links where required
  • Submit to mentor portal
  • Deadline: end of Week 4
Lab 1 — Linux Environment Setup & Practice
LAB 01 · WEEK 1
Goal: Install WSL2 (Windows Subsystem for Linux), navigate the filesystem, write a shell script, and practice file permissions.
1

Install WSL2 on Windows

~20 min

Open PowerShell as Administrator (right-click Start → Windows Terminal (Admin)). Run the install command and restart when prompted.

wsl --install
# Restart your computer when prompted
# On first launch, create a Linux username and password
# Verify it worked:
wsl --list --verbose
Tip: Your WSL username and Windows username can be different. Use a simple lowercase name with no spaces.
2

Practice Navigation

~15 min

Open your Ubuntu terminal and run these commands one by one. Understand what each returns before moving on.

pwd                         # show current location
ls -la                      # list files with permissions
mkdir projects              # create a directory
cd projects                 # move into it
touch hello.sh              # create an empty file
ls -la                      # confirm hello.sh appears
3

Write and Run a Shell Script

~20 min

Write a shell script that greets you, then make it executable and run it.

echo '#!/bin/bash' > hello.sh
echo 'echo "Hello from Linux, Devbhoomi Intern!"' >> hello.sh
echo 'echo "Today is: $(date)"' >> hello.sh
chmod +x hello.sh
./hello.sh
4

File Permissions Exercise

~20 min

Create 3 files with different permission levels. Use ls -la to verify and understand what each set of permissions means in practice.

touch file_public.sh file_private.sh file_readonly.sh
chmod 777 file_public.sh     # rwxrwxrwx — everyone can do everything
chmod 644 file_private.sh    # rw-r--r-- — owner edits, others read only
chmod 400 file_readonly.sh   # r-------- — only owner can read
ls -la *.sh                  # verify all three look correct
Warning: Never set 777 permissions on production files. It means any user on the system can modify or delete them.
5

Process Management

~15 min

Explore running processes on your Linux system. Identify the top CPU consumers and practice stopping a process safely.

top                          # live process monitor — press q to quit
ps aux                       # list all running processes
ps aux | grep bash           # filter for bash processes

# Find and kill a process by name (safe example)
sleep 60 &                   # start a background process
ps aux | grep sleep          # find its PID
kill <PID>                   # terminate it
Deliverable: Screenshot of your terminal showing (1) ls -la output with the three files and their different permissions, and (2) the output of running ./hello.sh. Label both screenshots clearly.
Lab 2 — Git Feature Branch Workflow
LAB 02 · WEEK 2
Goal: Create a real GitHub repository, work on a feature branch, open a Pull Request, and resolve a merge conflict manually.
1

Create Repository and Clone

~15 min

Create a new public repository called devops-practice on GitHub, then clone it to your local machine.

git clone https://github.com/YOUR_USERNAME/devops-practice.git
cd devops-practice
git status                   # confirm you're on main branch
2

Create Feature Branch and Write README

~30 min

Create a feature branch and write a professional README with real content for this practice repository.

git checkout -b feature/add-readme

# Write README.md with these sections:
# - Project Description
# - Prerequisites
# - Installation
# - Usage

git add README.md
git commit -m "Add professional README with setup instructions"
git push origin feature/add-readme

Go to GitHub and open a Pull Request from feature/add-readme into main. Write a clear PR description.

3

Simulate and Resolve a Merge Conflict

~30 min

This is the most important skill in Git. Create two branches that edit the same line, then merge and resolve the conflict.

git checkout main
git checkout -b feature/branch-a
echo "Line from Branch A" > conflict.txt
git add conflict.txt && git commit -m "Add conflict.txt from branch A"

git checkout main
git checkout -b feature/branch-b
echo "Different line from Branch B" > conflict.txt
git add conflict.txt && git commit -m "Add conflict.txt from branch B"

# Now merge branch-a into main, then try to merge branch-b
git checkout main
git merge feature/branch-a
git merge feature/branch-b   # this will create a conflict

# Edit conflict.txt to resolve: remove the <<<< ==== >>>> markers
# Keep whichever content (or both) makes sense
git add conflict.txt
git commit -m "Resolve merge conflict between branch-a and branch-b"
4

Set Up .gitignore

~15 min

Create a proper .gitignore file for a Node.js project. These file types should never be committed to a repository.

# .gitignore for Node.js projects
node_modules/
.env
.env.local
*.log
dist/
.DS_Store
.vscode/
coverage/
Warning: If you accidentally commit .env or node_modules, remove them from history with git rm --cached immediately — do not just add them to .gitignore after the fact.
Deliverable: Link to your GitHub Pull Request (feature/add-readme → main) + a screenshot of your terminal showing the merge conflict markers and your resolved file.
Lab 3 — Networking Diagnostics
LAB 03 · WEEK 3
Goal: Use standard Linux networking tools to diagnose connectivity, perform DNS lookups, and trace the path packets take across the internet.
1

Ping Tests — Latency and Connectivity

~15 min

Test connectivity to different endpoints and interpret the latency values you see.

ping -c 4 google.com         # test DNS + connectivity (4 packets)
ping -c 4 8.8.8.8            # test IP connectivity (Google's DNS)
ping -c 4 1.1.1.1            # Cloudflare DNS — should be very fast

# Record the average latency for each and note differences
# Under 50ms = excellent, 50-100ms = good, 200ms+ = slow
2

DNS Lookup

~20 min

Query DNS records directly and understand what an A record, CNAME, and MX record are.

nslookup google.com          # basic DNS lookup
nslookup amazon.com          # multiple A records — load balancing

# Install dig if needed: sudo apt install dnsutils
dig google.com               # detailed DNS query
dig google.com MX            # mail exchange records
dig amazon.com +short        # just the IP addresses
3

Port and Header Inspection

~20 min

Check if ports are open and inspect the HTTP headers that web servers return.

curl -I https://google.com   # get HTTP response headers only
curl -I https://github.com   # note the server and security headers

# Test if a specific port is open
nc -zv google.com 443        # port 443 (HTTPS) — should succeed
nc -zv google.com 22         # port 22 (SSH) — should fail for google.com

# Look for these response headers:
# Strict-Transport-Security — forces HTTPS
# X-Content-Type-Options — prevents MIME sniffing
4

Traceroute — Map the Internet Path

~15 min

Trace every router hop between your machine and a destination. This is how engineers diagnose where latency comes from.

# On Linux (WSL):
traceroute google.com        # install: sudo apt install traceroute

# On Windows CMD/PowerShell:
tracert google.com

# Count the number of hops (routers) between you and Google
# Note any hop with high latency — that's where delay is added
# * * * means the router doesn't respond to probes (normal)
5

Check Your IP Addresses

~10 min

Discover your public and private IP addresses and understand the difference.

ip addr show                 # all network interfaces + private IP
curl ifconfig.me             # your public IP (what servers see)
curl ifconfig.me/all         # detailed info including ISP

# Compare: private IP (192.168.x.x or 10.x.x.x) vs public IP
# They differ because of NAT (Network Address Translation)
Deliverable: Screenshot of your dig google.com DNS output showing the A record IP addresses + screenshot of traceroute google.com with the total hop count noted in writing.
Lab 4 — AWS Account Setup + First S3 Bucket
LAB 04 · WEEK 4
Goal: Create and secure your AWS account, set a billing alarm, configure the AWS CLI, and create your first S3 bucket programmatically.
1

Create AWS Account and Enable MFA

~30 min

Sign up at aws.amazon.com. Use a personal email address — not your company or college email. The first thing to do after signup is secure your root account.

After signup: go to IAM → Security credentials → Multi-factor authentication → Assign MFA device. Use Google Authenticator or Authy on your phone.

Warning: Your AWS root account is the highest privilege account. If it is compromised, someone can delete everything and run up a massive bill. Enable MFA before doing anything else.
2

Create a Billing Alarm

~15 min

Set a $5 monthly billing alert. This prevents surprise charges from forgotten resources.

AWS Console → search "Budgets" → Create Budget → Cost Budget → $5.00 → your email for alerts → Create.

Tip: Set the threshold at 80% ($4.00) so you're warned before you actually hit $5, not after.
3

Create IAM User with Programmatic Access

~20 min

Never use your root account for daily work. Create a dedicated IAM user for your CLI and code access.

IAM → Users → Create User → name: devbhoomi-cli → Attach policy: AdministratorAccess → Create → Security credentials tab → Create access key → CLI use case → Download the CSV file.

Warning: Download the CSV immediately — AWS will never show you the secret key again. Store it somewhere safe and never commit it to GitHub.
4

Install AWS CLI and Configure

~20 min 
# Install AWS CLI on Ubuntu/WSL
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install
aws --version                # should show aws-cli/2.x.x

# Configure with your IAM user credentials
aws configure
# Enter: AWS Access Key ID (from CSV)
# Enter: AWS Secret Access Key (from CSV)
# Enter: Default region: ap-south-1 (Mumbai — closest to India)
# Enter: Default output format: json

# Verify it works
aws sts get-caller-identity  # should show your account ID and username
5

Create S3 Bucket and Upload a File

~20 min

S3 bucket names are globally unique across all of AWS. Use your name in the bucket name to ensure uniqueness.

# Create the bucket (replace YOUR_NAME with your actual name)
aws s3 mb s3://YOUR_NAME-devbhoomi-test --region ap-south-1

# Upload the hello.sh script from Lab 1
aws s3 cp ~/projects/hello.sh s3://YOUR_NAME-devbhoomi-test/

# Verify the file is there
aws s3 ls s3://YOUR_NAME-devbhoomi-test/

# List all your buckets
aws s3 ls

# Clean up (free tier limits apply)
# aws s3 rm s3://YOUR_NAME-devbhoomi-test/hello.sh
# aws s3 rb s3://YOUR_NAME-devbhoomi-test
Deliverable: Screenshot of the AWS Console showing your S3 bucket in the ap-south-1 region + screenshot of your terminal showing the aws s3 ls output with the uploaded file listed.
