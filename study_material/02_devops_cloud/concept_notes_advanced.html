Week 9 — Kubernetes Fundamentals
W9

Kubernetes — Pods · Deployments · Services · ConfigMaps

Orchestrate containers at scale

▼
Analogy: If Docker is a shipping container, Kubernetes is the entire shipping port — it decides where each container goes, ensures containers are always running, scales up when traffic spikes, and automatically replaces crashed containers. The port manager never sleeps.

K8s Architecture

  • Control Plane: API server, scheduler, etcd
  • Worker Nodes: kubelet, kube-proxy, container runtime
  • etcd: distributed key-value store — cluster state
  • API server: all kubectl commands go here first
  • Scheduler: decides which node runs each pod

Core Objects

  • Pod: smallest deployable unit — 1+ containers
  • Deployment: manages replica count + rollouts
  • Service: stable network endpoint for pods
  • Namespace: logical isolation within cluster
  • ConfigMap/Secret: config and sensitive data

kubectl Commands

  • kubectl apply -f manifest.yaml
  • kubectl get pods -n namespace
  • kubectl describe pod pod-name
  • kubectl logs pod-name -f
  • kubectl exec -it pod -- /bin/sh
  • kubectl delete deployment name

Ingress

  • Routes external HTTP traffic to services
  • One load balancer for many services
  • Path-based routing: /api → api-svc
  • Host-based: app.example.com → frontend
  • TLS termination at the ingress layer
# deployment.yaml — Node.js app
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:v1.2
          ports:
            - containerPort: 3000
          resources:
            requests:
              memory: "128Mi"
              cpu: "250m"
            limits:
              memory: "256Mi"
              cpu: "500m"
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5

# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc
spec:
  selector:
    app: myapp
  ports:
    - port: 80
      targetPort: 3000
  type: ClusterIP

Week 9 Key Takeaways

  • Never run bare Pods in production — always use Deployments so K8s can restart failed pods
  • Services expose Pods to the network — without a Service, a Pod is completely unreachable
  • Use namespaces to separate dev/staging/prod in the same cluster — prevents accidental changes
  • Set resource requests and limits on every container — without them, one app can starve all others
Week 10 — Monitoring & Observability
W10

Prometheus · Grafana · Logs · Alerts · SLOs

See everything happening in production

▼

Three Pillars of Observability

  • Metrics: numbers over time (CPU, req/sec, error rate)
  • Logs: text events with timestamps and context
  • Traces: the full journey of a single request
  • All three together give a complete picture
  • Missing any one = blind spot in production

Prometheus

  • Pull-based: scrapes /metrics endpoints
  • PromQL: powerful query language for metrics
  • Scrape configs: define what to monitor
  • Exporters: node-exporter, mysqld-exporter
  • AlertManager: send alerts to Slack/PagerDuty

Grafana

  • Dashboards: visualize Prometheus metrics
  • Panels: graphs, gauges, tables, alerts
  • Alert rules: notify when thresholds crossed
  • Data sources: Prometheus, Elasticsearch, CloudWatch
  • Import community dashboards by ID

SLI / SLO / SLA

  • SLI: measurement (e.g. % requests < 200ms)
  • SLO: target (e.g. 99.9% uptime per month)
  • SLA: legal agreement with customers
  • Error budget: 100% − SLO (0.1% = 8.7 hrs/year)
  • Track error budget burn rate daily
# prometheus.yml — scrape config
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: 'myapp'
    static_configs:
      - targets: ['app:3000']
    metrics_path: '/metrics'

  - job_name: 'node-exporter'
    static_configs:
      - targets: ['node-exporter:9100']

# Useful PromQL queries
# Request rate per second
rate(http_requests_total[5m])

# Error rate percentage
rate(http_requests_total{status=~"5.."}[5m])
  / rate(http_requests_total[5m]) * 100

# P95 response time
histogram_quantile(0.95,
  rate(http_request_duration_seconds_bucket[5m])
)

Week 10 Key Takeaways

  • You can't fix what you can't see — monitoring is non-negotiable before any production launch
  • Set alerts before you need them, not after an incident has already hurt users
  • Error budget = 100% − SLO: 99.9% SLO means only 8.7 hours downtime allowed per year
  • The 4 Golden Signals: Latency, Traffic, Errors, Saturation — monitor these for every service
Week 11 — DevSecOps
W11

Security in the Pipeline — Shift Left DevSecOps

Find vulnerabilities before they reach production

▼

Shift Left Security

  • Find vulnerabilities early in development, not production
  • Cost: $1 to fix in dev = $10 in QA = $100 in prod
  • SAST: Static scan of source code (no runtime needed)
  • DAST: Dynamic test of a running application
  • SCA: Software Composition Analysis — scan dependencies

Container Security

  • Scan images with Trivy before deployment
  • Use non-root user — USER node in Dockerfile
  • Read-only root filesystem where possible
  • Use distroless or scratch base images
  • Never store credentials in Dockerfile ENV

Secrets Management

  • AWS Secrets Manager: store and rotate secrets
  • HashiCorp Vault: open-source secrets engine
  • Never commit secrets to Git — use .gitignore
  • Rotate all credentials on a schedule
  • Audit secret access — who accessed what, when

K8s Network Policies

  • Default: all pods can talk to all pods
  • Network policies restrict pod-to-pod traffic
  • Allow only: frontend → backend, backend → db
  • Deny all by default, allow explicitly
  • Use Calico or Cilium as network policy engine
# Trivy — scan Docker image before push
trivy image --severity HIGH,CRITICAL myapp:v1.2
trivy image --exit-code 1 --severity CRITICAL myapp:v1.2

# GitHub Actions — security scan job
  security-scan:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Scan image with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp:${{ github.sha }}
          severity: HIGH,CRITICAL
          exit-code: 1

      - name: Check for secrets with Gitleaks
        uses: gitleaks/gitleaks-action@v2

      - name: npm audit
        run: npm audit --audit-level=high

# Retrieve secret from HashiCorp Vault
vault kv get -field=db_password secret/myapp/prod

Week 11 Key Takeaways

  • Security is everyone's job — not just the security team's. DevSecOps means developers own security too
  • Scan container images before every deployment — a vulnerable base image is a production incident waiting
  • Rotate all credentials regularly — treat compromised credentials as a known risk, not a hypothetical one
  • Principle of least privilege everywhere — IAM, K8s RBAC, database users. Grant exactly what's needed, nothing more
Week 12 — SRE Principles & Production Reliability
W12

SRE — Error Budgets · Incident Management · Chaos Engineering

Keep systems reliable at scale

▼

SRE Philosophy

  • Reliability is a feature — users don't notice uptime, only downtime
  • Error budgets: quantify acceptable unreliability
  • Toil reduction: automate repetitive manual work
  • Blameless postmortems: find system failures, not human failures
  • Google's SRE book: required reading for this domain

Incident Management

  • On-call rotations: shared responsibility, fair hours
  • Runbooks: step-by-step playbooks for known incidents
  • Severity levels: SEV1 (all-hands) to SEV5 (no urgency)
  • War rooms: real-time incident coordination channel
  • Every SEV1/SEV2 requires a postmortem within 48h

Chaos Engineering

  • Intentionally break things to find weaknesses first
  • Netflix Chaos Monkey: randomly kills production servers
  • Start small: kill one pod, then a whole node
  • Game Days: scheduled chaos experiments with the team
  • Goal: prove the system recovers automatically

Disaster Recovery

  • RTO: Recovery Time Objective — how fast to recover
  • RPO: Recovery Point Objective — how much data loss is ok
  • Multi-region deployment for critical systems
  • Test your backups regularly — untested backups don't exist
  • Runbook for full environment rebuild from scratch

Week 12 Key Takeaways

  • Automate toil — if you perform the same manual task three times, write a script the third time
  • Every incident needs a blameless postmortem with concrete action items — blame fixes nothing
  • SRE targets 99.9–99.99% availability: 99.9% = 8.7 hours downtime/year; 99.99% = 52 minutes/year
  • Practice your disaster recovery before you need it — a DR plan you've never tested is not a DR plan
