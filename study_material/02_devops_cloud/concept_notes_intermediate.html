Week 5 — Docker Fundamentals
W5

Docker — Containers · Images · Dockerfile · docker-compose

Containerize your first application

▼
Analogy: Docker containers are like shipping containers — standardized boxes that work the same on any ship (server). Before containers, "works on my machine" was a real problem. With Docker, your app and all its dependencies are packed together so it runs identically everywhere: your laptop, the CI server, and production.

Why Docker?

  • Consistency across all environments
  • Isolation — apps don't interfere with each other
  • Lightweight compared to full VMs
  • Microservices-ready by design
  • 93% of companies now use containers

Dockerfile Instructions

  • FROM — base image (e.g. node:20-alpine)
  • WORKDIR — set working directory
  • COPY — copy files into image
  • RUN — run command during build
  • EXPOSE — document port number
  • CMD — default command to run

Docker CLI Commands

  • docker build -t name:tag .
  • docker run -d -p 3000:3000 name
  • docker ps — list running containers
  • docker stop name — stop a container
  • docker rm name — remove a container
  • docker images — list all images

docker-compose

  • Manages multi-container applications
  • Define services in one YAML file
  • Volumes: persist data between restarts
  • Networks: containers talk by service name
  • depends_on: control startup order
  • docker-compose up -d to start
# Dockerfile for a Node.js Express app
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

# docker-compose.yml — app + MySQL together
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DB_HOST: db
      DB_USER: root
      DB_PASS: secret
    depends_on:
      - db
  db:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: myapp
    volumes:
      - db-data:/var/lib/mysql
volumes:
  db-data:

Week 5 Key Takeaways

  • Always use .dockerignore — exclude node_modules and .env to keep images small
  • Use alpine images (node:20-alpine) — they are 5× smaller than full images
  • Never run containers as root — add USER node before CMD in your Dockerfile
  • Use docker-compose for local dev with multiple services — it saves hours of manual wiring
Week 6 — CI/CD with GitHub Actions
W6

CI/CD — Pipeline · Triggers · Jobs · Deploy

Automate every push: test, build, ship

▼
Analogy: CI/CD is like a factory assembly line. Every time code is pushed, an automatic conveyor belt runs: it tests the code, builds it, packages it, and deploys it — all without human intervention. Bad parts are rejected before they reach customers.

CI vs CD

  • CI: auto-test on every push to any branch
  • CD: auto-deploy when CI passes on main
  • CI catches bugs early — before code review
  • CD removes the manual deploy step entirely
  • Together: code is always deployable

GitHub Actions Concepts

  • Workflow: YAML file in .github/workflows/
  • on: — triggers (push, PR, schedule)
  • Jobs: independent units of work
  • Steps: sequential commands within a job
  • Secrets: encrypted env vars for credentials

Pipeline Stages

  • lint → test → build → docker build
  • → push image to registry (ECR)
  • → deploy to EC2 via SSH
  • Failed test = pipeline stops, no deploy
  • All stages visible in GitHub UI

AWS ECR (Container Registry)

  • Elastic Container Registry — store Docker images
  • Private registry inside your AWS account
  • EC2 can pull images without public access
  • Tag images: latest, v1.0, git-SHA
  • Enable image scanning for vulnerabilities
# .github/workflows/deploy.yml
name: Build and Deploy
on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-south-1

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build, tag, and push image to ECR
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          IMAGE_TAG: ${{ github.sha }}
        run: |
          docker build -t $ECR_REGISTRY/myapp:$IMAGE_TAG .
          docker push $ECR_REGISTRY/myapp:$IMAGE_TAG

      - name: Deploy to EC2
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.EC2_HOST }}
          username: ubuntu
          key: ${{ secrets.EC2_KEY }}
          script: |
            docker pull ${{ steps.login-ecr.outputs.registry }}/myapp:${{ github.sha }}
            docker stop myapp || true
            docker run -d --name myapp -p 80:3000 \
              ${{ steps.login-ecr.outputs.registry }}/myapp:${{ github.sha }}

Week 6 Key Takeaways

  • Store secrets in GitHub Secrets — never write credentials directly in YAML files
  • Use environment-specific workflows — staging deploys on every push, prod deploys on release tags
  • Always run tests before deploy — a failed test must block the pipeline
  • Failed deploys should auto-rollback — use docker stop || true + restart old container on failure
Week 7 — AWS Core Services
W7

AWS — EC2 · S3 · RDS · IAM · Security Groups

Build real cloud infrastructure on AWS

▼
Analogy: AWS is like a city you can build in minutes. EC2 = land to build on. S3 = warehouses for your goods. RDS = bank vaults for your data. IAM = the city's permits system — decides who can enter where. VPC = the city walls and districts that keep things separated.

EC2 — Compute

  • Instance types: t2.micro (free tier), t3.small (dev)
  • AMIs: Amazon Machine Images — pre-built OS
  • Key pairs: SSH access without passwords
  • Security Groups: virtual firewall for the instance
  • Elastic IP: static public IP address

S3 — Object Storage

  • Buckets hold any type of file at any scale
  • Bucket policies: control public/private access
  • Static website hosting from S3 directly
  • Lifecycle rules: archive/delete old files
  • Versioning: keep history of every object

RDS — Managed Database

  • Multi-AZ deployment for high availability
  • Automated daily backups, point-in-time restore
  • Parameter groups: tune database settings
  • Supports MySQL, PostgreSQL, Aurora
  • AWS handles patching and failover

IAM — Access Management

  • Least privilege: grant only what's needed
  • Roles (for services) vs Users (for humans)
  • Policies written as JSON documents
  • MFA: enable on root and admin accounts
  • Never use root account for daily work
# AWS CLI — create S3 bucket
aws s3 mb s3://devbhoomi-myapp-prod --region ap-south-1
aws s3 cp ./dist s3://devbhoomi-myapp-prod/ --recursive

# Launch EC2 instance
aws ec2 run-instances \
  --image-id ami-0f58b397bc5c1f2e8 \
  --instance-type t2.micro \
  --key-name my-keypair \
  --security-group-ids sg-0abc123 \
  --subnet-id subnet-0def456 \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=devbhoomi-app}]'

# Configure Security Group — allow SSH + HTTP
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc123 \
  --protocol tcp --port 22 --cidr 203.0.113.0/32

aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc123 \
  --protocol tcp --port 80 --cidr 0.0.0.0/0

Week 7 Key Takeaways

  • Security Groups are stateful firewalls — always restrict inbound traffic to only what's necessary
  • Keep EC2 instances in private subnets — only your load balancer needs a public IP
  • Use IAM roles not access keys for EC2-to-AWS communication — no credentials stored on the server
  • Enable MFA on your AWS root account immediately — this is the single most important security step
Week 8 — Terraform Introduction
W8

Terraform — IaC · Providers · Resources · State

Write code that builds cloud infrastructure

▼
Analogy: Terraform is like a recipe for your infrastructure. Instead of clicking through the AWS console (which you can't version control or repeat), you write code that says "I want 2 EC2 instances, 1 RDS, 1 S3 bucket" — and Terraform makes it happen, repeatedly and consistently, every single time.

Why Infrastructure as Code?

  • Reproducible: spin up identical environments
  • Version controlled: track every change in Git
  • No manual mistakes from clicking the console
  • Drift detection: know when reality differs from code
  • Review infra changes just like code changes

Terraform Workflow

  • terraform init — download provider plugins
  • terraform plan — preview what will change
  • terraform apply — create/update resources
  • terraform destroy — delete all resources
  • Always run plan before apply — review every change

HCL Syntax Basics

  • provider: configure AWS, GCP, etc.
  • resource: declare infrastructure to create
  • variable: parameterize your config
  • output: print values after apply
  • data: read existing AWS resources

State File

  • terraform.tfstate — tracks what exists
  • Never delete the state file manually
  • Store in S3 with DynamoDB locking for teams
  • Contains actual resource IDs from AWS
  • Can contain secrets — keep it private
# main.tf — create S3 bucket and EC2 instance
terraform {
  required_providers {
    aws = { source = "hashicorp/aws", version = "~> 5.0" }
  }
}

provider "aws" {
  region = var.region
}

resource "aws_s3_bucket" "app_bucket" {
  bucket = var.bucket_name
  tags = {
    Project     = "devbhoomi"
    Environment = var.environment
    Owner       = "batch-2026-27"
  }
}

resource "aws_instance" "app_server" {
  ami           = "ami-0f58b397bc5c1f2e8"
  instance_type = "t2.micro"
  key_name      = var.key_name
  tags = {
    Name        = "devbhoomi-app"
    Environment = var.environment
  }
}

# variables.tf
variable "region"      { default = "ap-south-1" }
variable "bucket_name" { description = "Globally unique S3 bucket name" }
variable "environment" { default = "dev" }
variable "key_name"    { description = "EC2 key pair name" }

# outputs.tf
output "bucket_arn"    { value = aws_s3_bucket.app_bucket.arn }
output "instance_ip"   { value = aws_instance.app_server.public_ip }

Week 8 Key Takeaways

  • Never delete terraform.tfstate manually — it is Terraform's source of truth for what exists
  • Run terraform plan before every apply — always review what will be created, changed, or destroyed
  • Store state in S3 with DynamoDB locking for team use — local state breaks in teams
  • Tag all resources with project/env/owner — makes cost tracking and cleanup much easier
