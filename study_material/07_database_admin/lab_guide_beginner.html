Lab 1

MySQL Setup & First Database (Week 1)

▼

Objective

Install MySQL, connect via Workbench, create your first database and table, insert data, and verify it visually.

Step-by-Step

  1. Install MySQL Community Server
    Windows: download MySQL Installer from dev.mysql.com/downloads/installer. Choose "Developer Default". During setup, set root password — note it down. The server runs on port 3306 by default. Also install MySQL Workbench (included in Developer Default).

    Linux (Ubuntu): sudo apt install mysql-server then sudo mysql_secure_installation.
  2. Connect via MySQL Workbench
    Open Workbench → click + next to "MySQL Connections" → Hostname: 127.0.0.1, Port: 3306, Username: root → Test Connection → enter password → OK.
  3. Explore default databases
    In the query window, run: 
    SHOW DATABASES;
    You will see: information_schema (metadata about all tables), mysql (user and privilege data), performance_schema (runtime performance data), sys (simplified performance views). Do not modify these.
  4. Create and use your database
    CREATE DATABASE it_university;
USE it_university;
SELECT DATABASE();  -- confirms current database
  5. Create the students table
    CREATE TABLE students (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  name        VARCHAR(100) NOT NULL,
  email       VARCHAR(150) NOT NULL UNIQUE,
  course      VARCHAR(50),
  phone       VARCHAR(15),
  enrolled_at DATE DEFAULT (CURRENT_DATE),
  is_active   BOOLEAN DEFAULT TRUE
);
  6. Insert 5 students
    INSERT INTO students (name, email, course) VALUES
  ('Priya Sharma',   'priya@example.com',   'Web Development'),
  ('Rahul Verma',    'rahul@example.com',   'Data Science'),
  ('Ankit Singh',    'ankit@example.com',   'Cyber Security'),
  ('Meena Kumari',   'meena@example.com',   'Web Development'),
  ('Vikram Patil',   'vikram@example.com',  'Database Admin');
  7. Query and verify
    SELECT * FROM students;
SELECT name, course FROM students WHERE course = 'Web Development';
SELECT COUNT(*) FROM students;
    In MySQL Workbench: click the table name in the left panel → right-click → "Select Rows" to see the data in a visual grid.

Deliverable Check

  • MySQL installed and accessible via Workbench
  • it_university database created
  • students table with 5 rows inserted
  • SELECT query returns all rows correctly
Lab 2

E-Commerce Schema Design (Weeks 2–3)

▼

Objective

Design and build a normalised e-commerce database schema from scratch, insert realistic sample data, and write queries that span multiple tables.

Step-by-Step

  1. Sketch on paper first
    Draw boxes for: customers, products, orders, order_items. Draw arrows showing relationships (customers 1→N orders, orders 1→N order_items, products 1→N order_items). Write the columns in each box.
  2. Create all four tables
    USE it_university;

CREATE TABLE customers (
  id         INT PRIMARY KEY AUTO_INCREMENT,
  name       VARCHAR(100) NOT NULL,
  email      VARCHAR(150) NOT NULL UNIQUE,
  phone      VARCHAR(15),
  city       VARCHAR(50),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE products (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  name        VARCHAR(100) NOT NULL,
  description TEXT,
  price       DECIMAL(10,2) NOT NULL CHECK (price >= 0),
  stock       INT NOT NULL DEFAULT 0 CHECK (stock >= 0),
  category    VARCHAR(50)
);

CREATE TABLE orders (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  customer_id INT NOT NULL,
  total       DECIMAL(10,2),
  status      ENUM('pending','shipped','delivered','cancelled') DEFAULT 'pending',
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE RESTRICT
);

CREATE TABLE order_items (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  order_id    INT NOT NULL,
  product_id  INT NOT NULL,
  quantity    INT NOT NULL CHECK (quantity > 0),
  unit_price  DECIMAL(10,2) NOT NULL,
  FOREIGN KEY (order_id)   REFERENCES orders(id)   ON DELETE CASCADE,
  FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE RESTRICT
);

  4. Write 5 analytical queries
    -- 1. All orders for customer 1 with customer name
SELECT c.name, o.id AS order_id, o.total, o.status, o.created_at
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE c.id = 1;

-- 2. Total revenue per customer (only customers with orders)
SELECT c.name, COUNT(o.id) AS order_count, SUM(o.total) AS total_spent
FROM customers c
JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name
ORDER BY total_spent DESC;

-- 3. Most popular product by quantity sold
SELECT p.name, SUM(oi.quantity) AS total_sold
FROM order_items oi
JOIN products p ON oi.product_id = p.id
GROUP BY p.id, p.name
ORDER BY total_sold DESC
LIMIT 1;

-- 4. Orders placed this month
SELECT * FROM orders
WHERE created_at >= DATE_FORMAT(NOW(), '%Y-%m-01');

-- 5. Average order value by status
SELECT status, COUNT(*) AS order_count, AVG(total) AS avg_order_value
FROM orders
GROUP BY status;
Lab 3

JOINs Practice (Week 3)

▼

Objective

Write 8 progressively complex JOIN queries on the e-commerce schema. Understand when to use INNER vs LEFT vs self JOIN.

8 JOIN Queries

-- 1. INNER JOIN: orders with customer names
SELECT o.id, c.name, o.total, o.status
FROM orders o
JOIN customers c ON o.customer_id = c.id;

-- 2. LEFT JOIN: ALL customers, even those with no orders
-- (NULL values appear for customers with no orders)
SELECT c.name, c.city, o.id AS order_id, o.total
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id;

-- 3. Find customers with NO orders
SELECT c.name, c.email
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.id IS NULL;

-- 4. Multi-table JOIN: full order receipt
SELECT
  c.name           AS customer,
  o.id             AS order_id,
  p.name           AS product,
  oi.quantity,
  oi.unit_price,
  (oi.quantity * oi.unit_price) AS line_total
FROM order_items oi
JOIN orders    o  ON oi.order_id   = o.id
JOIN customers c  ON o.customer_id = c.id
JOIN products  p  ON oi.product_id = p.id
ORDER BY o.id, p.name;

-- 5. (Setup for self-join) Create employees table
CREATE TABLE IF NOT EXISTS employees (
  id         INT PRIMARY KEY AUTO_INCREMENT,
  name       VARCHAR(100),
  manager_id INT,
  FOREIGN KEY (manager_id) REFERENCES employees(id)
);
INSERT INTO employees (id, name, manager_id) VALUES
  (1, 'Rajesh Kumar',   NULL),
  (2, 'Priya Sharma',   1),
  (3, 'Rahul Verma',    1),
  (4, 'Ankit Singh',    2),
  (5, 'Meena Kumari',   2);

-- 6. Self JOIN: employee + their manager name
SELECT e.name AS employee, COALESCE(m.name, 'CEO') AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- 7. Aggregate with JOIN: total spend per customer
SELECT c.name, SUM(oi.quantity * oi.unit_price) AS total_spent
FROM customers c
JOIN orders    o  ON c.id          = o.customer_id
JOIN order_items oi ON o.id        = oi.order_id
GROUP BY c.id, c.name
ORDER BY total_spent DESC;

-- 8. GROUP BY + HAVING: customers who spent > ₹5000
SELECT c.name, SUM(o.total) AS total_spent
FROM customers c
JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name
HAVING total_spent > 5000
ORDER BY total_spent DESC;

-- Bonus: CTE version of query 7
WITH customer_spend AS (
  SELECT c.id, c.name, SUM(oi.quantity * oi.unit_price) AS total_spent
  FROM customers c
  JOIN orders    o  ON c.id = o.customer_id
  JOIN order_items oi ON o.id = oi.order_id
  GROUP BY c.id, c.name
)
SELECT name, total_spent FROM customer_spend ORDER BY total_spent DESC;
Lab 4

Indexes & EXPLAIN (Week 4 Advanced Practice)

▼

Objective

See indexes in action: compare EXPLAIN output before and after adding indexes, understand what makes a query fast or slow.

Step-by-Step

  1. Baseline: EXPLAIN without index
    -- orders.customer_id has no index yet
EXPLAIN SELECT * FROM orders WHERE customer_id = 1;
-- Look for: type=ALL (full scan), key=NULL, rows=high number
  2. Add index, re-run EXPLAIN
    CREATE INDEX idx_cust ON orders(customer_id);

EXPLAIN SELECT * FROM orders WHERE customer_id = 1;
-- Now: type=ref, key=idx_cust, rows=much lower
  3. Composite index on order_items
    CREATE INDEX idx_order_product ON order_items(order_id, product_id);

EXPLAIN
SELECT oi.quantity, p.name
FROM order_items oi
JOIN products p ON oi.product_id = p.id
WHERE oi.order_id = 2;
-- key should show idx_order_product
  4. DROP index, see performance change
    DROP INDEX idx_cust ON orders;

EXPLAIN SELECT * FROM orders WHERE customer_id = 1;
-- Back to type=ALL after dropping the index
  5. UNIQUE index
    -- customers.email already has UNIQUE constraint (which creates a UNIQUE index)
SHOW INDEXES FROM customers;
-- You should see the unique index on email

-- Try inserting a duplicate email — observe the error
INSERT INTO customers (name, email) VALUES ('Test', 'priya@example.com');
-- Error: Duplicate entry 'priya@example.com' for key 'email'
  6. SHOW INDEXES and understand output
    SHOW INDEXES FROM orders;
-- Key columns:
-- Table: table name
-- Key_name: index name (PRIMARY for PK)
-- Column_name: indexed column
-- Non_unique: 0 = unique index, 1 = non-unique
-- Cardinality: estimated distinct values (higher = better candidate for index)
-- Index_type: BTREE (most common)
Lab 5

Transactions Practice (Week 4 Bonus Lab)

▼

Objective

Write a stored procedure that processes an order transactionally — checking stock, deducting it, and rolling back if stock is insufficient.

Step-by-Step

  1. Create the order processing procedure
    DELIMITER //
CREATE PROCEDURE place_order(
  IN p_customer_id INT,
  IN p_product_id  INT,
  IN p_quantity    INT,
  OUT p_order_id   INT,
  OUT p_message    VARCHAR(200)
)
BEGIN
  DECLARE v_stock   INT;
  DECLARE v_price   DECIMAL(10,2);
  DECLARE v_total   DECIMAL(10,2);

  -- Error handler: rollback on any SQL error
  DECLARE EXIT HANDLER FOR SQLEXCEPTION
  BEGIN
    ROLLBACK;
    SET p_order_id = -1;
    SET p_message  = 'Transaction failed and rolled back.';
  END;

  START TRANSACTION;

  -- Lock the product row to prevent concurrent over-selling
  SELECT price, stock INTO v_price, v_stock
  FROM products WHERE id = p_product_id FOR UPDATE;

  -- Check stock
  IF v_stock < p_quantity THEN
    ROLLBACK;
    SET p_order_id = -1;
    SET p_message  = CONCAT('Insufficient stock. Available: ', v_stock);
  ELSE
    SET v_total = v_price * p_quantity;

    -- Deduct stock
    UPDATE products SET stock = stock - p_quantity WHERE id = p_product_id;

    -- Create order
    INSERT INTO orders (customer_id, total, status)
    VALUES (p_customer_id, v_total, 'pending');
    SET p_order_id = LAST_INSERT_ID();

    -- Add order item
    INSERT INTO order_items (order_id, product_id, quantity, unit_price)
    VALUES (p_order_id, p_product_id, p_quantity, v_price);

    COMMIT;
    SET p_message = CONCAT('Order placed successfully. Order ID: ', p_order_id);
  END IF;
END //
DELIMITER ;
  2. Test happy path (stock available)
    -- Check stock before
SELECT id, name, stock FROM products WHERE id = 1;

-- Place order
CALL place_order(3, 1, 2, @order_id, @msg);
SELECT @order_id, @msg;

-- Verify stock deducted
SELECT id, name, stock FROM products WHERE id = 1;

-- Verify order created
SELECT * FROM orders WHERE id = @order_id;
SELECT * FROM order_items WHERE order_id = @order_id;
  3. Test failure path (zero stock)
    -- Set stock to 0 first
UPDATE products SET stock = 0 WHERE id = 1;

-- Try to order
CALL place_order(3, 1, 2, @order_id, @msg);
SELECT @order_id, @msg;
-- order_id should be -1, message should mention insufficient stock

-- Verify: no order was created, stock still 0
SELECT stock FROM products WHERE id = 1;
SELECT COUNT(*) FROM orders WHERE customer_id = 3 AND status = 'pending';
  4. SAVEPOINT: partial rollback
    -- Update stock back to test value
UPDATE products SET stock = 10 WHERE id = 1;

START TRANSACTION;
  -- First item goes through
  UPDATE products SET stock = stock - 1 WHERE id = 1;
  INSERT INTO orders (customer_id, total) VALUES (1, 599);
  SET @order_id = LAST_INSERT_ID();
  SAVEPOINT after_first_item;

  -- Second item: intentional error (invalid product_id)
  INSERT INTO order_items (order_id, product_id, quantity, unit_price)
  VALUES (@order_id, 999, 1, 0.00);  -- product 999 doesn't exist -> FK error

  -- Roll back only to after first item
  ROLLBACK TO after_first_item;

  -- First item's order entry still exists, stock still deducted
COMMIT;

-- Verify: order exists, but no items (we rolled back to savepoint)
SELECT * FROM orders WHERE id = @order_id;
SELECT * FROM order_items WHERE order_id = @order_id;
