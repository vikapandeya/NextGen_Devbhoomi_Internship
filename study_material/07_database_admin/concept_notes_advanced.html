Week 9

PostgreSQL Advanced Features

▼

Why PostgreSQL over MySQL?

  • Full ACID compliance with MVCC (Multi-Version Concurrency Control) — readers don't block writers
  • Rich data types: arrays, JSONB, geometric, network address, ranges, composite types
  • More powerful query planner (parallel queries, partial indexes, expression indexes)
  • Extension ecosystem: PostGIS (geospatial), pgvector (AI embeddings), timescaledb
  • Better standards compliance (ANSI SQL)
  • Default in cloud managed services: Supabase, Neon, Railway use PostgreSQL

JSON and JSONB Columns

-- JSON stores exact text; JSONB stores binary (faster queries, supports GIN index)
-- Use JSONB for querying; JSON only if you need to preserve key order/whitespace
CREATE TABLE products (
  id     SERIAL PRIMARY KEY,
  name   TEXT,
  meta   JSONB    -- flexible attributes: {"weight": "1kg", "color": "red", "tags": ["sale"]}
);

-- Extract value as JSON
SELECT meta->'color' FROM products WHERE id = 1;
-- Extracts: "red" (with quotes)

-- Extract value as text
SELECT meta->>'color' FROM products WHERE id = 1;
-- Extracts: red (without quotes)

-- Nested extraction
SELECT meta->'dimensions'->>'height' FROM products;

-- Filter by JSONB value (containment operator @>)
SELECT * FROM products WHERE meta @> '{"color": "red"}';

-- GIN index for fast JSONB queries
CREATE INDEX idx_meta ON products USING GIN (meta);

-- Update a JSONB field
UPDATE products
SET meta = jsonb_set(meta, '{color}', '"blue"')
WHERE id = 1;

Array Columns

CREATE TABLE posts (
  id    SERIAL PRIMARY KEY,
  title TEXT,
  tags  TEXT[]  -- native array type
);

INSERT INTO posts (title, tags) VALUES
  ('Intro to PostgreSQL', ARRAY['database', 'postgresql', 'beginner']),
  ('Advanced Indexing',   '{database,indexing,performance}');

-- Query: posts with 'database' tag
SELECT * FROM posts WHERE 'database' = ANY(tags);

-- Query: posts with ALL of these tags
SELECT * FROM posts WHERE tags @> ARRAY['database','postgresql'];

-- Expand array into rows
SELECT id, unnest(tags) AS tag FROM posts;

Window Functions

-- ROW_NUMBER, RANK, DENSE_RANK: ranking within partitions
SELECT
  name,
  department,
  salary,
  ROW_NUMBER()  OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
  RANK()        OVER (PARTITION BY department ORDER BY salary DESC) AS rank,
  DENSE_RANK()  OVER (PARTITION BY department ORDER BY salary DESC) AS dense_rank
FROM employees;
-- ROW_NUMBER: 1,2,3,4 always unique
-- RANK:       1,2,2,4 (gap after tie)
-- DENSE_RANK: 1,2,2,3 (no gap after tie)

-- LAG / LEAD: access previous/next row's value
SELECT
  order_date,
  revenue,
  LAG(revenue) OVER (ORDER BY order_date) AS prev_revenue,
  revenue - LAG(revenue) OVER (ORDER BY order_date) AS change
FROM daily_sales;

-- Running total with SUM() OVER
SELECT
  order_date,
  revenue,
  SUM(revenue) OVER (ORDER BY order_date ROWS UNBOUNDED PRECEDING) AS running_total
FROM daily_sales;

-- NTILE: divide into N buckets (quartiles, deciles)
SELECT name, score, NTILE(4) OVER (ORDER BY score DESC) AS quartile
FROM exam_results;

CTEs (Common Table Expressions)

-- Regular CTE: named subquery, run once
WITH monthly_revenue AS (
  SELECT
    DATE_TRUNC('month', created_at) AS month,
    SUM(total) AS revenue
  FROM orders
  WHERE status = 'delivered'
  GROUP BY 1
)
SELECT month, revenue,
  revenue - LAG(revenue) OVER (ORDER BY month) AS growth
FROM monthly_revenue;

-- Recursive CTE: tree traversal (org chart, category hierarchy)
WITH RECURSIVE org_chart AS (
  -- Base case: top-level employees (no manager)
  SELECT id, name, manager_id, 0 AS level
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- Recursive case: each employee's direct reports
  SELECT e.id, e.name, e.manager_id, oc.level + 1
  FROM employees e
  JOIN org_chart oc ON e.manager_id = oc.id
)
SELECT level, name FROM org_chart ORDER BY level, name;

Full-Text Search

-- Create text search vector
ALTER TABLE posts ADD COLUMN search_vector tsvector;
UPDATE posts SET search_vector = to_tsvector('english', title || ' ' || body);

-- GIN index on search vector
CREATE INDEX idx_fts ON posts USING GIN(search_vector);

-- Search: find posts matching 'database performance'
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'database & performance') AS query
WHERE search_vector @@ query
ORDER BY rank DESC;

User Management & Least Privilege

-- Create application user with minimal permissions
CREATE USER 'app_user'@'%' IDENTIFIED BY 'strong_password_here';
GRANT SELECT, INSERT, UPDATE, DELETE ON myapp.* TO 'app_user'@'%';

-- Create read-only reporting user
CREATE USER 'reporter'@'localhost' IDENTIFIED BY 'pass';
GRANT SELECT ON myapp.* TO 'reporter'@'localhost';

-- Verify permissions
SHOW GRANTS FOR 'app_user'@'%';

-- Revoke a permission
REVOKE DELETE ON myapp.orders FROM 'app_user'@'%';

-- PostgreSQL role-based approach
CREATE ROLE readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly;
CREATE USER analyst WITH PASSWORD 'pass';
GRANT readonly TO analyst;

Backup Strategies

Logical backup (mysqldump / pg_dump): exports SQL statements — portable, human-readable, slow for large databases.

-- MySQL: backup single database
mysqldump -u root -p myapp > myapp_backup_2024-06-15.sql

-- MySQL: backup all databases
mysqldump -u root -p --all-databases > full_backup.sql

-- MySQL: restore
mysql -u root -p myapp < myapp_backup_2024-06-15.sql

-- PostgreSQL: backup
pg_dump -U postgres myapp -f myapp_backup.sql
pg_dump -U postgres myapp -Fc -f myapp_backup.dump  # compressed format

-- PostgreSQL: restore compressed
pg_restore -U postgres -d myapp_restored myapp_backup.dump

Physical backup: file-level copy of data directory. Use Percona XtraBackup for MySQL (hot backup — no downtime). Faster for very large databases (>100GB).

Point-in-Time Recovery (PITR): restore to any moment by replaying binary logs (MySQL) or WAL archives (PostgreSQL) on top of a full backup. Essential for "undo" operations.

Replication

MySQL replication (primary → replica):

  • Primary writes to binary log (binlog)
  • Replica reads binlog and applies changes
  • Use cases: read scaling (route reads to replicas), failover, reporting without impacting production
  • Statement-based vs row-based replication: row-based is safer for non-deterministic queries

PostgreSQL logical replication: more flexible — replicate specific tables, apply transformations, replicate to different PostgreSQL versions.

Monitoring & Maintenance

-- MySQL: enable slow query log
SET GLOBAL slow_query_log = 1;
SET GLOBAL long_query_time = 1;  -- log queries taking >1 second
SET GLOBAL slow_query_log_file = '/var/log/mysql-slow.log';

-- MySQL sys schema: find the worst queries
SELECT query, exec_count, avg_latency
FROM sys.statement_analysis
ORDER BY avg_latency DESC LIMIT 10;

-- PostgreSQL: pg_stat_statements extension (add to postgresql.conf)
-- shared_preload_libraries = 'pg_stat_statements'
SELECT query, calls, mean_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC LIMIT 10;

-- PostgreSQL VACUUM: reclaims space from deleted rows
VACUUM ANALYZE posts;          -- manual vacuum + update statistics
-- autovacuum handles this automatically in production

-- MySQL: rebuild table and reclaim space
OPTIMIZE TABLE orders;

Partitioning

When tables exceed ~100 million rows, queries slow down even with indexes. Partitioning splits one logical table into multiple physical sub-tables based on a column value:

-- MySQL range partitioning: by year
CREATE TABLE orders (
  id         INT AUTO_INCREMENT,
  created_at DATETIME NOT NULL,
  total      DECIMAL(10,2),
  PRIMARY KEY (id, created_at)  -- partition key must be in PK
)
PARTITION BY RANGE (YEAR(created_at)) (
  PARTITION p2022 VALUES LESS THAN (2023),
  PARTITION p2023 VALUES LESS THAN (2024),
  PARTITION p2024 VALUES LESS THAN (2025),
  PARTITION pmax  VALUES LESS THAN MAXVALUE
);
-- Query WHERE YEAR(created_at)=2023 only scans p2023 partition

NoSQL Databases

▼

CAP Theorem

In a distributed system, you can guarantee at most 2 of these 3 properties:

  • Consistency: every read receives the most recent write
  • Availability: every request receives a response (not necessarily the latest data)
  • Partition Tolerance: the system continues operating even when network partitions occur

Network partitions always happen in distributed systems, so you choose CA (traditional RDBMS, single node) or CP (MongoDB, HBase) or AP (Cassandra, DynamoDB with eventual consistency).

MongoDB — Document Store

// Documents (BSON = Binary JSON) in a "products" collection
{
  "_id": ObjectId("..."),
  "name": "MacBook Pro",
  "price": 199999,
  "specs": { "ram": "16GB", "storage": "512GB" },  // embedded document
  "tags": ["laptop", "apple", "sale"],              // array
  "reviews": [                                       // embedded array of docs
    { "user": "Priya", "rating": 5, "text": "Great!" }
  ]

const productSchema = new mongoose.Schema({
  name:  { type: String, required: true },
  price: { type: Number, min: 0 },
  tags:  [String],
  specs: mongoose.Schema.Types.Mixed

const Product = mongoose.model('Product', productSchema);

// CRUD
await Product.create({ name: 'Laptop', price: 45000, tags: ['electronics'] });
await Product.find({ tags: 'electronics', price: { $lte: 50000 } }).sort('-price').limit(5);
await Product.updateOne({ _id: id }, { $set: { price: 42000 }, $push: { tags: 'sale' } });
await Product.deleteMany({ price: { $lt: 100 } });

Redis — Key-Value Store

// Redis data structures and use cases:

// String: caching, counters, flags
SET user:42:name "Priya Sharma" EX 3600    // with 1-hour TTL
GET user:42:name
INCR page:views:home                       // atomic counter

// Hash: object storage (like a row)
HSET user:42 name "Priya" email "p@e.com" role "admin"
HGET user:42 name                          // "Priya"
HGETALL user:42                            // all fields

// List: queues, recent activity
LPUSH notifications:42 "Your order shipped"
LRANGE notifications:42 0 9               // last 10 notifications

// Sorted Set: leaderboards, time-series
ZADD leaderboard 1500 "alice" 1200 "bob" 1800 "charlie"
ZREVRANGE leaderboard 0 4 WITHSCORES     // top 5 with scores
ZINCRBY leaderboard 100 "bob"            // add 100 to bob's score

// Node.js with ioredis
const Redis = require('ioredis');
const redis = new Redis();
await redis.set('key', JSON.stringify(data), 'EX', 300);  // 5-min cache
const cached = await redis.get('key');
if (cached) return JSON.parse(cached);    // cache hit

ACID transactions required (financial, inventory)Horizontal scale needed (billions of records)

Data is highly structured and stableVery high write throughput (IoT, logs)

Cloud Databases & Modern Patterns

▼

AWS Managed Databases

  • RDS (Relational Database Service): managed MySQL, PostgreSQL, MariaDB, Oracle, SQL Server. Handles patching, backups, monitoring. Multi-AZ for automatic failover. Read replicas for scaling reads. You still control the schema and queries.
  • Aurora: MySQL/PostgreSQL compatible but built from scratch by AWS. Claims 5× faster than MySQL, 3× faster than PostgreSQL. Auto-scales storage up to 128TB. Aurora Serverless v2 pauses/resumes on demand. Best for variable workloads.
  • DynamoDB: serverless key-value + document database. Millisecond latency at any scale. Pay per request or provisioned capacity. GSI (Global Secondary Index) enables alternate access patterns. On-demand billing — pay only for what you use.

Modern Database Services

  • Google Cloud Spanner: the first globally distributed, horizontally scalable RDBMS. Full SQL support with ACID across regions. Used by Google (YouTube, Google Ads). Very expensive but unique capability.
  • PlanetScale (MySQL Serverless): built on Vitess (YouTube's MySQL sharding layer). Key feature: branching — create a branch of your database schema for a PR, test migrations in isolation, merge like code. No-downtime schema changes.
  • Supabase: PostgreSQL as a service with auto-generated REST and GraphQL APIs, Auth, Realtime subscriptions, Storage. Popular with Next.js / React developers. Free tier generous.
  • Neon: serverless PostgreSQL with branching, auto-scaling, and a generous free tier. Pauses when idle — ideal for development and low-traffic projects.

Database Migrations

Version-controlled schema changes — just like Git for code, but for your database schema:

-- Migration files (Flyway format, alphabetically sorted)
V1__create_users.sql
V2__create_posts.sql
V3__add_tags_column.sql
V4__create_comments.sql

-- V3__add_tags_column.sql
ALTER TABLE posts ADD COLUMN tags JSON;
-- Undo: ALTER TABLE posts DROP COLUMN tags;

-- Flyway commands:
-- flyway migrate    — apply pending migrations
-- flyway info       — show migration status
-- flyway validate   — verify checksums match
-- flyway repair     — fix failed migration records

-- V5 migration: safe pattern for large tables
-- Add column (instant with DEFAULT NULL)
ALTER TABLE posts ADD COLUMN search_vector tsvector;
-- Backfill in batches (don't lock the whole table)
UPDATE posts SET search_vector = to_tsvector('english', title || ' ' || body)
WHERE id BETWEEN 1 AND 10000;
-- Add index CONCURRENTLY (PostgreSQL, no table lock)
CREATE INDEX CONCURRENTLY idx_search ON posts USING GIN(search_vector);

Event Sourcing & CQRS (Advanced Pattern)

CQRS (Command Query Responsibility Segregation): separate models for reads and writes. The write model handles commands (place order, cancel order) and appends events to an event log. The read model is a denormalised "view" of the data optimised for queries.

Event Sourcing: instead of storing the current state, store every event that led to it:

-- Events table (append-only — never UPDATE or DELETE)
events(id, aggregate_type, aggregate_id, event_type, payload JSON, created_at)

-- Events for order #42:
event_type='ORDER_CREATED',  payload={"customer_id":1, "items":[...]}
event_type='PAYMENT_RECEIVED', payload={"amount":499.99}
event_type='ORDER_SHIPPED',   payload={"tracking":"IN1234"}

-- Current state = replay all events for aggregate_id=42
-- Benefit: complete audit trail, time travel, replayable
