Week 1

What is a Database?

▼

Database vs Flat Files (Excel)

A flat file like a spreadsheet works for personal data but breaks down at scale. Consider what happens when 500 students all try to update the same Excel file simultaneously — conflicts, overwrites, corruption. A Database Management System (DBMS) solves this:

  • Multi-user access: thousands of users can read/write simultaneously without conflicts
  • Data integrity: rules enforced at the database level, not dependent on user discipline
  • Scale: billions of rows are handled efficiently with indexes and query optimisation
  • Security: fine-grained access control — this user can SELECT, that user can only INSERT
  • Recovery: transactions and WAL (Write-Ahead Log) protect against data loss on crash

DBMS vs RDBMS

A DBMS (Database Management System) is any software for managing data: MySQL, PostgreSQL, MongoDB, Redis, Cassandra.

An RDBMS (Relational DBMS) specifically stores data in related tables with rows and columns, enforcing relationships between them: MySQL, PostgreSQL, SQLite, Oracle Database, Microsoft SQL Server.

Non-relational DBMSes (NoSQL) include document stores (MongoDB), key-value stores (Redis), and column-family databases (Cassandra) — covered in Week 11.

The Relational Model

  • Table (Relation): stores data about one type of entity — e.g. a students table
  • Row (Record/Tuple): one instance of the entity — one student
  • Column (Field/Attribute): one property of the entity — student's name, email, course
  • Primary Key: a column (or combination of columns) that uniquely identifies each row — no two rows can have the same primary key, and it cannot be NULL
  • Foreign Key: a column that references the primary key of another table, creating a relationship between the two tables
-- students table
id (PK) | name      | email               | course_id (FK)
--------|-----------|---------------------|---------------
1       | Priya     | priya@example.com   | 3
2       | Rahul     | rahul@example.com   | 1

-- courses table
id (PK) | course_name
--------|--------------------
1       | Web Development
3       | Data Science

ACID Properties

ACID is the set of guarantees that make database transactions reliable, even when things go wrong:

  • Atomicity: A transaction is all-or-nothing. If you're transferring ₹500 from account A to account B: debit A AND credit B happen together, or neither happens. A crash between the two operations leaves the database in a consistent state.
  • Consistency: A transaction takes the database from one valid state to another valid state. Constraints (foreign keys, NOT NULL, CHECK) are enforced — a transaction that would violate them is rolled back.
  • Isolation: Concurrent transactions do not interfere with each other. Two users booking the last seat simultaneously — only one succeeds; the other gets an error, not a double-booking.
  • Durability: Once a transaction is committed, it survives crashes. The data is written to disk (via WAL/redo log) before COMMIT returns success to the application.

SQL Overview

SQL (Structured Query Language) is the standard language for all RDBMS systems. It has four sub-languages:

  • DDL — Data Definition Language: defines the structure of the database 
    CREATE TABLE, ALTER TABLE, DROP TABLE, TRUNCATE TABLE
  • DML — Data Manipulation Language: works with the data inside tables 
    SELECT, INSERT, UPDATE, DELETE
  • DCL — Data Control Language: manages permissions 
    GRANT SELECT ON students TO 'reporting_user';
REVOKE INSERT ON students FROM 'reporting_user';
  • TCL — Transaction Control Language: manages transaction boundaries 
    COMMIT, ROLLBACK, SAVEPOINT, START TRANSACTION

Setting Up MySQL

  1. Download MySQL Community Server from dev.mysql.com/downloads (free). On Windows, use MySQL Installer — it installs both the server (port 3306 by default) and MySQL Workbench GUI.
  2. During installation, set a root password. Remember it.
  3. Open MySQL Workbench → New Connection → localhost, port 3306, root. Test connection.
  4. For CLI access: open a terminal and run mysql -u root -p, enter your password.
-- First commands to run
SHOW DATABASES;          -- see default databases
CREATE DATABASE myapp;   -- create your first database
USE myapp;               -- select it as the active database
SHOW TABLES;             -- list tables (empty at first)

SQL Fundamentals

▼

Creating Tables with Data Types

CREATE DATABASE it_university;
USE it_university;

CREATE TABLE students (
  id           INT PRIMARY KEY AUTO_INCREMENT,
  name         VARCHAR(100) NOT NULL,
  email        VARCHAR(150) NOT NULL UNIQUE,
  phone        VARCHAR(15),
  course       VARCHAR(50),
  gpa          DECIMAL(3,2),
  enrolled_at  DATE DEFAULT (CURRENT_DATE),
  is_active    BOOLEAN DEFAULT TRUE,
  notes        TEXT
);

Common MySQL data types:

  • INT — whole numbers (-2B to 2B). Use BIGINT for very large IDs.
  • VARCHAR(n) — variable-length string up to n characters. More efficient than TEXT for short strings.
  • TEXT — long-form text with no practical length limit. No indexing on full column.
  • DATE — YYYY-MM-DD. DATETIME adds time. TIMESTAMP auto-converts to UTC.
  • DECIMAL(p,s) — exact decimal: p = total digits, s = decimal places. Use for money — never FLOAT.
  • BOOLEAN — stored as TINYINT(1): 0=false, 1=true.

Constraints

CREATE TABLE products (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  name        VARCHAR(100) NOT NULL,
  sku         VARCHAR(50) UNIQUE,
  price       DECIMAL(10,2) NOT NULL CHECK (price >= 0),
  stock       INT DEFAULT 0 CHECK (stock >= 0),
  category    ENUM('electronics','clothing','food','other') NOT NULL
);
  • PRIMARY KEY — unique + NOT NULL. Table can have only one.
  • NOT NULL — column cannot contain NULL value.
  • UNIQUE — all values in column must be distinct. Can have NULLs (multiple NULLs allowed).
  • DEFAULT — value used when INSERT does not specify the column.
  • CHECK — enforces a condition (MySQL 8.0+ fully enforces these).

INSERT, SELECT, UPDATE, DELETE

-- INSERT single row
INSERT INTO students (name, email, course)
VALUES ('Priya Sharma', 'priya@example.com', 'Web Development');

-- INSERT multiple rows
INSERT INTO students (name, email, course) VALUES
  ('Rahul Verma', 'rahul@example.com', 'Data Science'),
  ('Ankit Singh', 'ankit@example.com', 'Cyber Security'),
  ('Meena Kumari', 'meena@example.com', 'Web Development');

-- SELECT all columns
SELECT * FROM students;

-- SELECT specific columns with WHERE
SELECT name, email, course
FROM students
WHERE course = 'Web Development'
  AND is_active = TRUE;

-- Comparison operators: =, !=, <, >, <=, >=, BETWEEN, LIKE, IN
SELECT * FROM students
WHERE enrolled_at BETWEEN '2024-01-01' AND '2024-12-31';

SELECT * FROM students WHERE name LIKE 'P%';    -- starts with P
SELECT * FROM students WHERE course IN ('Web Development', 'Data Science');

-- ORDER BY, LIMIT, OFFSET
SELECT name, enrolled_at
FROM students
ORDER BY enrolled_at DESC
LIMIT 10 OFFSET 20;   -- page 3 of 10-per-page results

-- UPDATE — ALWAYS use WHERE
UPDATE students
SET course = 'Full Stack Development'
WHERE id = 1;

-- UPDATE multiple columns
UPDATE students
SET course = 'Data Science', is_active = FALSE
WHERE email = 'old@example.com';

-- ⚠ UPDATE without WHERE updates ALL rows — dangerous!
-- DELETE — ALWAYS use WHERE
DELETE FROM students WHERE id = 5;

-- Verify before deleting — run SELECT first:
SELECT * FROM students WHERE id = 5;  -- check, then
DELETE FROM students WHERE id = 5;
Week 3

Joins & Relationships

▼

Why Multiple Tables?

Storing everything in one table causes redundancy. If you store the customer's city in every order row, you need to update every row when their city changes — one update becomes thousands. Normalisation splits data into related tables, storing each fact exactly once.

E-Commerce Schema Example

CREATE TABLE customers (
  id         INT PRIMARY KEY AUTO_INCREMENT,
  name       VARCHAR(100) NOT NULL,
  email      VARCHAR(150) UNIQUE,
  city       VARCHAR(50)
);

CREATE TABLE orders (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  customer_id INT NOT NULL,
  total       DECIMAL(10,2),
  status      ENUM('pending','shipped','delivered','cancelled') DEFAULT 'pending',
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (customer_id) REFERENCES customers(id)
);

CREATE TABLE products (
  id    INT PRIMARY KEY AUTO_INCREMENT,
  name  VARCHAR(100),
  price DECIMAL(10,2)
);

CREATE TABLE order_items (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  order_id    INT NOT NULL,
  product_id  INT NOT NULL,
  quantity    INT NOT NULL,
  unit_price  DECIMAL(10,2) NOT NULL,
  FOREIGN KEY (order_id)   REFERENCES orders(id),
  FOREIGN KEY (product_id) REFERENCES products(id)
);

JOIN Types

-- INNER JOIN: rows where customer_id matches in both tables
SELECT c.name, o.id AS order_id, o.total, o.status
FROM orders o
JOIN customers c ON o.customer_id = c.id;

-- LEFT JOIN: ALL customers, even those with zero orders
-- (o.id will be NULL for customers with no orders)
SELECT c.name, COUNT(o.id) AS order_count
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name;

-- Multi-table JOIN: full order receipt
SELECT
  c.name        AS customer,
  o.id          AS order_id,
  p.name        AS product,
  oi.quantity,
  oi.unit_price,
  (oi.quantity * oi.unit_price) AS line_total
FROM order_items oi
JOIN orders   o ON oi.order_id   = o.id
JOIN customers c ON o.customer_id = c.id
JOIN products  p ON oi.product_id = p.id
WHERE o.id = 1;

-- Self JOIN: employee reports to manager (same table)
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id
ORDER BY m.name;
Week 4

Aggregations & Grouping

▼

Aggregate Functions

-- COUNT: total orders, orders with notes
SELECT COUNT(*) AS total_orders FROM orders;
SELECT COUNT(notes) AS orders_with_notes FROM orders;  -- ignores NULLs

-- SUM, AVG, MIN, MAX
SELECT
  SUM(total)  AS revenue,
  AVG(total)  AS avg_order_value,
  MIN(total)  AS smallest_order,
  MAX(total)  AS largest_order
FROM orders
WHERE status = 'delivered';

GROUP BY and HAVING

-- Total orders and revenue per customer
SELECT
  c.name,
  COUNT(o.id)  AS order_count,
  SUM(o.total) AS total_spent
FROM customers c
JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name
ORDER BY total_spent DESC;

-- HAVING filters AFTER grouping (WHERE filters BEFORE)
-- Find customers who spent more than ₹5000
SELECT
  c.name,
  SUM(o.total) AS total_spent
FROM customers c
JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name
HAVING total_spent > 5000;

-- Count orders per day
SELECT DATE(created_at) AS order_date, COUNT(*) AS daily_orders
FROM orders
GROUP BY DATE(created_at)
ORDER BY order_date;

NULL Handling

-- IS NULL / IS NOT NULL
SELECT * FROM customers WHERE phone IS NULL;
SELECT * FROM customers WHERE phone IS NOT NULL;

-- COALESCE: return first non-NULL value
SELECT name, COALESCE(phone, 'Not Provided') AS phone
FROM customers;

-- NULLIF: returns NULL if two values are equal (avoids division by zero)
SELECT total / NULLIF(items_count, 0) AS avg_item_price
FROM orders;

String & Date Functions

-- String functions
SELECT UPPER(name), LOWER(email) FROM customers;
SELECT CONCAT(first_name, ' ', last_name) AS full_name FROM users;
SELECT SUBSTRING(email, 1, LOCATE('@', email) - 1) AS username FROM users;
SELECT LENGTH('Hello World');     -- 11
SELECT TRIM('  hello  ');         -- 'hello'
SELECT REPLACE(phone, '-', '');   -- remove dashes

-- Date functions
SELECT NOW();                                  -- 2024-06-15 14:30:00
SELECT CURDATE();                              -- 2024-06-15
SELECT DATE_FORMAT(created_at, '%d %M %Y');    -- 15 June 2024
SELECT DATEDIFF(NOW(), created_at) AS days_ago FROM orders;
SELECT DATE_ADD(expiry_date, INTERVAL 30 DAY) AS new_expiry FROM memberships;
SELECT * FROM orders WHERE YEAR(created_at) = 2024;
-- ⚠ Better for index performance:
SELECT * FROM orders WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31';

DISTINCT

-- Unique cities in the customers table
SELECT DISTINCT city FROM customers ORDER BY city;

-- Count of distinct courses enrolled
SELECT COUNT(DISTINCT course) AS course_variety FROM students;
