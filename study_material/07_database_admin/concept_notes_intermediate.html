Week 5

Database Design & Normalisation

▼

Entity-Relationship (ER) Diagrams

An ER diagram is a visual blueprint of the database structure before writing any SQL. It shows:

  • Entities: things the database tracks (Customer, Product, Order)
  • Attributes: properties of each entity (Customer has name, email, phone)
  • Relationships: how entities connect (Customer PLACES Order)
  • Cardinality: how many instances of each entity relate
    • 1:1 — one passport per person
    • 1:N — one customer, many orders
    • M:N — many students enrol in many courses (requires a junction table)

Tool recommendation: dbdiagram.io (free, web-based, generates SQL from diagram).

Normalisation Goals

Normalisation eliminates data anomalies:

  • Update anomaly: change customer's city → must update 10,000 order rows
  • Insert anomaly: can't add a course until a student enrols in it
  • Delete anomaly: delete the only student in a course → course data is lost

Normal Forms

1NF (First Normal Form): Each cell contains exactly one atomic value. No repeating groups (no "phone1, phone2, phone3" columns; use a separate phones table instead).

-- Violates 1NF: comma-separated tags in one column
-- ✗ product_id=1, tags='electronics,mobile,android'
-- ✓ product_tags(product_id, tag) with one row per tag

2NF (Second Normal Form): 1NF + no partial dependencies. Every non-key column depends on the entire primary key (applies when you have a composite primary key).

-- Violates 2NF: composite PK (order_id, product_id)
-- product_name depends only on product_id, not the full PK
-- ✗ order_items(order_id, product_id, product_name, quantity)
-- ✓ separate products table; order_items has only order-item-specific data

3NF (Third Normal Form): 2NF + no transitive dependencies. Non-key columns depend only on the primary key, not on other non-key columns.

-- Violates 3NF: zip_code → city → state (transitive)
-- ✗ customers(id, name, zip_code, city, state)
--   (city and state depend on zip_code, not on id)
-- ✓ zip_codes(zip_code, city, state) + customers references zip_code FK

BCNF (Boyce-Codd Normal Form): Every determinant is a candidate key. Stricter than 3NF — handles edge cases with multiple overlapping candidate keys.

Denormalisation: Intentional reversal — add some redundancy to speed up read-heavy analytical queries. Analytics dashboards often use denormalised "star schema" data warehouses because reporting joins hurt performance.

Physical Design Choices

  • Use INT for IDs < 2 billion rows; BIGINT for larger tables (social networks, IoT data)
  • Use VARCHAR for short variable strings; TEXT only for truly long content (blog posts)
  • Use DECIMAL(10,2) for money — never FLOAT or DOUBLE (binary floating point loses pennies)
  • Use ENUM sparingly — adding new values requires ALTER TABLE; a lookup table is more flexible
  • Store timestamps as DATETIME (local time stored as-is) or TIMESTAMP (auto-converted UTC, limited to 2038)
Week 6

Indexing & Performance

▼

What is an Index?

An index is a separate data structure (usually a B-tree) maintained alongside the table. It stores a sorted copy of the indexed column values with pointers to the corresponding rows. Just as a book's index tells you which page a topic is on without reading every page, a database index lets MySQL find rows without scanning the whole table.

Cost: Indexes take disk space and slow down INSERT/UPDATE/DELETE (the index must be updated). The trade-off is almost always worth it for columns used in WHERE, JOIN, and ORDER BY.

Types of Indexes

-- Primary index: automatically created on PRIMARY KEY
-- Cannot be dropped (dropping PK drops the index)

-- Secondary index
CREATE INDEX idx_customer_email ON customers(email);

-- Composite index: multi-column, ORDER MATTERS
-- This index helps queries that filter on (order_id) OR (order_id AND product_id)
-- It does NOT help queries that filter on (product_id) alone
CREATE INDEX idx_order_product ON order_items(order_id, product_id);

-- Covering index: includes all columns needed by the query
-- Query: SELECT name, email FROM users WHERE status='active'
-- Covering index: (status, name, email) -- query never touches the main table
CREATE INDEX idx_active_users ON users(status, name, email);

-- Unique index (also enforces uniqueness)
CREATE UNIQUE INDEX idx_unique_email ON users(email);

-- Show all indexes on a table
SHOW INDEXES FROM orders;

-- Drop an index
DROP INDEX idx_customer_email ON customers;

When NOT to Index

  • Write-heavy tables: audit_logs, event_streams — every INSERT updates all indexes
  • Low-cardinality columns: is_active (2 values), status with 3 values — a full table scan is often faster because MySQL must read half the table anyway
  • Small tables: (<10,000 rows) — full scan fits in memory, index overhead exceeds benefit

Reading EXPLAIN

-- Before adding index
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;
-- type: ALL (full table scan — reads every row)
-- key: NULL (no index used)
-- rows: 50000 (estimated rows scanned)

CREATE INDEX idx_cust ON orders(customer_id);

-- After adding index
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;
-- type: ref (index lookup — only matching rows read)
-- key: idx_cust (this index was used)
-- rows: 3 (only 3 rows scanned!)

EXPLAIN output key fields:

  • type=ALL — full table scan (almost always bad for large tables)
  • type=ref — index used for non-unique match (good)
  • type=eq_ref — index used for unique/PK match (very good)
  • type=const — single row retrieved via PK (best possible)
  • key=NULL — no index used
  • rows — estimated rows scanned (lower = faster)

Query Optimisation Tips

-- ✗ Avoid: function on indexed column in WHERE (disables index use)
SELECT * FROM orders WHERE YEAR(created_at) = 2024;

-- ✓ Better: use range condition (index can be used)
SELECT * FROM orders WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31';

-- ✗ Avoid: leading wildcard (can't use index)
SELECT * FROM customers WHERE email LIKE '%@gmail.com';

-- ✓ Better: trailing wildcard (index is used)
SELECT * FROM customers WHERE email LIKE 'priya%';

-- ✗ Avoid: SELECT * (fetches unnecessary columns, prevents covering index)
SELECT * FROM orders WHERE customer_id = 1;

-- ✓ Better: select only needed columns
SELECT id, total, status FROM orders WHERE customer_id = 1;

-- Pagination: OFFSET gets slow at high pages (scans and discards N rows)
-- ✗ Slow for page 5000: LIMIT 10 OFFSET 49990
-- ✓ Keyset pagination: WHERE id > last_seen_id LIMIT 10
Week 7

Transactions & Concurrency

▼

Transaction Lifecycle

START TRANSACTION;
  -- or BEGIN; (synonym in MySQL)

  UPDATE accounts SET balance = balance - 500 WHERE id = 1;
  UPDATE accounts SET balance = balance + 500 WHERE id = 2;

  -- If anything fails above, run:
  ROLLBACK;   -- reverts ALL changes since START TRANSACTION

  -- If everything succeeded:
COMMIT;       -- permanently saves changes

Isolation Levels

Isolation levels control how much a transaction can "see" of other concurrent transactions' work:

LevelDirty ReadNon-Repeatable ReadPhantom Read
READ UNCOMMITTEDPossiblePossiblePossible
READ COMMITTEDPreventedPossiblePossible
REPEATABLE READ (MySQL default)PreventedPreventedPossible*
SERIALIZABLEPreventedPreventedPrevented
  • Dirty Read: reading uncommitted data from another transaction — dangerous, that transaction may ROLLBACK
  • Non-Repeatable Read: same row gives different values when read twice in one transaction (another transaction committed a change in between)
  • Phantom Read: same query returns different rows when run twice (another transaction inserted/deleted rows in between)
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

Deadlock

A deadlock occurs when two transactions each hold a lock that the other needs:

  • Transaction A locks row 1, waits for row 2
  • Transaction B locks row 2, waits for row 1
  • Neither can proceed — they wait forever

MySQL detects this automatically and kills one transaction (the "victim"), rolling it back. Your application should catch the error and retry.

Prevention: Always lock rows in the same order across all transactions (always lock customer row before order row, never the reverse).

Locking & SAVEPOINTs

-- Pessimistic locking: lock rows for update immediately
SELECT * FROM products WHERE id = 5 FOR UPDATE;
-- Other transactions that try to UPDATE this row will WAIT until you COMMIT or ROLLBACK

-- Optimistic locking: use a version column
-- Check that version hasn't changed before updating
UPDATE products SET stock = stock-1, version = version+1
WHERE id = 5 AND version = 3;
-- If 0 rows updated, another transaction changed it first — retry

-- SAVEPOINT: partial rollback
START TRANSACTION;
  INSERT INTO orders (customer_id, total) VALUES (1, 299.99);
  SAVEPOINT after_order;

  INSERT INTO order_items VALUES (...);  -- this fails
  ROLLBACK TO after_order;              -- order stays, item reverted

  -- continue with corrected data
COMMIT;

Connection Pooling

Opening a new database connection per HTTP request is expensive — it takes 50–200ms. Connection pooling maintains a pool of open connections that are reused:

  • Application requests a connection from the pool
  • Pool checks out an existing open connection (instant)
  • Application uses it, returns it to pool when done
  • Pool keeps connections warm, ready for the next request

Node.js: mysql2 with createPool(), pg with Pool. Set connectionLimit: 10 for most apps. Too many connections overwhelm the database.

Week 8

Stored Procedures, Views & Triggers

▼

Views

A view is a saved SELECT query that behaves like a virtual table. You query it like a table, but no data is physically stored — the query runs on access.

-- Create a view for monthly sales (complex query saved once)
CREATE VIEW monthly_sales AS
SELECT
  DATE_FORMAT(created_at, '%Y-%m') AS month,
  COUNT(*)                          AS order_count,
  SUM(total)                        AS revenue
FROM orders
WHERE status = 'delivered'
GROUP BY DATE_FORMAT(created_at, '%Y-%m');

-- Now use it simply
SELECT * FROM monthly_sales WHERE month = '2024-06';

-- Drop a view
DROP VIEW IF EXISTS monthly_sales;

Use cases: simplify complex multi-table queries for reporting, restrict columns visible to certain users (grant SELECT on view but not underlying table).

Stored Procedures

Pre-compiled SQL blocks with parameters, stored on the server. Reduce network round-trips — the application calls the procedure name rather than sending a large SQL string.

DELIMITER //
CREATE PROCEDURE get_customer_orders(IN cust_id INT)
BEGIN
  SELECT
    o.id,
    o.total,
    o.status,
    o.created_at
  FROM orders o
  WHERE o.customer_id = cust_id
  ORDER BY o.created_at DESC;
END //
DELIMITER ;

-- Call from application or CLI:
CALL get_customer_orders(42);

-- Procedure with OUT parameter
DELIMITER //
CREATE PROCEDURE count_pending_orders(OUT total INT)
BEGIN
  SELECT COUNT(*) INTO total FROM orders WHERE status = 'pending';
END //
DELIMITER ;

CALL count_pending_orders(@result);
SELECT @result;

Triggers

A trigger is a stored procedure that MySQL executes automatically when a specific event (INSERT, UPDATE, DELETE) happens on a table.

-- AFTER INSERT trigger: write to audit log when order is placed
DELIMITER //
CREATE TRIGGER after_order_insert
AFTER INSERT ON orders
FOR EACH ROW
BEGIN
  INSERT INTO audit_log (action, table_name, record_id, created_at)
  VALUES ('ORDER_PLACED', 'orders', NEW.id, NOW());
END //
DELIMITER ;

-- BEFORE DELETE trigger: prevent deleting products with active orders
DELIMITER //
CREATE TRIGGER before_product_delete
BEFORE DELETE ON products
FOR EACH ROW
BEGIN
  DECLARE order_count INT;
  SELECT COUNT(*) INTO order_count
  FROM order_items WHERE product_id = OLD.id;

  IF order_count > 0 THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Cannot delete product with existing orders';
  END IF;
END //
DELIMITER ;

Pitfalls: Triggers are invisible to application developers — bugs in triggers are hard to find. Use them selectively for audit logs and referential integrity. Avoid business logic in triggers.

Event Scheduler

MySQL's built-in scheduler runs SQL tasks on a schedule (like cron, inside the database).

-- Enable event scheduler
SET GLOBAL event_scheduler = ON;

-- Delete old audit logs older than 90 days, runs daily at midnight
CREATE EVENT cleanup_old_logs
ON SCHEDULE EVERY 1 DAY STARTS '2024-01-01 00:00:00'
DO
  DELETE FROM audit_log WHERE created_at < DATE_SUB(NOW(), INTERVAL 90 DAY);
