Project Briefs — Database Administration

Three graded projects taking you from schema design to production-grade multi-database architecture. Each builds directly on the previous.

Beginner

P1 — Student Management System

Deadline: Week 4 Pass Mark: 60 / 100
Tools Required: MySQL MySQL Workbench CLI (optional)

Description

Design and populate a complete relational database for a college student management system. This project tests your ability to design a normalised schema from a real-world description, enforce data integrity with constraints, and write SQL queries that span multiple tables.

Requirements

  • ER Diagram first: Draw the ER diagram on paper, photograph it, and include the photo in your submission PDF. The diagram must show all entities, attributes, and relationships with cardinality notation.
  • Create all 5 tables: students, courses, enrollments, instructors, grades — all with proper constraints (PRIMARY KEY, NOT NULL, UNIQUE where appropriate) and correct FOREIGN KEY relationships between tables.
  • Insert realistic sample data: at least 20 students, 10 courses, 3 instructors. Populate enrollments and grades tables with appropriate data (not just 3 rows per table — make it realistic for a college).
  • Write 10 SQL queries that demonstrate a variety of SQL features:
    • A simple SELECT with WHERE filter and ORDER BY
    • A JOIN between 2 tables
    • A JOIN between 3 or more tables
    • An aggregate function (COUNT, SUM, AVG)
    • GROUP BY with a HAVING clause
    • A subquery (either correlated or non-correlated)
    • LEFT JOIN to find records that have no match
    • UPDATE with a WHERE condition
    • NULL handling (IS NULL or COALESCE)
    • A query you consider the most interesting or complex (explain why)

Grading Rubric

CriterionMarksWhat Examiners Look For
ER Design25All 5 entities present, relationships correct, cardinality labelled, photo submitted
Table Creation + Constraints25All FK relationships working, appropriate constraints on each column, no redundant data in schema
Sample Data Quality20Realistic data (real-sounding names, valid emails, sensible grades/scores), sufficient volume
SQL Query Variety + Correctness30All 10 queries run without error, correct results, variety of SQL features demonstrated
Total100Pass mark: 60. Submit as a .sql file with comments labelling each section.
Intermediate

P2 — E-Commerce Database with Analytics

Deadline: Week 8 Pass Mark: 65 / 100
Tools Required: MySQL or PostgreSQL Any query runner

Description

Build the complete data layer for an e-commerce platform. This goes beyond basic CRUD — you will add performance optimisation through indexes, automate business logic with stored procedures and triggers, and write analytical queries suitable for a business intelligence dashboard.

Requirements

  • Full e-commerce schema: customers, products, categories (with hierarchy), orders, order_items, payments, reviews. All with proper constraints and FK relationships.
  • Sample data: at least 100 rows across the main tables (use realistic data — real city names, real product names in a category, plausible prices).
  • 5 indexes with documented rationale: for each index, include a comment explaining: which query pattern it supports, the EXPLAIN output before and after. Include at least one composite index.
  • 5 stored procedures: sp_place_order (validates stock, creates order and items, updates stock), sp_apply_discount (recalculates order total by percentage), sp_cancel_order (updates status, restores stock), sp_get_customer_summary (customer info + stats), sp_inventory_report (products with low stock threshold parameter).
  • 3 triggers: audit log trigger on order INSERT, stock deduction trigger (AFTER INSERT on order_items), trigger to prevent deleting products that have order history.
  • 10 analytical queries: monthly revenue trend, week-over-week growth, customer lifetime value ranking, product performance by category, cart abandonment proxy (orders that stayed in pending status for >24h), repeat customer rate, top products by review score, revenue contribution by city, average delivery time, churn indicator query.
  • Performance report: pick 3 of your analytical queries. For each: show the EXPLAIN output before optimisation, describe the problem (full scan? filesort?), add the index, show EXPLAIN after, and note the estimated rows scanned improvement.

Grading Rubric

Schema Design20Normalised to 3NF, all relationships correct, appropriate data types, no redundancy
Sample Data10100+ rows, realistic content, good variety for analytics to be meaningful
Indexes155 indexes with EXPLAIN before/after, rationale explained, at least 1 composite index
Stored Procedures + Triggers25All 5 procedures and 3 triggers run correctly, edge cases handled (rollback on failure)

Advanced

P3 — Multi-Database Architecture

Deadline: Week 12 Pass Mark: 70 / 100
Tools Required: MySQL MongoDB Redis Node.js/Express Docker Compose

Description

Build a complete, production-architecture data layer for a social media platform. Use the right database for each use case — not one database for everything. Demonstrate that you understand why each technology was chosen, not just how to use it.

Requirements

  • MySQL: users, posts (metadata only — no content), followers (user_id + following_id), likes (user_id + post_id) — these need ACID transactions and complex JOIN queries. Design the schema for correctness and performance at scale.
  • MongoDB: post content document (embedded tags array, rich text body, media references array, engagement stats). Schema flexibility is why MongoDB is right here — each post type (text, photo, video) has different fields.
  • Redis: use 4 Redis patterns:
    1. Feed cache per user (sorted set: score = timestamp, value = post_id)
    2. Trending hashtags (sorted set: score = mention count in last hour)
    3. Active session tokens (hash with 24-hour TTL)
    4. Rate limiting (counter with 1-minute TTL per user)
  • Docker Compose file: starts all 3 databases with a single docker-compose up -d. Include health checks.
  • Express API with 3 key endpoints:
    1. GET /feed — check Redis sorted set first → if cache miss, query MySQL for followed user IDs + PostgreSQL/MongoDB for post content → cache result for 5 minutes → on new post creation, invalidate affected users' feeds
    2. POST /posts — insert metadata in MySQL, insert content in MongoDB (handle partial failure gracefully)
    3. GET /trending — return top 10 hashtags from Redis sorted set, with fallback to MongoDB aggregate if Redis is empty
  • Backup/restore script: a shell script that backs up all 3 databases with timestamps, and a restore script. Both should be idempotent.
  • Architecture decision document (1 page): for each of the 3 databases, explain: what data it stores, why this database was chosen over the alternatives, what would break if you moved that data to a different database. This document is worth 15% of your grade.

Grading Rubric

CriterionMarksWhat Examiners Look For
MySQL Schema20Normalised, correct FKs, indexes on join columns, appropriate constraints for social data
MongoDB Design15Document structure leverages embedding appropriately, queries use array operators and aggregation
Redis Patterns20All 4 patterns implemented correctly: sorted sets, hashes, TTLs, atomic operations
Docker Compose10Single command starts all services, health checks present, environment variables used for secrets
API Integration20All 3 endpoints work end-to-end, cache patterns correct, partial failure handled gracefully
Architecture Document15Clearly explains WHY each DB was chosen, shows understanding of trade-offs, not just "MongoDB is flexible"
Total100Pass mark: 70. Submit: GitHub repo link + architecture PDF + docker-compose up screenshot.
