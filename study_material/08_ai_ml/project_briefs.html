Project 1 — ML Model Comparison Notebook

Beginner Level
📅 Due: End of Week 4 ⏱ Estimated: 8–12 hours ✅ Pass mark: 60/100 📦 Deliverable: Jupyter Notebook + GitHub

Choose a real-world dataset (not Iris, not Titanic — find something that interests you), build a complete EDA + preprocessing + multi-model comparison analysis, and present your findings in a structured notebook that a non-technical manager could follow.

Project Objectives

  • Apply the complete ML workflow end-to-end on a novel dataset
  • Demonstrate understanding of data preprocessing best practices
  • Compare at least 3 ML models fairly using cross-validation
  • Communicate insights clearly through visualizations and markdown
Dataset Requirements
  • At least 500 samples and 5 features
  • Must have a clear prediction target (classification or regression)
  • Sourced from Kaggle, UCI ML Repository, data.gov.in, or World Bank Open Data
  • Must NOT be a standard tutorial dataset (Iris, Titanic, Boston Housing, MNIST, CIFAR-10)
  • Suggested themes: crop yield prediction, loan default prediction, e-commerce churn, air quality, hospital readmission, IPL match outcome
Required Notebook Sections
  • Section 1 — Problem Statement: What are you predicting? Why does it matter? What would the business impact be of a good model?
  • Section 2 — Data Loading & Overview: Shape, dtypes, target distribution, missing value audit
  • Section 3 — EDA (Exploratory Data Analysis): Minimum 5 meaningful visualizations with clear conclusions written below each
  • Section 4 — Preprocessing: Handle missing values, encode categoricals, scale where needed, use sklearn Pipeline
  • Section 5 — Model Training: At least 3 different algorithm families (e.g., Linear, Tree-based, Distance-based)
  • Section 6 — Evaluation: Cross-validated metrics, comparison table, select correct metrics for task type
  • Section 7 — Conclusions: Best model + rationale, top 3 business insights, limitations, next steps
Deliverables
  • GitHub repository with README.md (dataset source, how to run, key findings)
  • Jupyter Notebook (.ipynb) — all cells must be run top-to-bottom without errors
  • Exported HTML version of the notebook (File → Download → .html)
  • Optional: Streamlit app deployed on Streamlit Community Cloud (bonus +5 points)

Grading Rubric

Preprocessing Correctness20No data leakage. Pipeline used. Appropriate imputation and encoding strategies with justification.

Project 2 — Text AI Application

Intermediate Level
📅 Due: End of Week 8 ⏱ Estimated: 15–20 hours ✅ Pass mark: 65/100 📦 Deliverable: App + Notebook + GitHub

Build a working text AI application that solves a real problem using NLP techniques. The application must be interactive — either a Streamlit web app or a FastAPI service — and must include both a classical ML approach and a comparison with a pre-trained transformer model.

Choose Your Application Theme

  • Option A — News Classifier: Classify Indian news articles (business, sports, politics, technology, entertainment) using TF-IDF + Classifier AND Hugging Face zero-shot classification. Dataset: inshorts-news or Times of India dataset on Kaggle.
  • Option B — Customer Support Triage: Auto-categorize customer support tickets (billing, technical, account, feedback) and detect urgency. Route to appropriate department.
  • Option C — Resume Skill Extractor: Extract key skills, experience years, and job titles from resume text using NLP + Claude API structured output.
  • Option D — Hinglish Sentiment Analyzer: Classify sentiment in code-mixed Hindi-English text from social media. Handle transliteration challenges.
  • Option E — Open Theme: Propose your own text AI application (pre-approval required from mentor within Week 5).
Technical Requirements
  • Collect or source a text dataset with at least 500 labeled examples for your task
  • Implement complete text preprocessing pipeline (tokenization, cleaning, normalization)
  • Train a TF-IDF + ML model (baseline) with cross-validated evaluation
  • Integrate at least one pre-trained model (Hugging Face pipeline OR Claude/GPT API)
  • Build an interactive Streamlit app or FastAPI endpoint that accepts user input
  • Compare baseline vs pre-trained model performance quantitatively
  • Handle edge cases: empty input, very long text, non-English/Hinglish input
Deliverables
  • GitHub repository with README explaining the problem, approach, and how to run
  • Analysis notebook (.ipynb) with EDA, model training, and evaluation
  • Application code (app.py for Streamlit or FastAPI) — runnable with one command
  • Live demo: deploy Streamlit app on Streamlit Cloud OR record a 3-minute demo video
  • requirements.txt with pinned library versions
  • Model comparison table: accuracy, F1, inference time, and qualitative observations

Grading Rubric

CriterionPointsWhat we look for
Problem Definition & Data15Clear problem statement, appropriate dataset, data quality analysis, class distribution checked.
NLP Pipeline Quality20Proper tokenization, stopword removal, feature engineering relevant to task. Bigrams/ngrams where appropriate.
Model Performance25Baseline ML model properly evaluated. Pre-trained model integrated. Honest comparison with analysis of when each is better.
Application Quality25Works without crashing, handles edge cases, user-friendly interface, meaningful output (not just a number).

Project 3 — Production ML System

Advanced Level
📅 Due: End of Week 12 ⏱ Estimated: 25–35 hours ✅ Pass mark: 70/100 📦 Deliverable: Full System + Demo + GitHub

Build a complete, production-ready ML system from data ingestion through serving with proper experiment tracking, API, and monitoring. This is your capstone project — it should demonstrate that you can build AI systems that would work in a real company, not just in a Colab notebook.

Required System Components

  • Data Pipeline: Automated data loading, validation (schema checks, range checks, missing value audit), and feature engineering — all in reusable functions.
  • Training Pipeline: MLflow experiment tracking with parameter logging, metric logging, and model artifact storage. Must log at least 3 experiment runs comparing different approaches.
  • Model Quality Gate: Automated evaluation that promotes a model to "production" only if it meets a defined minimum performance threshold (e.g., AUC > 0.82).
  • FastAPI REST Service: Production-grade API with Pydantic input validation, error handling, health endpoint, and /predict endpoint. Must handle invalid inputs gracefully.
  • Monitoring: Data drift detection using at least one statistical method (PSI, KS test, or Wasserstein distance) comparing training distribution to simulated production data.
  • Documentation: Architecture diagram (can be ASCII), API documentation (FastAPI auto-docs is acceptable), and runbook explaining how to retrain and redeploy.
Choose Your Domain
  • Domain A — Financial Risk: Credit default prediction with regulatory-grade explainability (SHAP). Must handle class imbalance. Target: AUC > 0.85.
  • Domain B — Retail Intelligence: Demand forecasting + inventory optimization. Time-series features. RMSE as primary metric.
  • Domain C — Healthcare: Patient readmission prediction. Precision must be high (no false alarms). Integration with a simulated EHR data format.
  • Domain D — LLM-Powered App: Build a RAG system with ChromaDB, evaluation metrics (retrieval recall@k), and a FastAPI interface. Must handle at least 50 documents.
  • Domain E — Open Theme: Any domain with pre-approval. Must include all 5 required components.
Deliverables
  • GitHub repository with well-organized folder structure (data/, src/, tests/, docs/)
  • All code runnable via a single script or Makefile command
  • MLflow tracking — minimum 3 logged runs with different configurations
  • FastAPI application tested with at least 5 test cases (using pytest or httpx)
  • Drift monitoring report (can be generated HTML from Evidently or custom)
  • Architecture diagram and system README (500+ words explaining design decisions)
  • 5-minute recorded demo video showing the complete system end-to-end
