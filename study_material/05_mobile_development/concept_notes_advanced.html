Week 9 — Performance Optimization
W9

React Native Performance — Threads, Memo & FlatList

JS thread · UI thread · React.memo · useMemo · Hermes · Flipper

▼
JS Thread — runs your React/JavaScript logic, state updates, business rules
UI Thread — renders native views, must run at 60fps (16ms per frame)
Native Thread — handles device calls: GPS, camera, storage, sensors

Problem: heavy JS work blocks the JS thread → UI drops frames → janky scrolling

React.memo

  • Wraps a component — skips re-render if props haven't changed
  • const Card = React.memo(function Card({'{'}item{'}'}) {'{'}...{'}'})
  • Best for: FlatList renderItem components, expensive pure renders
  • Not free — adds a shallow comparison cost per render
  • Only use when you have profiled and confirmed unnecessary re-renders

useMemo & useCallback

  • useMemo — memoize expensive computed values
  • const sorted = useMemo(() => [...list].sort(), [list])
  • useCallback — memoize a function reference
  • const handler = useCallback(() => doThing(), [dep])
  • Pass memoized callbacks to React.memo children — otherwise the memo is bypassed

FlatList Optimisation Props

  • getItemLayout — provide fixed row height to skip measurement
  • removeClippedSubviews={true} — unmount off-screen items
  • maxToRenderPerBatch={10} — items per JS batch
  • windowSize={5} — render window in viewport multiples
  • initialNumToRender={10} — visible items on first paint
  • Memoize renderItem with useCallback

Hermes & Profiling

  • Hermes: Expo's default JS engine since RN 0.70 — faster startup, lower memory
  • Flipper: connect your device → JS thread, UI thread, network, layout inspector
  • React DevTools: component tree + props + render highlights
  • Use expo-image (not the built-in Image) — better cache and lazy loading
  • Profile first, optimise second — never guess at bottlenecks
// Optimised FlatList pattern
const renderItem = useCallback(({ item }) => <ArticleCard article={item} />, []);
const keyExtractor = useCallback(item => item.id.toString(), []);
const getItemLayout = useCallback(
  (_, index) => ({ length: 96, offset: 96 * index, index }),
  []
);

<FlatList
  data={articles}
  renderItem={renderItem}
  keyExtractor={keyExtractor}
  getItemLayout={getItemLayout}
  removeClippedSubviews={true}
  maxToRenderPerBatch={8}
  windowSize={5}
  initialNumToRender={12}
/>

// Memoised card — only re-renders when article changes
const ArticleCard = React.memo(function ArticleCard({ article }) {
  return (
    <View style={styles.card}>
      <Text style={styles.title}>{article.title}</Text>
    </View>
  );
});

Week 9 Key Takeaways

  • The UI thread must hit 60fps — any heavy JS work that blocks it causes visible jank
  • React.memo + useCallback together prevent FlatList from re-rendering every card on every parent render
  • getItemLayout is the single biggest FlatList performance win for fixed-height rows
  • Profile with Flipper before optimising — the bottleneck is rarely where you think
Week 10 — App Architecture & Testing
W10

Clean Architecture, Jest & Detox

Feature folders · Custom hooks · Unit tests · Component tests · E2E

▼
Architecture matters at scale: When your project has 5 screens, folder structure doesn't matter. At 30 screens with 5 developers, a bad structure becomes a daily source of merge conflicts and "where does this go?" discussions. Feature-based folders eliminate that friction.

Feature-Based Folder Structure

  • src/features/auth/ — screens, hooks, API calls for auth
  • src/features/feed/ — everything related to the post feed
  • src/features/profile/ — user profile feature
  • src/shared/ — reusable components, utils, constants
  • Each feature is self-contained — easy to delete or hand to another developer

Custom Hooks Pattern

  • Extract logic from components into useX hooks
  • useAuth — login, logout, token, user state
  • usePosts — fetch, add, delete posts
  • useDebounce(value, delay) — debounced search input
  • Hook lives in src/hooks/ or src/features/x/hooks/
  • Components become thin: just render, call hooks, pass props

Jest — Unit & Component Tests

  • Expo includes Jest preconfigured — just run npm test
  • Unit test: test a pure function (reducer, validator) in isolation
  • @testing-library/react-native — render components, fire events
  • fireEvent.press(getByText('Save')) — simulate button press
  • expect(getByText('Saved!')).toBeTruthy() — assert result
  • Mock external modules: jest.mock('@react-native-async-storage/async-storage')

Detox — End-to-End Tests

  • Tests the real compiled app on a real simulator/device
  • device.launchApp() → element(by.id('emailInput')).typeText('...')
  • Slower to run but catches integration bugs Jest cannot
  • Run nightly or on release branch in CI
  • Target: login flow, core user journey, payment flow
// reducer.test.js — unit test a pure reducer
import { todosReducer } from '../src/features/todos/todosReducer';

test('ADD_TODO adds an item to the list', () => {
  const state = { todos: [] };
  const action = { type: 'ADD_TODO', payload: { id: 1, text: 'Buy groceries' } };
  const next = todosReducer(state, action);
  expect(next.todos).toHaveLength(1);
  expect(next.todos[0].text).toBe('Buy groceries');
});

// TodoItem.test.js — component test
import { render, fireEvent } from '@testing-library/react-native';
import TodoItem from '../src/features/todos/TodoItem';

test('tapping a todo marks it done', () => {
  const onToggle = jest.fn();
  const { getByText } = render(
    <TodoItem todo={{'{'}id:1, text:'Read notes', done:false{'}'}} onToggle={onToggle} />
  );
  fireEvent.press(getByText('Read notes'));
  expect(onToggle).toHaveBeenCalledWith(1);
});

Week 10 Key Takeaways

  • Feature-based folders scale — flat file structures break down past ~20 screens
  • Custom hooks make components readable and make logic independently testable
  • Test reducers and custom hooks first — they contain most of your business logic
  • Use Detox sparingly for core user journeys; Jest for everything unit-testable
Week 11 — Publishing & CI/CD
W11

EAS Build, App Store & GitHub Actions

EAS Build · APK vs AAB · Play Console · TestFlight · OTA updates · CI

▼
EAS — Expo Application Services: Building a signed Android APK/AAB used to require a local Android Studio installation, a keystore, Gradle, and 45 minutes. EAS Build runs the entire process in the cloud. You push code, EAS builds it, and returns a download link.

EAS Build

  • npm install -g eas-cli → eas login
  • eas build:configure — creates eas.json
  • eas build -p android --profile preview — builds APK
  • eas build -p android --profile production — builds AAB for Play Store
  • EAS manages your keystore — no manual key management

APK vs AAB

  • APK: install directly on any Android device (sideload)
  • AAB: Android App Bundle — required by Google Play
  • AAB is smaller — Play Store generates device-specific APKs from it
  • Use APK for: internal testing, sharing with testers directly
  • Use AAB for: Google Play submission

Google Play Console

  • One-time $25 developer account registration
  • Internal testing track → Closed testing → Open testing → Production
  • Upload AAB → wait ~1 hour for review on internal track
  • Add test users via email to internal testing
  • Production review: 3–7 days for new apps

EAS Update — OTA

  • Update JS bundle without a new App Store submission
  • eas update --branch production --message "fix nav bug"
  • Users get the update next time they open the app
  • Cannot update native code (new Expo modules, app.json changes)
  • Dramatically shortens the bug-fix cycle
# .github/workflows/mobile-ci.yml
name: Mobile CI

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci

      - name: Lint
        run: npm run lint

      - name: Test
        run: npm test -- --coverage --ci

      - name: EAS Build (development)
        uses: expo/expo-github-action@v8
        with:
          expo-version: latest
          eas-version: latest
          token: ${'$'}{{ secrets.EXPO_TOKEN }}

      - run: eas build -p android --profile development --non-interactive

Week 11 Key Takeaways

  • EAS Build is the fastest path to a signed Android APK — no local toolchain required
  • AAB for Play Store, APK for direct install — know the difference before your first submission
  • EAS Update lets you ship JS fixes in minutes, bypassing the Play Store review queue
  • A GitHub Actions pipeline that runs lint + tests on every push is non-negotiable in professional teams
Week 12 — Animations, Native Modules & Security
W12

Reanimated 2, Gestures, Lottie & Security

useSharedValue · withSpring · GestureDetector · expo-secure-store · New Architecture

▼
Why Reanimated 2, not Animated API? The original Animated API runs on the JS thread — heavy JS work can drop animation frames. Reanimated 2 runs entirely on the UI thread using worklets (tiny JS functions compiled to native code). Animations stay at 60fps even when the JS thread is busy.

React Native Reanimated 2

  • npm install react-native-reanimated + add Babel plugin
  • useSharedValue(0) — animated value on UI thread
  • withTiming(1, {'{'}duration: 300{'}'}) — linear animation
  • withSpring(1, {'{'}damping: 10{'}'}) — physics-based spring
  • withRepeat(withTiming(...), -1) — loop forever
  • useAnimatedStyle — derive animated styles from shared values

Gesture Handler

  • npm install react-native-gesture-handler
  • Wrap app root in <GestureHandlerRootView>
  • Gesture.Pan(), Gesture.Pinch(), Gesture.Tap()
  • GestureDetector gesture={gesture}> wraps your component
  • Use useAnimatedGestureHandler for combined gesture + animation
  • Runs on UI thread — no JS communication during gesture

Lottie Animations

  • npm install lottie-react-native
  • Uses JSON animation files exported from Adobe After Effects
  • Free animations at LottieFiles.com
  • <LottieView source={require('./success.json')} autoPlay loop={false} />
  • Great for: success states, empty states, loading indicators, onboarding

App Security

  • expo-secure-store — stores sensitive data in iOS Keychain / Android Keystore
  • Never store auth tokens in AsyncStorage — it is plain text
  • Certificate pinning: verify the server's SSL certificate matches an expected value
  • Obfuscate JS bundle in production: enabled by default with Hermes + EAS Build
  • React Native New Architecture (JSI/Fabric): direct JS-to-native bridge — faster, no JSON serialisation
import Animated, { useSharedValue, useAnimatedStyle, withSpring }
  from 'react-native-reanimated';
import { Gesture, GestureDetector } from 'react-native-gesture-handler';

function LikeButton() {
  const scale = useSharedValue(1);

  const tap = Gesture.Tap().onBegin(() => {
    scale.value = withSpring(1.3, { damping: 6, stiffness: 200 });
  }).onFinalize(() => {
    scale.value = withSpring(1);
  });

  const animStyle = useAnimatedStyle(() => ({
    transform: [{ scale: scale.value }],
  }));

  return (
    <GestureDetector gesture={tap}>
      <Animated.View style={[styles.heart, animStyle]}>
        <Text style={styles.heartIcon}>❤️</Text>
      </Animated.View>
    </GestureDetector>
  );
}

// Secure token storage
import * as SecureStore from 'expo-secure-store';
await SecureStore.setItemAsync('auth_token', token);
const token = await SecureStore.getItemAsync('auth_token');

Week 12 Key Takeaways

  • Reanimated 2 runs on the UI thread — use it for all production animations, not the Animated API
  • One polished spring animation on your like button communicates quality better than ten scattered effects
  • Store auth tokens in expo-secure-store, not AsyncStorage — security is not optional
  • The React Native New Architecture (JSI/Fabric) is the future — understand the concept even if you don't implement it yet
