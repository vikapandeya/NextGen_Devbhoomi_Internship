Week 1 — Introduction to Mobile Development
W1

Introduction to Mobile Development

Native vs Cross-platform · iOS vs Android · React Native · Expo setup

▼
Real-world analogy: Building a mobile app is like opening a restaurant. You can open one kitchen that only serves one cuisine (native iOS or native Android), or you can build a central kitchen with an adaptor menu that serves both markets (React Native / Flutter). The single kitchen costs less to run — but the specialist kitchen can always do something the multi-cuisine one cannot quite match.

Three App Development Approaches

  • Native: Swift/SwiftUI for iOS, Kotlin/Compose for Android — best performance, full platform access, two codebases
  • Cross-platform: React Native, Flutter — one codebase, compiles to native widgets, ~95% of native capability
  • PWA: Web app in a browser shell — easiest, but limited device API access, no App Store listing

Why React Native for This Program

  • One codebase → iOS + Android simultaneously
  • JavaScript — you already know the language
  • Huge community (Meta + Microsoft backing)
  • Used by: Facebook, Instagram, Shopify, Discord
  • Expo makes setup effortless (no Xcode/Android Studio needed for learning)

Mobile UX Principles

  • Thumb zones: Most users hold phone with one hand — keep primary actions in the bottom third of the screen
  • Tap targets: Minimum 44×44 px — fingers are not mouse cursors
  • Mobile-first thinking: Small screen, slow network, interrupted use — design for constraints first
  • Gestures: Swipe, pinch, tap — feel natural; respect platform conventions

Setting Up: Expo

  • Install Node.js (nodejs.org) — includes npm
  • No global install needed: npx create-expo-app MyApp --template blank
  • Start: npx expo start → shows QR code in terminal
  • Install Expo Go on your phone (iOS App Store / Google Play)
  • Scan QR code → your app runs on your real device instantly
  • Save a file → app reloads automatically (Hot Reload)
# Create and run your first Expo app
npx create-expo-app MyFirstApp --template blank
cd MyFirstApp
npx expo start

# Folder structure:
MyFirstApp/
  App.js          # your root component
  app.json        # app name, icon, version
  package.json    # dependencies
  assets/         # images, fonts, icons
  node_modules/   # installed packages (never edit)

Week 1 Key Takeaways

  • React Native is the right choice for this program: JS-based, cross-platform, widely used in industry
  • Expo removes 90% of setup friction — you can test on a real phone in 5 minutes
  • Mobile UX rules differ from web: 44px tap targets, thumb zones, no hover states
  • Never tell a hiring manager "I only know web" — React Native skills transfer directly
Week 2 — React Native Fundamentals
W2

React Native Core Components

View · Text · StyleSheet · FlatList · Flexbox · Platform · Dimensions

▼
Key mental shift: In React web you use HTML tags like <div> and <p>. In React Native there are no HTML tags — everything is a native component. <View> is your <div>, <Text> wraps every word, and all layout is Flexbox.

Core Components

  • View — layout container (like div)
  • Text — every string must be inside Text
  • Image — local or remote images
  • ScrollView — scrollable container for short lists
  • TouchableOpacity — tappable with opacity feedback
  • Pressable — modern replacement, more control

StyleSheet API

  • No CSS files — styles are JavaScript objects
  • camelCase: backgroundColor not background-color
  • No cascading — each component is independent
  • Flexbox only — no Grid, no floats
  • StyleSheet.create({}) — validates and optimises styles
  • Numbers not strings for most sizes: padding: 16 not '16px'

React Native Flexbox

  • Default direction: column (not row like CSS!)
  • flex: 1 — fill all available space
  • flexDirection: 'row' — lay children horizontally
  • justifyContent — along main axis
  • alignItems — along cross axis
  • gap — spacing between children (RN 0.71+)

Platform & Dimensions APIs

  • Platform.OS === 'ios' or 'android'
  • Platform.select({ios: x, android: y}) — return different values per OS
  • Dimensions.get('window').width — screen width in dp
  • Dimensions.get('window').height — screen height
  • Use for responsive font sizes, padding adjustments
import { View, Text, StyleSheet, TouchableOpacity, Platform, Dimensions } from 'react-native';

const { width } = Dimensions.get('window');

export default function HomeScreen() {
  return (
    <View style={styles.container}>
      <Text style={styles.title}>Hello, Uttarakhand! 🏔️</Text>
      <Text style={styles.subtitle}>Running on {Platform.OS}</Text>
      <TouchableOpacity style={styles.btn} onPress={() => alert('Tapped!')} activeOpacity={0.7}>
        <Text style={styles.btnText}>Tap me</Text>
      </TouchableOpacity>
    </View>
  );
}

const styles = StyleSheet.create({
  container: {
    flex: 1,
    backgroundColor: '#F8F6F0',
    alignItems: 'center',
    justifyContent: 'center',
    padding: 24,
  },
  title: {
    fontSize: 24,
    fontWeight: '800',
    color: '#1B3A6B',
    marginBottom: 8,
    textAlign: 'center',
  },
  subtitle: { fontSize: 14, color: '#5A6070', marginBottom: 32 },
  btn: {
    backgroundColor: '#1B3A6B',
    paddingVertical: 14,
    paddingHorizontal: 36,
    borderRadius: 12,
    width: width * 0.8, // 80% of screen width
    alignItems: 'center',
  },
  btnText: { color: 'white', fontWeight: '700', fontSize: 15 },
});

Week 2 Key Takeaways

  • React Native's default flexDirection is column — this trips up every web developer once
  • All text must be inside a <Text> component — no bare strings in JSX
  • StyleSheet.create() runs validation at startup — use it, not plain objects
  • Platform.OS lets one codebase behave correctly on both platforms
Week 3 — Navigation in React Native
W3

React Navigation

Stack · Tabs · Drawer · Params · useNavigation hook

▼
Real-world analogy: Navigation is the backbone of every mobile app. Stack Navigator works like a stack of papers — push a new screen on top, pop it off to go back. Tab Navigator is the bottom bar you see in Instagram or WhatsApp — always visible, always one tap away.

React Navigation Setup

  • npm install @react-navigation/native
  • npx expo install react-native-screens react-native-safe-area-context
  • npm install @react-navigation/native-stack
  • npm install @react-navigation/bottom-tabs
  • Wrap your entire app in <NavigationContainer>

Navigator Types

  • Stack: Push/pop pattern — for drill-down flows (Home → Detail → Edit)
  • Bottom Tabs: Persistent tab bar — for top-level sections (Feed, Search, Profile)
  • Drawer: Side menu that slides in — for secondary navigation options
  • Navigators can be nested: Tabs inside a Stack, Stack inside Tabs

Passing Parameters

  • Send: navigation.navigate('Detail', {'{'} id: 42 {'}'})
  • Receive: const {'{'} id {'}'} = route.params
  • The receiving screen gets route prop automatically
  • Use useRoute() hook instead of prop if deeply nested
  • Params are not persisted — re-pass on navigation

Navigation Hooks

  • useNavigation() — get navigation object in any component
  • useRoute() — get route (params) in any component
  • Without hooks, only direct children of screens get nav props
  • navigation.goBack() — go back one screen
  • navigation.navigate('Home') — go to a named screen
  • navigation.setOptions({'{'}title: 'Edit Note'{'}'}) — change header title dynamically
import { NavigationContainer } from '@react-navigation/native';
import { createNativeStackNavigator } from '@react-navigation/native-stack';
import { createBottomTabNavigator } from '@react-navigation/bottom-tabs';

const Stack = createNativeStackNavigator();
const Tab   = createBottomTabNavigator();

// Tab navigator for main sections
function MainTabs() {
  return (
    <Tab.Navigator screenOptions={{'{'}tabBarActiveTintColor: '#1B3A6B'{'}'}}>
      <Tab.Screen name="Home"    component={HomeScreen} />
      <Tab.Screen name="Profile" component={ProfileScreen} />
    </Tab.Navigator>
  );
}

// Stack wraps the tabs + detail screens
export default function App() {
  return (
    <NavigationContainer>
      <Stack.Navigator>
        <Stack.Screen name="Main"   component={MainTabs} options={{'{'}headerShown: false{'}'}} />
        <Stack.Screen name="Detail" component={DetailScreen} options={{'{'}title: 'Details'{'}'}} />
      </Stack.Navigator>
    </NavigationContainer>
  );
}

// Navigate with params from HomeScreen
navigation.navigate('Detail', { id: 42, name: 'Hemkund Lake' });

// Read params in DetailScreen
const { id, name } = route.params;

Week 3 Key Takeaways

  • NavigationContainer must be at the root of your app — not inside any screen
  • Bottom tabs for top-level sections; Stack for drill-down flows — combine both
  • Use useNavigation() hook inside components that are not direct screen children
  • Params are not persistent — don't use them to store state, use AsyncStorage for that
Week 4 — Lists, Forms & User Input
W4

Lists, Forms & AsyncStorage

FlatList · SectionList · TextInput · Form validation · AsyncStorage

▼
Why FlatList, not ScrollView? ScrollView renders everything at once. For a 500-item list that causes lag. FlatList is virtualized — it renders only the ~15 items visible on screen at any moment, recycling cells as you scroll. For lists over 20 items, always use FlatList.

FlatList

  • data — the array to render
  • renderItem={'{'}({'{'}item{'}'}) => <Card item={item} />{'}'}
  • keyExtractor={'{'}item => item.id.toString(){'}'} — unique key per row
  • refreshing + onRefresh — pull-to-refresh
  • ListEmptyComponent — shown when data=[]
  • numColumns={2} — grid layout

SectionList

  • Like FlatList but with section headers
  • Data format: [{'{'}title, data: [...]{'}'}, ...]
  • renderSectionHeader — renders the section label
  • Good for: contacts (A-Z), grouped settings, categorised menus

TextInput

  • value + onChangeText — controlled input (always use this)
  • placeholder — hint text
  • keyboardType="email-address" — right keyboard layout
  • secureTextEntry={true} — password field
  • multiline — textarea equivalent
  • returnKeyType="done" — labels the return key

AsyncStorage

  • npm install @react-native-async-storage/async-storage
  • Key-value store — values must be strings
  • await AsyncStorage.setItem('key', JSON.stringify(data))
  • await AsyncStorage.getItem('key') — returns string or null
  • await AsyncStorage.removeItem('key')
  • Survives app close — perfect for user preferences, cached data
import { FlatList, TextInput, KeyboardAvoidingView, Platform } from 'react-native';
import AsyncStorage from '@react-native-async-storage/async-storage';

// Controlled form with validation
const [email, setEmail] = useState('');
const [error, setError]  = useState('');

function validate() {
  if (!email.includes('@')) { setError('Enter a valid email address'); return false; }
  setError('');
  return true;
}

<KeyboardAvoidingView behavior={Platform.OS === 'ios' ? 'padding' : 'height'}>
  <TextInput
    value={email}
    onChangeText={setEmail}
    placeholder="your@email.com"
    keyboardType="email-address"
    autoCapitalize="none"
    style={styles.input}
  />
  {error ? <Text style={styles.errorText}>{error}</Text> : null}
</KeyboardAvoidingView>

// Save to AsyncStorage
await AsyncStorage.setItem('todos', JSON.stringify(todosArray));

// Load on mount
useEffect(() => {
  async function load() {
    const raw = await AsyncStorage.getItem('todos');
    if (raw) setTodos(JSON.parse(raw));
  }
  load();
}, []); // empty deps = run once on mount

Week 4 Key Takeaways

  • Use FlatList for any list longer than ~20 items — ScrollView is for short, static content
  • Always wrap forms in KeyboardAvoidingView so the keyboard doesn't hide your inputs
  • AsyncStorage stores strings only — always JSON.stringify() on save, JSON.parse() on load
  • Load AsyncStorage inside a useEffect(()=>{}, []) so it runs once when the screen mounts
